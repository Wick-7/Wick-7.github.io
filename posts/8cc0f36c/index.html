Agentic AI Frameworks / AutoGen & Multimodal Knowledge Assistant

由两个相互衔接的部分组成：

  1. Agentic AI Frameworks & AutoGen

    • 为什么未来的 AI 应用会从单一模型转向 Agentic / Compound AI Systems；
    • 如何用统一的 Agent 抽象组织 LLM、工具与人类；
    • AutoGen 如何以 multi-agent conversation programming 为核心实现复杂工作流。

  2. Multimodal Knowledge Assistant

    • 为什么 Basic RAG 无法支撑真正的企业级知识助手；
    • 如何处理含有文本、表格、图片、图表和复杂版式的文档；
    • 如何在 Multimodal RAG 之上加入 Tool Use、Planning、Memory、Reflection；
    • 如何把 Agent workflow 部署成可扩展的生产系统。

两部分共同说明：

Agent 应用不只是“调用一个更强的 LLM”，而是围绕模型构建数据、工具、记忆、控制流、验证和人机协作等完整系统。

Part I：Agentic AI Frameworks & AutoGen

1. 从 Generative AI 到 Agentic AI

Agentic AI 的典型应用包括：

  • Personal assistant；
  • Autonomous robot；
  • Gaming agent；
  • Science agent；
  • Web agent；
  • Software engineering agent。

这里“Agent”并不是一个全新的概念。个人助手、机器人和游戏智能体早已存在。真正的新变化来自 LLM：

  1. Agent 能够通过自然语言理解高层目标；
  2. Agent 可以在很少或没有专门训练的情况下迁移到新任务；
  3. Agent 可以调用代码、搜索、数据库和 API；
  4. Agent 可以通过多轮交互完成长链条任务；
  5. 多个 Agent 可以承担不同角色并协作。

2. Agentic AI 的三类价值

Chi Wang 将 Agentic AI 的价值概括为三个方面。

2.1 Useful Interface：更自然的人机接口

用户不需要提供底层操作步骤，只需要用自然语言说明目标：

“构建一个能够搜索并下载 Hugging Face 模型的网站。”

Agent 可以自行：

  • 分析任务；
  • 查找资料；
  • 安装依赖；
  • 创建文件；
  • 编写与执行代码；
  • 根据错误信息修复程序。

因此，自然语言不再只是聊天接口，也可以成为一种高层编程接口。

2.2 Strong Capability：以较少人工监督完成复杂任务

传统的单次 LLM 调用通常只能生成一次回答，而 Agent workflow 可以：

  • 连续执行多个步骤；
  • 根据环境反馈调整计划；
  • 在失败后重试；
  • 调用外部工具完成模型本身不擅长的任务；
  • 通过验证 Agent 或程序检查中间结果。

2.3 Useful Architecture：新的软件架构

Agentic AI 也是一种新的软件组织方式。

传统程序通常明确编写每一条控制逻辑；Agentic program 则可以将系统划分为多个角色：

  • 每个角色处理相对明确的子任务；
  • 角色之间通过自然语言或结构化消息协作；
  • Agent 内部还可以嵌套其他 Agent；
  • 人类、LLM 和工具都可以作为工作流中的参与者。

因此，Agent 不仅是“更智能的聊天机器人”，还可能成为复杂软件系统的模块化构件。

3. 供应链优化案例：Commander、Writer 与 Safeguard

课程使用供应链优化问题说明 Multi-Agent Workflow：

如果禁止 supplier 1 向 roastery 2 发货，运营成本会发生怎样的变化？

单独询问 ChatGPT 很难完成该任务，因为它需要：

  • 理解用户的业务约束；
  • 修改已有优化模型；
  • 生成 Python 代码；
  • 判断代码是否安全；
  • 执行代码并读取结果；
  • 将数值结果解释为自然语言。

AutoGen 将任务拆分给多个 Agent。

3.1 角色定义

Agent 主要职责
User 提交业务问题
Commander 协调任务和嵌套对话，执行安全代码
Writer 理解问题、生成或修改优化代码、解释执行结果
Safeguard 检查代码是否安全、是否允许执行

3.2 执行流程

如果代码执行失败，错误信息会再次反馈给 Writer，使其修改代码。整个流程可以重复，直到回答完成或达到 timeout。

3.3 AutoGen 风格的核心代码结构

这里最重要的并非具体 API，而是 Agentic Programming 的基本步骤：

  1. Define agents：定义参与者及其能力；
  2. Define interaction patterns：定义哪些 Agent 在什么条件下交互；
  3. Initiate conversation：发送初始任务，其余步骤由工作流推进。

4. 为什么要使用 Multi-Agent

4.1 通过自然迭代提升质量

多个 Agent 可以不断审阅和修正彼此的输出：

相比让单个 Agent 一次完成所有事情，迭代过程更容易发现错误。

4.2 Divide and Conquer

复杂任务可以拆分为相对简单的子任务：

  • Writer 只负责生成代码；
  • Safeguard 只负责安全检查；
  • Commander 只负责协调与执行。

当任务更复杂或模型能力较弱时，合理分工通常更重要。

这并不意味着 Agent 越多越好，而是说明：

当一个角色被要求同时完成生成、检查、推理和执行等多种职责时，任务分解可能提高可靠性。

4.3 Grounding 与 Validation

Agent 不一定由 LLM 驱动，也可以由以下组件实现：

  • Python function；
  • 数据库查询器；
  • 规则系统；
  • 优化求解器；
  • 安全检查器；
  • 人类专家。

这使得系统可以利用外部知识与确定性程序弥补 LLM 的不足。

4.4 模块化与可维护性

如果需要替换 Safeguard，可以：

  • 将 GPT-4 换成较小模型；
  • 改用静态代码分析工具；
  • 改为人工审核；
  • 同时组合多种审核机制。

其他 Agent 不需要发生大规模变化。模块化设计有利于：

  • 理解；
  • 调试；
  • 维护；
  • 扩展；
  • 快速实验；
  • Human-in-the-loop。

5. 一个 Agent Framework 应满足什么要求

一个通用的 Agentic AI Framework 需要同时满足四类需求：

  1. Intuitive unified agentic abstraction
  2. Flexible multi-agent orchestration
  3. Effective implementation of agentic design patterns
  4. Support diverse application needs

5.1 Unified Agentic Abstraction

系统需要统一抽象不同实体：

  • 不同厂商的 LLM；
  • 搜索、代码执行和数据库工具；
  • 不同角色的人类；
  • 专用程序；
  • 由其他 Agent 组合形成的复合 Agent。

5.2 Multi-Agent Orchestration 的关键设计维度

维度 一端 另一端
流程 Static Dynamic
控制语言 Natural Language Programming Language
上下文 Context Sharing Context Isolation
关系 Cooperation Competition
管理 Centralized Decentralized
人类参与 Automation Intervention

这些维度之间没有统一的最优选择。

例如：

  • 对高风险金融操作，可能需要静态流程和明确审批；
  • 对开放式研究问题，可能需要动态规划；
  • 对隐私数据，不同 Agent 之间应隔离上下文；
  • 对创意生成，可以让多个 Agent 竞争并由 Reviewer 选择；
  • 对长流程任务，需要允许人类随时接管。

5.3 Agentic Design Patterns

通用框架需要支持：

  • Conversation；
  • Prompting 与 Reasoning；
  • Tool Use；
  • Planning；
  • Reflection；
  • Chain-of-Thought / Tree-of-Thought；
  • 多模型与多模态；
  • Memory；
  • Human-in-the-loop；
  • Multi-Agent Collaboration。

AutoGen 的设计选择是：

以 Conversation 作为最核心的统一机制，再由对话组合出其他 Agent pattern。

6. AutoGen 的核心抽象

AutoGen 是一个用于构建 Agentic AI 的编程框架。它最初在 FLAML 项目内部发展，之后成为独立项目。

课程将 AutoGen 的使用方式压缩为两步：

对应两个核心概念：

6.1 Conversable Agent

Conversable Agent 可以由不同后端驱动：

  • LLM；
  • Tool；
  • Human input；
  • Code executor；
  • 多种能力的组合。

Agent 的关键不只是“调用某个模型”，而是：

  • 能够接收消息；
  • 根据消息执行内部逻辑；
  • 生成回复或动作；
  • 与其他 Agent 持续交流。

6.2 Conversation Program

Conversation Program 定义 Agent 如何交互，包括：

  • 谁先说话；
  • 下一条消息发送给谁；
  • 对话什么时候结束；
  • 是否触发嵌套对话；
  • 如何总结子对话；
  • 哪些 Agent 可以成为下一位 Speaker；
  • 何时执行工具或请求人类输入。

Agent 本身和对话模式分离，使同一个 Agent 可以在不同工作流中复用。

如果仅让两个 LLM 直接下棋，它们可能生成非法棋步。

解决方法是加入 Chessboard Agent：

  • LLM Agent 提议棋步并进行对话；
  • Chessboard Agent 调用 Python chess library；
  • 工具判断棋步是否合法；
  • 非法棋步被拒绝；
  • LLM 根据反馈重新生成。

工具不需要承担开放式推理，而负责提供确定性约束和环境反馈。

  • 根据任务状态调整交互顺序；

9.2 Adaptive Build

Adaptive Build 进一步让 Agent team 随任务动态变化：

  2. 为当前步骤选择或创建最合适的 Agent；

Part II：Building a Multimodal Knowledge Assistant

11. 从 RAG 系统到 Knowledge Assistant

Jerry Liu 将 Knowledge Assistant 的目标定义为：

接收不同复杂度的任务，在企业数据上完成检索、推理、生成与行动，并返回适合任务的输出。

可能的输入包括：

  • 简单事实问题；
  • 复杂问题；
  • 多文档比较；
  • Research task；
  • 数据分析请求；
  • 行动请求。

可能的输出包括：

  • Short answer；
  • Structured output；
  • Research report；
  • Slide deck；
  • 数据分析结果；
  • Tool action。

这意味着 Knowledge Assistant 不能只做“从向量数据库检索五个片段，然后生成一段答案”。

12. Basic RAG

12.1 基本流程

设文档被切分为若干文本块：

$$ D=\{c_1,c_2,\ldots,c_n\}. $$

使用 Embedding Model 得到向量：

$$ e_i=f_{\text{emb}}(c_i),\qquad e_q=f_{\text{emb}}(q). $$

根据相似度检索 Top-$k$：

$$ R_k(q) = \operatorname{TopK}_{c_i\in D} \operatorname{sim}(e_q,e_i). $$

最后将问题和检索结果放入 LLM：

$$ y=\operatorname{LLM}\big(q,R_k(q)\big). $$

12.2 Basic RAG 的四个主要问题

1. Naive Data Processing

固定按照 500 或 1000 tokens 切分文本，通常忽略：

  • 表格结构；
  • 图片；
  • 图表；
  • 标题层级；
  • 页眉和页脚；
  • 多栏排版；
  • 一个语义单元跨越多个 Chunk 的情况。

2. Poor Query Understanding / Planning

用户的复杂问题可能需要：

  • 拆分子问题；
  • 检索多个文档；
  • 先检索再比较；
  • 汇总整个文档；
  • 迭代补充证据。

Basic RAG 直接将原始 Query 用于一次检索，缺乏规划。

3. No Tool Use

Basic RAG 一般不能：

  • 查询 SQL；
  • 运行 Python；
  • 调用 API；
  • 进行数据分析；
  • 发送邮件或执行其他行动。

4. Stateless / No Memory

每次问答彼此独立：

  • 不记住用户偏好；
  • 不记住此前检索到的信息；
  • 不维护长期任务状态；
  • 无法形成个性化知识助手。

课程提出四个核心方向：

  1. High-quality Multimodal RAG

  1. Complex Output Generation

  1. Agentic Reasoning over Complex Inputs

14. 数据质量是系统的基础

Any LLM App is only as good as your data.

即使模型很强，如果输入给模型的数据已经被错误解析，模型仍然难以生成可靠答案。

生产级 LLM 应用中的 ETL for LLM 主要包括：

  • Parsing；
  • Chunking；
  • Indexing。

复杂文档常见问题：

  • Embedded tables；
  • Charts and images；
  • Irregular layouts；
  • Headers / footers；
  • Scanned pages；
  • 多栏排版；
  • 文本和视觉元素互相引用。

这些内容广泛存在于：

  • 年报和 10-K；
  • 研究论文；
  • PowerPoint；
  • Excel；
  • 表单；
  • 咨询报告；
  • 法律和财务文档。

15. LLM-Native Document Parser

理想的 GenAI-native Parser 应当把复杂文档转换为适合下游应用的结构化节点。

15.1 Requirements

  • 准确解析表格，并生成文本或半结构化表示；
  • 将正文切分为语义连贯的 Chunk；
  • 提取图片、图表和 Diagram；
  • 为视觉元素生成结构化信息或 Image Chunk；
  • 自动提取 Metadata；
  • 保留不同元素之间的层级和引用关系。

15.2 Non-Requirements

对于 RAG 和 Agent 应用，并不一定需要：

  • 为每个元素生成极其详细的 JSON；
  • 精确恢复每个字符的 Bounding Box；
  • 完整重建文档编辑格式。

Parser 的目标不是复刻 PDF 编辑器，而是生成适合检索、理解和推理的表示。

16. Advanced Parsing + Advanced Indexing

直接对原始表格和图片做 Embedding 往往并不理想。

更有效的方法是：

  1. 将文档解析为 Text、Table、Image、Diagram 等 Source Element；
  2. 为每个 Source Element 生成一个或多个文本表示；
  3. 对这些文本表示建立索引；
  4. 检索到文本表示后，沿 Metadata 找回原始 Source Element；
  5. 将真正的文本、表格或图片交给 Multimodal LLM。

设原始元素为 $x_i$，为其生成多个可检索表示：

$$ g(x_i)=\{r_{i1},r_{i2},\ldots,r_{im}\}. $$

每个表示保留对原始元素的引用：

$$ r_{ij}\longrightarrow x_i. $$

查询时先检索表示：

$$ \mathcal R(q) = \operatorname{TopK}_{r_{ij}} \operatorname{sim}\big(f(q),f(r_{ij})\big), $$

再取得其对应的 Source Element：

$$ \mathcal C(q) = \{\operatorname{source}(r)\mid r\in\mathcal R(q)\}. $$

这种方法的优势：

  • 用成熟的 Text Embedding 检索；
  • 一个表格或图片可以有多个语义描述；
  • 最终仍将原始图片或表格交给 Multimodal LLM；
  • 可以做 Hierarchical / Recursive Retrieval。

17. Multimodal RAG Pipeline

17.1 Indexing

  1. 使用 Parser 将文档拆分为 Text Chunk 和 Image Chunk；
  2. 通过 Metadata 将文本表示与图片关联；
  3. 对文本 Chunk 或图片的文本描述做 Embedding；
  4. 将它们写入 Vector Index。

17.2 Retrieval

  1. 使用 Query Embedding 检索相关文本表示；
  2. 根据 Metadata 找回对应图片或表格；
  3. 同时将 Text 和 Image 输入 Multimodal LLM；
  4. 由模型进行最终 Synthesis。

“Multimodal RAG”不只是对图片使用 CLIP Embedding。另一条常见路径是：

  • 使用视觉模型为图片生成多个文字描述；
  • 对文字描述建索引；
  • 检索后把原图送给 Multimodal LLM。

18. 从 Chat Response 到 Complex Output Generation

企业用户常常不只需要一段聊天回答，而希望系统生成可直接使用的产物：

  • Report；
  • Slide deck；
  • Structured JSON；
  • Code repository；
  • 数据分析结果；
  • 表格；
  • Tool action。

这会显著提高 Agent 应用的潜在价值，因为它节省的不只是“搜索信息”的时间，还可能自动完成知识工作的部分流程。

但 Action-taking 和复杂输出也带来风险：

  • 输出必须更加可靠；
  • 执行动作可能产生真实后果；
  • 系统需要明确权限边界；
  • 需要 Human-in-the-loop 建立信任；
  • 需要保留来源、日志和中间状态。

19. 为什么复杂问题需要 Agentic RAG

Naive RAG 适合 Pointed Question，例如：

某个具体指标是多少？

但难以处理以下问题。

19.1 Summarization Question

总结整份公司 10-K 年报。

这需要覆盖整份文档，而不是只取与 Query 最相似的五个 Chunk。

19.2 Comparison Question

比较候选人 A 和候选人 B 的开源贡献。

这需要分别检索两个实体，并按照共同维度进行比较。

19.3 Multi-Part Question

分析文章 A 中支持 X 的论据和文章 B 中支持 Y 的论据，按照内部 Style Guide 生成表格，再给出结论。

该任务同时包含：

  • 多源检索；
  • 信息抽取；
  • 结构化生成；
  • 规则遵循；
  • 综合推理。

19.4 Research Task

生成当前 supervised fine-tuning 技术的研究综述。

这需要：

  • Query Planning；
  • 多轮 Retrieval；
  • 去重与筛选；
  • 证据整合；
  • 报告规划；
  • 引用管理；
  • Review。

因此需要在 RAG 之上增加 Agent reasoning layer。

20.2 Preplanning / DAG

另一类方法先生成任务依赖图：

$$ G=(V,E), $$

其中：

  • $V$ 表示子任务；
  • $E$ 表示依赖关系。

能够并行执行的节点可以同时运行，之后根据结果 Replan。

20.3 Memory

Memory 可以保存：

  • 用户偏好；
  • 历史对话；
  • 中间检索结果；
  • 任务状态；
  • 已尝试但失败的方法；
  • 长期知识。

20.4 Query Decomposition

复杂 Query 可以拆解为：

$$ q\longrightarrow \{q_1,q_2,\ldots,q_m\}. $$

子问题可以串行或并行执行，再进行聚合：

$$ y=\operatorname{Synthesize}\big(y_1,y_2,\ldots,y_m\big). $$

21. Agentic RAG

Agentic RAG 的关键观点是：

每一个 Data Interface 都可以被视为 Tool，Agent 决定何时、为何以及如何调用这些 Tool。

例如：

  • Vector RAG Tool；
  • SQL Tool；
  • Web Search Tool；
  • Python Tool；
  • Document Summary Tool；
  • Table Analysis Tool；
  • Image Understanding Tool。

与 Basic RAG 的区别：

Basic RAG Agentic RAG
Query 直接进入 Retriever Agent 先理解并规划 Query
固定调用一次检索 可多次调用不同 Tool
固定 Top-$k$ 可根据中间结果调整检索
无记忆 可维护状态和 Memory
只生成答案 可生成报告或执行动作
无自我检查 可加入 Reflection / Validation

22. Constrained Flow 与 Unconstrained Flow

22.1 Constrained Flow

开发者预先规定主要控制逻辑：

特点：

  • More reliable；
  • Less expressive；
  • 更容易测试；
  • 成本和延迟更可控；
  • 适合明确的生产场景。

22.2 Unconstrained Flow

Agent Orchestrator 自主决定：

  • 调用什么工具；
  • 调用顺序；
  • 是否循环；
  • 何时结束；
  • 是否重新规划。

特点：

  • More expressive；
  • 能处理更多未预见任务；
  • Less reliable；
  • 更容易产生循环或偏离目标；
  • Token 和工具成本较高。

22.3 实践中的选择

课程给出的经验是：

  • 不要因为“Agent 更自主”就默认选择完全开放流程；
  • 生产场景常从较约束的 Workflow 开始；
  • 只在确实需要动态决策的局部开放自主性；
  • ReAct 初始可只提供约 4～5 个 Tool；
  • Tool 数量过多会增加选择难度、Prompt 长度和错误率。

因此更合理的设计通常是：

用代码确定不可违反的边界，用 LLM 处理难以穷举的语义决策。

23. Multimodal Report Generation

课程给出的示例架构包含 Researcher 与 Writer。

24.1 Researcher

Researcher 负责：

  • 分析任务；
  • 调用 RAG；
  • 检索文本、图片和表格；
  • 将相关证据放入 Data Cache；
  • 保留来源与 Metadata。

24.2 Writer

Writer 读取 Data Cache，生成结构化输出：

最终结果可以是文本和图片交错的报告，甚至进一步生成 Slide Deck。

24.3 Reviewer（可选）

Reviewer 可以检查：

  • 内容是否覆盖任务；
  • 引用是否支持结论；
  • 图片是否与段落相关；
  • 格式是否符合要求；
  • 是否遗漏关键证据；
  • 是否存在幻觉。

这一结构与 AutoGen 的 Writer–Critic / Nested Review 非常相似，说明不同框架背后的系统思想具有共通性。

24. Running Agents in Production

Agent prototype 通常始于 Jupyter Notebook，但生产系统还需要解决：

  • Encapsulation and reuse；
  • 标准化 Agent 通信接口；
  • 多用户和多 Session；
  • Agent 数量扩展；
  • Human-in-the-loop；
  • Debugging 和 Observability；
  • 权限、安全和资源管理。

Agent 在信息不足或风险较高时可以暂停：

这比“失败后从头运行”更适合长任务，也有助于逐步建立用户信任。

25. 核心结论

  1. 未来 AI 应用的单位可能不再是单一模型，而是 Compound AI System。
  2. Agentic AI 同时是一种交互方式、自动化能力和软件架构。
  3. Multi-Agent 的价值来自合理分工、迭代、Grounding 和 Validation，而不是单纯增加 Agent 数量。
  4. AutoGen 用 Conversable Agent 与 Conversation Program 统一 LLM、工具和人类。
  5. Nested Chat 使一个 Agent 能够由内部 Agent team 递归构成。
  6. Group Chat 在动态规划与流程约束之间提供折中。
  7. Basic RAG 的上限通常首先受数据解析与检索质量限制。
  8. Multimodal RAG 需要保存文本、表格、图片及其关联关系。
  9. Agentic RAG 将数据接口视为 Tool，由 Agent 进行 Query Planning、Tool Selection、Memory 和 Reflection。
  10. 生产系统通常更偏好“受约束的 Agent 自主性”，而不是完全自由的循环。
  11. 真正的 Agent 部署还需要 Message Queue、Service API、Observability、状态管理和 Human-in-the-loop。
  12. Agent workflow 的设计本质上是在质量、成本、延迟、可靠性和人工投入之间进行权衡。
