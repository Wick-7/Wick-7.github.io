大型语言模型的学习历程：Pre-training、SFT 与 RLHF

以大金老师讲解和ppt为主，穿插Andrej Karpathy的Deep Dive into LLMs like ChatGPT

1. 整体框架：大型语言模型如何一步步变成聊天助手

大型语言模型的训练分为三个阶段：

阶段 全称 主要目标 人类提供什么 形象比喻
Pre-training Pre-training 熟悉语言、吸收世界知识，学会基本的文字接龙 大量原始文本，人类直接标注较少 学龄前：看到什么学什么
SFT Supervised Fine-Tuning 学会按照指令回答，形成有用的助手行为 问题与标准答案 上学校：老师给标准答案
RLHF Reinforcement Learning with Human Feedback 根据人类偏好改善完整回答 对答案的偏好、排序或分数 出社会：别人只告诉你好不好

1.1 Alignment 的含义

Alignment（对齐） 是让模型的行为符合人类的需求、偏好和价值观。

狭义文献中，Alignment 有时主要指 RLHF；但从课程的定义看：

  • SFT 用人类提供的标准答案进行对齐；
  • RLHF 用人类提供的反馈进行对齐；
  • 因此，SFT 和 RLHF 都属于 Alignment。

1.2 三阶段的关系不是重新训练，而是连续初始化

每个阶段都继承前一阶段的参数：

这样做的原因是：前一阶段已经把参数带到一个较好的区域，后一阶段不需要从随机状态重新学习。

1.3 三阶段真正改变的主要是“教材”

从机器学习的三个步骤看：

  1. 定义学习目标与训练数据；
  2. 确定模型的函数搜索范围，例如 Transformer；
  3. 使用优化器寻找较好的参数，例如 Adam。

Pre-training、SFT 和 RLHF 通常沿用同一模型架构，参数也需要能够继承。因此三阶段最核心的差异是：训练数据和评价信号不同。

不过，严格来说第三阶段还改变了目标函数：前两阶段最小化 token-level Cross Entropy，第三阶段最大化整段回答的 Reward。

2. 三个阶段共同的基础：都在进行文字接龙

2.1 文字接龙本质上是分类问题

在图像分类中，模型从若干类别中选择一类；在语言模型中，词表中的每一个 Token 都是一个类别。
给定前文，模型输出下一个 Token 的概率分布。

2.2 自回归概率分解

一句话由多个 Token 构成。自回归语言模型把整段文本的概率分解为逐 Token 的条件概率：

$$ P_\theta(x_1,x_2,\ldots,x_T) = \prod_{t=1}^{T}P_\theta(x_t\mid x_{<t}) $$

其中：

  • $x_t$：第 $t$ 个 Token；
  • $x_{<t}$：第 $t$ 个 Token 之前的全部上下文；
  • $θ$：语言模型的参数。

2.3 Softmax 输出

模型先对词表中的每个 Token 输出一个 logit，再通过 Softmax 变成概率：

$$ p_\theta(v\mid x_{<t}) = \frac{\exp(z_v)}{\sum_{u\in\mathcal V}\exp(z_u)} $$

其中 $V$ 是词表，词表大小就是分类类别数，现代模型通常有数万到数十万个 Token 类别。

2.4 Cross Entropy / Negative Log-Likelihood

若真实的下一个 Token 是 $x_t$，单个位置的损失为：

$$ \ell_t = -\log P_\theta(x_t\mid x_{<t}) $$

整段训练文本的损失为：

$$ L = -\sum_{t=1}^{T}\log P_\theta(x_t\mid x_{<t}) $$

模型使用 Gradient Descent 调整参数，使正确 Token 的概率增大，从而降低 Cross Entropy。

3. 从文字到 Token：训练与推理如何完成“文字接龙”

3.1 为什么需要 Tokenization

课件把语言模型描述为一个拥有巨大类别数的分类器。要真正送入 Transformer，文字还需要先经过 Tokenization，被转换成有限符号集合中的一维 Token 序列。神经网络处理的是这些 Token ID，而不是人眼直接看到的字词。

一种极端做法是使用 bit：词表只有 $0$ 和 $1$，但序列会极长。另一种做法是使用更大的符号集合，换取更短的序列。因此 Tokenization 本质上在权衡：

  • Vocabulary Size；
  • Sequence Length；
  • 对不同语言和字符的表示效率。

3.2 Byte Pair Encoding 的直觉

常见 Tokenizer 会从 byte 出发，反复寻找最常出现的相邻符号对，把它们合并成新 Token。高频字符串会成为一个整体 Token，低频字符串则被拆成多个 Token。
因此：

  • Token 不一定等于一个字或一个单词；
  • 大小写、空格、标点都可能改变切分；
  • 模型实际看到的是 Token ID，而不是人眼中的字符；
  • 拼写、逐字符操作和罕见语言可能因此更困难。

3.3 神经网络训练的输入与输出

输入是长度不超过 Context Window 的 Token 序列，输出是下一个 Token 的概率分布。

训练时不断从语料中截取窗口：

训练初期参数随机，输出近似随机；经过大量梯度更新后，概率分布逐渐符合训练语料中的统计规律。

3.4 Inference：模型如何生成文本

训练完成后，参数固定，进入 Inference 阶段：

  1. 输入已有 Token；
  2. 模型预测下一个 Token 的概率；
  3. 从概率分布中选择或采样一个 Token；
  4. 将新 Token 加入上下文；
  5. 重复以上过程，直到生成结束符或达到长度限制。

因此，一次完整回答是逐 Token 生成的。只要中间某个 Token 改变，之后的上下文就会随之改变，后续整条生成路径也可能不同。这正是为什么 SFT 或 RLHF 只需轻微调整少数关键 Token 的概率，就可能明显改变整段回答。

Part I：Pre-training

4. Pre-training 的目标

Pre-training 是整个训练流程的起点，主要让模型获得两类能力。

4.1 语言知识

模型需要学习：

  • 语法结构；
  • 常见搭配；
  • 不同词形和句型；
  • 哪些 Token 序列符合自然语言规律。

语言知识相对容易饱和。如图片的实验显示，当语料达到一定规模后，句法知识的提升会逐渐趋缓。

4.2 世界知识

世界知识近乎无穷，并且事实、条件与语境相互关联，所以比单纯语法需要更多、更丰富的训练数据。

5. 为什么 Pre-training 属于 Self-supervised Learning

假设网络中有一句话：

它可以自动产生多条训练样本：

正确答案直接来自原始文本本身，不需要人类逐条写标签。因此它属于：

Self-supervised Learning（自监督学习，课件称自督导式学习）。

注意：Self-supervised 不代表完全没有人工参与。数据爬取、过滤、去重、版权与隐私处理仍需要大量工程设计，只是 Token 标签可以从文本自动构造。

6. 预训练数据从哪里来

6.1 数据规模

课件以 LLaMA 3、DeepSeek-V3 和 FineWeb 为例，展示了约 15T tokens 的训练规模。

6.2 从原始网页到训练语料：生产级数据处理流水线

原始网页不能直接用于训练。以 Common Crawl / FineWeb 类流程为例，常见步骤包括：

  1. URL Filtering：删除恶意网站、垃圾站、成人内容、低质量营销站等；
  2. Text Extraction：从 HTML 中提取正文，去掉导航栏、CSS、脚本和模板文本；
  3. Language Filtering：识别语言，并根据模型目标确定各语言的比例；
  4. Heuristic Filtering：根据长度、字符比例、重复符号、可读性等规则过滤；
  5. Deduplication：去除完全重复或近似重复的数据；
  6. Model-based Filtering：用分类模型判断文本质量；
  7. PII Removal：删除地址、证件号等 Personally Identifiable Information；
  8. Mixture Design：控制网页、书籍、代码、论文、对话和不同语言之间的比例。

数据组成会直接塑造模型能力。例如某种语言占比过低，模型在该语言上的能力通常也会较弱。

所以“Self-supervised”只表示下一个 Token 的标签由文本自动产生，并不表示数据无需人工工程。真正昂贵的环节常常是决定抓取什么、删除什么、如何去重，以及各种来源应按什么比例混合。

7. 模型不是“把互联网逐字背下来”

Pre-training 的目标是通过文本学习预测规律，而不是建立一个可以逐字检索的数据库。

因此模型可能：

  • 知道一篇小说的主要人物和情节；
  • 能写出风格相似、内容大致合理的续写；
  • 却不能准确复述原文。

更合适的理解是：模型把文本中的统计规律和知识压缩进参数。压缩与生成过程可能产生信息损失，因此会出现：

  • 细节遗漏；
  • 原文混写；
  • 看似合理但并非真实原文的内容；
  • Hallucination。

会概括、会续写，不等于能够忠实检索。

模型参数中的知识更像经过压缩后的长期记忆：高频、反复出现的内容通常更牢固，罕见细节则可能只留下模糊印象。若把原始资料直接放入 Context Window，模型可以依据当前上下文作答，这更接近临时的 Working Memory。因此，对精确事实而言，“从参数中回忆”和“根据上下文读取”是两种可靠性不同的方式。

8. 数据越多越好吗：模型、数据与算力的平衡

8.1 两种资源的作用

  • 更多数据：相当于读过更多教材，可降低过拟合并增加知识覆盖；
  • 更大模型：相当于函数搜索空间更大，表达能力更强；
  • 更多训练计算：才能让大模型充分读取和消化更多数据。

算力有限时，不能无限增大模型和数据，必须进行配置。

8.2 Chinchilla Scaling Law

在固定训练计算量下，模型参数量与训练 Token 数存在较优配比。

模型过大但数据不足，参数没有得到充分训练；模型过小但数据过多，模型容量不足。二者都不能充分利用计算预算。

可以用一个粗略的计算量关系理解：

$$ C \propto N\times D $$

其中：

  • $C$：训练计算量；
  • $N$：模型参数量；
  • $D$：训练 Token 数。

实际比例和常数取决于架构及训练设置，但核心思想是：在固定算力下，需要共同配置模型规模和数据规模。

9. 数据质量往往比单纯增加数据更重要

9.1 低质量数据可能破坏训练

课件用包含大量重复字母的异常 Reddit 文本举例：某些极端、无意义的序列可能导致 Loss 突然升高或训练不稳定。

因此网络语料通常经过层层过滤。课件引用 DataComp 的流程说明：

  • Heuristic Filtering 会首先删除大量原始数据；
  • Deduplication 继续去除重复内容；
  • Model-based Filtering 只保留被判断为高质量的部分；
  • 最终保留比例可能很低，但绝对规模仍然很大。

9.2 高质量数据的价值

在相同算力下，高质量语料通常可以取得更好效果；达到相同效果时，也可能使用更少数据和算力。

因此 Pre-training 数据设计要同时考虑：

Quantity；Quality；Diversity；Coverage；Safety；Privacy；Duplication；Language Balance。

9.3 Rephrasing the Web

课件介绍了一种思路：先让已有 LLM 把网页内容重新表述，使文本更有组织、更易读，再用改写后的数据训练新模型。

它说明“数据清洗”不一定只是删除数据，也可以进行：

  • 重写；
  • 结构化；
  • 降噪；
  • 统一表达风格。

不过这种方法也可能把已有模型的偏差、风格和错误传播给新模型。

10. 为什么 Base Model 往往不会好好回答问题

Pre-training 只要求模型模仿互联网文本分布。此时的 Base Model 更像一个 Internet Document Simulator：给它一段开头，它会继续生成网络上可能出现的后文，而不是天然把自己当作正在服务用户的助手。因此，互联网中“问题”后面不一定紧跟正确答案，可能是：

  • 选择题选项；
  • 网友讨论；
  • “知道的人请留言”；
  • 另一个问题；
  • 不完整或错误的答案。

所以 Base Model 即使拥有相关知识，也未必知道：

  • 用户是在向它提问；
  • 它应该直接给出答案；
  • 什么时候应该停止；
  • 什么语气和格式更有帮助。

10.1 Base Model 是“璞玉”

Pre-trained Model 可能已经给正确答案分配了一定概率，只是该概率未必最高。SFT 和 RLHF 的重要作用，是改变输出分布，让有用、正确、符合人类期待的回答更容易被选中。

因此 SFT 也叫 Instruction Fine-Tuning：模型根据 Instruction 学习如何产生合适 Response。

从计算形式看，SFT 并没有换掉“预测下一个 Token”的核心算法；真正变化的是教材从一般网络文本变成了经过设计的对话示范。模型通过模仿这些示范，逐渐学会 Chat Template、回答边界、助手语气和结束位置。

11.1 SFT Loss

设第 $i$ 条样本的问题为 $x_i$，回答为 $y_i$，只在 Assistant 回答部分计算损失：

$$ L_{\mathrm{SFT}} = -\sum_i\sum_{t\in\mathcal A_i} \log P_\theta(y_{i,t}\mid x_i,y_{i,<t}) $$

其中 $A_i$ 表示第 $i$ 条数据中属于 Assistant 回答的 Token 位置。

很多训练实现会 Mask 掉 User Prompt 的 Loss，因为目标是学习“如何回答”，而不是让模型复述用户问题。

12. SFT 为什么能让模型“脱胎换骨”

SFT 学到的重点包括：

  • 识别问题和指令；
  • 使用助手式语气；
  • 输出正确的格式；
  • 直接回答，而不是继续模仿网页；
  • 在适当位置生成结束符；
  • 遵守部分安全和行为规范。

课件强调：SFT 的成功建立在 Pre-training 之上。没有充分预训练，只靠少量问答数据，模型容易过拟合成表面规则。

13. Pre-training 如何帮助 SFT 举一反三

课件引用 Physics of Language Models 的实验说明：

  • 若某个事实在 Pre-training 中只以一种固定顺序出现一次，模型可能只学到局部 Token 邻接关系；
  • 如果同一知识从多个角度、不同措辞反复出现，模型更容易把人物、属性和关系连接起来；
  • SFT 再教模型“看到某类问题应该如何调用关系”，它才能迁移到未在 SFT 中出现的实体。

核心结论：

同一个知识以多种表达反复出现，比单次机械出现更有利于形成可调用、可泛化的内部表示。

因此 Pre-training 的“大量”不仅表示事实数量多，也表示同一知识在不同上下文中的多样化表达多。

14. SFT 通常更像激活与规范，而不是灌入大量新知识

课件把问题分为：

  • Highly Known：模型用 Greedy Decoding 就能回答；
  • Maybe Known：换一种问法后可以回答；
  • Weakly Known：采样多次时偶尔答对；
  • Unknown：无论如何采样都很难生成正确答案。

实验结论是，SFT 对 Maybe Known 数据往往最有效，因为模型已有相关知识，只需要学习：

  • 如何理解问题；
  • 如何调用已有知识；
  • 如何以正确方式表达。

对于完全 Unknown 的事实，少量 SFT 容易变成死记硬背，泛化较差，训练过久甚至可能损害整体表现。

因此可以概括为：

这不是绝对规律：SFT 仍可能注入某些知识，但它通常不是大规模知识获取的高效阶段。

15. Alignment 往往只需改变少量关键概率

课件介绍的研究把 Alignment 前后的 Token 分成：

  • Unshift：Alignment 前后都是概率第一；
  • Marginal：原来排名第二或第三，之后升到第一；
  • Shifted：原来排名较后，之后升到第一。

结果显示，许多 Token 的相对排序并未发生巨大变化，显著变化的常常包括：

  • 连接和表达风格相关 Token；
  • 结束符 $\texttt{[END]}$。

这可以解释为什么参数概率只发生局部变化，生成结果却可能天差地别：自回归生成具有路径依赖，只要早期一个 Token 改变，后续上下文和整段输出都会改变。

16. Pre-training 会留下长期影响

即使经过 SFT 和 RLHF，模型仍可能保留 Pre-training 数据分布留下的“灰烬”。

课件以 ROT Cipher 为例：模型对训练资料中更常出现的 ROT13 表现更好，却不一定能迁移到同等难度、但在语料中较少出现的其他位移。

这说明：

  • 模型表现不只由任务抽象难度决定；
  • 训练语料中的出现频率和表达方式会留下强烈偏置；
  • “在一个例子上会做”不一定表示已经掌握通用算法。

17. SFT 的两条路线：专才与通才

17.1 路线一：打造专才

给模型单一任务的数据，例如：

  • 翻译；
  • 摘要；
  • 分类；
  • 文本修订。

模型在特定任务上表现好，但不一定能处理其他任务。BERT 时代的任务微调是典型思路。

17.2 路线二：打造通才

混合大量不同任务的数据，让模型学习一般性的 Instruction Following，并期待迁移到未见任务。

FLAN 是早期代表性工作之一，使用大量任务进行 Instruction Fine-Tuning，目标是让单一模型处理多类任务。

现代 Chat Assistant 主要采用通才路线，但内部也可能加入代码、数学、安全等领域的专门数据。

18. SFT 数据：质量、长度、复杂度与多样性

SFT 通常不需要达到 Pre-training 那样的数量级。课件列举：

  • InstructGPT 使用的 SFT 数据只有一万多条量级；
  • LLaMA 2 报告数万条高质量 SFT 数据；
  • LIMA 仅使用约 1000 条精挑细选的数据，也能形成可用模型。

核心并不是“数据越少越好”，而是：

SFT 的边际收益很快受数据质量制约，百万条低质量样本未必优于数万条高质量样本。

18.1 数据选择仍是开放问题

课件展示了多种现象：

  • 某些看似奇怪的问题数据也可能有效；
  • 只选较长样本有时优于使用全部数据；
  • 但答案长度不是唯一解释；
  • 可以训练 Complexity Scorer 和 Quality Scorer；
  • 还需要 Diversity Selection，避免样本高度相似。

一个系统化选择流程可以写成：

19. Knowledge Distillation 与合成 SFT 数据

人工写高质量长答案成本很高，因此可以让更强模型担任 Teacher：

这种做法叫 Knowledge Distillation。Alpaca、Vicuna 等早期工作都采用了类似思路。

19.1 优势

  • 降低人工写答案成本；
  • 快速生成多领域数据；
  • 可以把大模型的回答风格迁移给小模型；
  • 结合筛选后，可以获得高质量小数据集。

19.2 局限

  • Teacher 的错误会被 Student 学到；
  • 数据生成、清洗和 API 调用本身仍有成本；
  • 模型可能只学到 Teacher 的风格，而非真正提高基础知识；
  • 可能造成模型之间输出趋同。

19.3 Non-instructional Fine-Tuning 与 Response Tuning

课件进一步介绍了一些反直觉现象：

  • 不一定需要人工准备标准问题，可以截取普通文本前半句，让强模型续写，再用续写数据训练；
  • Response Tuning 甚至只训练 Assistant 风格的“回答”，不提供对应问题，也能激活部分 Instruction Following 能力；
  • 提高结束符概率、降低重复等简单解码调整，也能让 Base Model 的行为更像对齐模型。

这些结果支持一个观点：

Base Model 已包含大量潜在能力，SFT 有时更像施加一种“解锁咒语”，把模型推向助手式输出分布。

但这些方法通常不能完全取代标准、高质量的 Instruction Tuning。

Part III：RLHF

20. RLHF 与 SFT 的人类参与方式不同

20.1 SFT：人类负责生成标准答案

人类负担较重，特别是在数学证明、诗歌、复杂写作等任务中，写出理想答案可能非常困难。

20.2 RLHF：模型先回答，人类只判断好坏

通常“判断几个答案哪个更好”比“从零写出最佳答案”容易，这体现了 Discriminator–Generator Gap：辨别一个答案的质量，往往比从零生成同等质量的答案更容易。

21. 前两阶段与第三阶段的核心差别：Token Loss 与 Sequence Reward

21.1 Pre-training / SFT：每个 Token 都有标准答案

损失是每个 Token Cross Entropy 的总和：

$$ L_{\mathrm{sup}} = \sum_{i=1}^{N}\ell_i $$

这里每个 $ℓ_i$ 对应一个 Token 的预测误差。

21.2 RLHF：评价整段回答

模型先生成完整回答 $y$，人类或 Reward Model 给出一个整体分数 $r(x,y)$。

训练目标是最大化期望 Reward：

$$ J(\theta) = \mathbb E_{x\sim\mathcal D,\,y\sim\pi_\theta(\cdot\mid x)} \left[r(x,y)\right] $$

若统一写成要最小化的 Loss：

$$ L_{\mathrm{RL}} = -\mathbb E\left[r(x,y)\right] $$

21.3 为什么 Sequence-level Reward 更符合人类判断

课件举例：标准回答是“我不可以教你”。

候选回答：

若逐 Token 与标准答案机械对齐，A 可能只错一个字，而 B 的 Token 位置整体错位，Cross Entropy 反而更大；但人类明显认为 B 更好。

因此：

  • Token-level Loss 关注局部模仿；
  • Sequence-level Reward 关注完整语义和最终效果。

课件将其概括为：

  • 前两阶段：比较重视过程中的每个 Token 是否一致；
  • 第三阶段：只问完整结果好不好，不要求与某个标准答案逐字一致。

22. 为什么 RL 比监督学习更难优化

22.1 Reward 不可直接微分

前两阶段知道标准 Token，因此模型概率变化如何影响 Loss 可以通过 Backpropagation 精确计算。

RL 中，人类只给一个离散反馈，例如 $+1$ 或 $-1$。模型参数稍微变化后，人类评分会怎样变化并不直接可知，且评分信号通常不连续。

22.2 Credit Assignment Problem

一段回答可能有数百个 Token，但只得到一个总分。模型不知道：

  • 哪个 Token 导致低分；
  • 是事实错误、格式错误还是语气问题；
  • 哪一段推理值得保留；
  • 哪个局部选择应该改变。

这就是 Credit Assignment：如何把最终 Reward 归因到生成过程中的动作。

22.3 Reward 稀疏且有噪声

  • 人类评价可能不一致；
  • 同一回答在不同偏好下得分不同；
  • 二元赞/踩信号信息量有限；
  • 很多中间 Token 没有即时 Reward。

因此 RL 算法必须用采样和统计估计来获得更新方向。

23. RL 的优势：更贴近结果，也更能因材施教

23.1 评价目标与真实需求更接近

只要 Reward 设计得合理，优化 Reward 就直接对应：

  • 回答是否正确；
  • 是否有帮助；
  • 是否安全；
  • 是否符合格式；
  • 是否满足人类偏好。

23.2 训练样本由模型自己产生

SFT 的标准答案可能采用人类认为最佳、但模型很难模仿的解法。RL 则从模型自己能够生成的候选中选择较优路径，再提高这些路径的概率。

所以课程把 RL 描述为“因材施教”：

  • 不同模型会生成不同错误；
  • 收到的反馈针对自身当前弱点；
  • 模型可以从自己可达到的解法逐步改进。

可以把三阶段理解为：

  • Pre-training：广泛阅读教材，形成语言与知识基础；
  • SFT：模仿专家写好的示范答案，学会标准作答方式；
  • RL：亲自做练习题，根据结果反复试错，找到模型自己更容易执行的解题路径。

24. RL 术语与语言模型的对应关系

Reinforcement Learning 语言模型中的含义
Agent / Actor 语言模型
Policy 语言模型的生成概率分布
Observation / State 当前 Prompt 与已生成的 Token
Action 选择下一个 Token
Environment 用户、工具、验证器或对话环境
Reward 对完整回答或中间行为的评分
Trajectory / Rollout 一次完整生成过程

语言模型策略可以写成：

$$ \pi_\theta(y\mid x) = \prod_{t=1}^{T}P_\theta(y_t\mid x,y_{<t}) $$

其对数概率为：

$$ \log\pi_\theta(y\mid x) = \sum_{t=1}^{T}\log P_\theta(y_t\mid x,y_{<t}) $$

这与下围棋的类比是：

  • 棋盘状态对应当前上下文；
  • 下一手落子对应下一个 Token；
  • 一局棋对应一段完整回答；
  • 最终胜负对应回答 Reward。

25. Policy Gradient 的核心精神

Policy Gradient 的基本形式是：

$$ \nabla_\theta J(\theta) = \mathbb E \left[ R\,\nabla_\theta\log\pi_\theta(y\mid x) \right] $$

直觉：

  • 若回答得到 Positive Reward，提高这段回答的生成概率；
  • 若回答得到 Negative Reward，降低这段回答的生成概率。

课件用“拉近 / 拉远”解释：

实际算法会加入 Baseline、Advantage、Clipping、KL 约束等机制，以减少方差并避免策略更新过猛。课程提到的相关方法包括：

  • Policy Gradient；
  • PPO；
  • DPO；
  • KTO；
  • GRPO。

这些方法实现细节不同，但共同目标都是：利用偏好或 Reward 调整模型的输出概率。

26. Reward Model：把有限人类反馈扩展成大量评分

直接让人类评价每一次模型生成不现实，所以经典 RLHF 往往先训练一个 Reward Model。这里利用的仍是“批评通常比创作容易”：人类先提供少量比较结果，模型再学习近似这种判断。

流程：

Reward Model 的输入是 Prompt 与候选回答，输出一个标量分数：

$$ r_\phi(x,y)\in\mathbb R $$

若人类更喜欢 $y+$ 而不是 $y-$，一种常见的偏好损失是：

$$ L_{\mathrm{RM}} = -\log\sigma \left( r_\phi(x,y^+)-r_\phi(x,y^-) \right) $$

它要求 Reward Model 给较优回答更高的分数。

27. RLHF、RLAIF 与 Self-Rewarding

27.1 RLHF

Human 提供偏好数据，Reward Model 学习模拟人类评分，再用于训练生成模型。

27.2 RLAIF

把 Human Feedback 的部分工作交给 AI：

  • 由强模型评价弱模型的答案；
  • 由 AI 根据原则或规则给出偏好；
  • 降低大规模人工评分成本。

27.3 Self-Rewarding

生成与评价甚至可以由同一个模型或同系列模型完成。其可能有效的原因是：

生成正确答案很难，但识别明显错误有时相对容易。

不过它也可能导致自我确认偏差：模型看不出自己的系统性错误，或奖励自己偏好的风格。

28. Reward 的可验证性与 Reward Hacking

28.1 Verifiable Domains

数学、代码等任务往往有自动验证器：

  • 最终数值是否正确；
  • 程序能否通过测试；
  • 证明是否满足形式检查。

Reward 更可靠，模型可大量试错。RL 可以发现不同于人类示范、但确实有效的解题策略。

28.2 Unverifiable Domains

诗歌、摘要、幽默、写作风格等没有简单的客观判定函数，需要人类偏好或 Reward Model。

RLHF 使这些领域也能进行 RL，但 Reward Model 只是人类偏好的有损模拟。

28.3 Reward Hacking

如果持续优化一个不完美 Reward Model，策略可能找到“骗高分”的输入，而不是产生真正更好的回答。

这称为 Reward Hacking 或对 Reward Model 的 Gaming。可能表现为：

  • 无意义重复却获得高分；
  • 利用 Reward Model 没见过的异常模式；
  • 形式上符合评分器偏好，但内容质量下降。

因此 RLHF 通常需要：

  • 限制更新幅度；
  • 监控真实人类评价；
  • 持续改进 Reward Model；
  • 防止策略偏离参考模型过远；
  • 在效果开始恶化前 Early Stop。
