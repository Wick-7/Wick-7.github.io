Enterprise GenAI Trends & AI Agents

0. 课程主线

这节课不是深入讲某一种 Agent 算法，而是从企业应用视角回答三个问题：

  1. 为什么 Generative AI 在近几年快速爆发？
  2. 企业采用大模型时，技术与产品形态正在发生什么变化？
  3. 要把基础模型变成可用的 AI Agent，需要哪些关键组件？

课程后半部分将 Agent 应用的核心能力归纳为四块：

  • Fine-tuning / 模型适配：让通用模型更适合具体任务；
  • Distillation / 蒸馏：把大模型能力压缩到更小、更快的模型；
  • Grounding / 外部知识对齐：结合 Search、RAG 和可信数据，提高事实性；
  • Function Calling / Tool Use：调用外部函数与系统，让模型能够执行真实动作。

可以把整节课概括为：

1. Generative AI 为什么能够快速发展？

1.1 从传统机器学习到 Foundation Model

早期 Neural Network 往往并不比 SVM、Decision Tree、Regression 等方法更有优势。后来出现转折，主要来自三个因素的共同作用：

  • Architecture：Transformer 等架构更适合大规模并行计算；
  • Data：互联网文本、图像、音频和视频提供了海量训练语料；
  • Compute：GPU、TPU 与分布式训练系统快速发展。

传统监督学习需要人工构造大量带标签样本，而语言模型可以从无标注文本中自动构造训练目标。给定序列

$$ x_1,x_2,\ldots,x_T, $$

Autoregressive Language Modeling 的训练目标为：

$$ \mathcal{L}_{\text{NTP}}(\theta) =-\sum_{t=1}^{T}\log p_{\theta}(x_t\mid x_{<t}). $$

也就是根据前文预测下一个 token。这样，几乎任意文本都可以变成训练样本，极大缓解了人工标注数据不足的问题。

1.3 SFT 与 RLHF

Foundation Model 经过预训练后，还需要进一步对齐为“有用、可控、符合人类偏好”的模型。

Supervised Fine-Tuning（SFT）

给定 Prompt $x$ 与专家答案 $y$，SFT 直接最大化专家答案的条件概率：

$$ \mathcal{L}_{\text{SFT}}(\theta) =-\sum_t \log p_{\theta}(y_t\mid x,y_{<t}). $$

SFT 效果通常很好，但高质量专家答案昂贵且难以大规模获得。

Reinforcement Learning from Human Feedback（RLHF）

RLHF 的核心思路是：

  1. 模型针对同一 Prompt 生成多个候选回答；
  2. 人类不必完整重写答案，只需比较、排序或点赞/点踩；
  3. 用偏好数据训练 Reward Model；
  4. 再利用 Reward Model 优化语言模型。

相较于让专家逐条撰写完整答案，比较两个候选答案通常更快，因此更容易扩展。

2. Multimodal 与 Long Context

2.1 原生多模态模型

课程以 Gemini 为例说明一个明显趋势：模型不再分别处理文本、图像、音频和视频，而是从训练阶段就将多种模态放入统一模型中。

一个原生多模态模型应当能够：

  • 接收交错的 text、image、audio、video；
  • 在不同模态之间联合推理；
  • 最终生成文本或其他模态输出。

这比“分别训练单模态模型，再在应用层拼接结果”更有利于跨模态理解与推理。

2.2 Long Context 的意义

长上下文可以把大量文档、代码、音频或视频直接放入 Context Window，使模型在推理时访问原始信息。

课程强调了两点：

  • 上下文中的信息没有经过大量训练步骤与其他数据混合，通常保留得更加直接；
  • Long Context 能支持更强的 In-context Learning 与长文档理解。

但上下文变长并不自动等于模型一定能够正确找到信息，因此需要专门评测。

2.3 Needle in a Haystack

Needle in a Haystack 测试会把一条目标信息放在极长的文本、音频或视频中，再考察模型能否准确检索该信息。

该测试主要检查：

  • 不同上下文长度下的信息召回能力；
  • Needle 放在不同位置时是否都能被找到；
  • 模型是否随着序列变长而出现明显遗忘。

课程更关注这种 测试思路，而不是某一次榜单或某个具体模型的分数。

可插入课件图（第 18 页）：建议放在 Long Context 讨论之后，直观展示文本、音频、视频上的 Needle in a Haystack 评测。

2.4 Long Context 能否取代 Search？

课程提出了一个开放问题：

当 Context Window 足够大时，是否可以把所有数据直接放进上下文，从而取代 Keyword Search、Embedding Retrieval 或 Deep Retrieval？

长上下文与搜索并不是简单替代关系：

  • Long Context 适合对已选定材料进行统一理解与推理；
  • Search 负责从巨大且不断变化的信息空间中找到最相关内容；
  • 在真实系统中，两者往往组合使用。

3. Enterprise GenAI 的六个趋势

3.1 Trend 1：AI 的发展速度显著加快

企业对 AI 的采用从少量试点变成了大规模主动需求。背后有两个核心原因：

所需专有数据减少

过去企业往往需要：

整个过程可能持续数月甚至数年。Foundation Model 已经具备大量通用知识，企业通常只需少量任务数据、Prompt 或外部知识即可开始构建应用。

开发门槛降低

过去构建 AI 系统通常需要专业 Data Scientist 或 ML Researcher。现在普通 Developer 也可以通过 API、Prompt、RAG 和 Function Calling 构建 AI 应用。

这使 AI 开发者群体从少量机器学习专家扩大到更广泛的软件开发者。

3.2 Trend 2：技术形态发生变化

从任务专用模型到通用模型

过去常为每个任务或行业训练独立模型；现在基础模型能够覆盖大量任务，并通过 Prompt、RAG 或轻量适配满足不同需求。

课程中的一个观察是：领域模型开发周期较长，而新的通用基础模型迭代很快。一个专门训练的领域模型完成时，新一代基础模型可能已经超过它。

因此研究重点逐渐从“为每个领域训练一个新基础模型”，转向：

如何低成本、精确地将强大的通用模型适配到具体 use case？

从 Dense Model 到 Efficient Sparse Model

Dense Model 在一次推理中会激活全部或大部分参数；Sparse Model 或 Mixture-of-Experts 只激活一部分路径。

潜在收益包括：

  • 更低的推理成本；
  • 更低的延迟；
  • 在总参数规模很大的情况下控制实际计算量。

从 Single Modality 到 Multimodality

文本、图像、语音和视频能力正在被整合到统一模型中。未来模型不仅能理解多模态输入，也会生成多模态输出。

3.3 Trend 3：平台选择比单个模型选择更重要

模型榜单变化很快，企业很难永久绑定某一个模型。真正重要的是平台是否具备以下能力：

  • 能访问多种 proprietary、open-source 和 partner models；
  • 能用企业自己的数据进行 customization；
  • 能部署、监控和管理生产模型；
  • 在模型、硬件、工具和云服务层面保持选择与灵活性；
  • 避免过度 vendor lock-in。

可插入课件图（第 29 页）：适合放在“平台选择”小节，用于总结企业 GenAI 平台的四个成功因素。

3.4 Trend 4：API 成本与延迟持续下降

推动成本下降的因素包括：

  • Sparse / MoE 等高效模型结构；
  • 更好的 GPU、TPU 和推理芯片；
  • Quantization、Caching、Batching 与 Serving Optimization；
  • 竞争加剧与规模效应。

课程指出，Open-source Model 并不一定天然更便宜，因为自行部署还包含：

  • 计算硬件成本；
  • 运维与扩缩容成本；
  • 可靠性与监控成本；
  • 工程团队成本。

因此比较成本时不能只看模型权重是否免费，而应看 Total Cost of Ownership。

延迟下降同样重要，因为它决定了模型能否进入客服、实时推荐、交互式应用等 latency-sensitive 场景。

3.5 Trend 5：LLM 与 Search 必须结合

LLM 存在三个事实性问题：

  1. Frozen in the past：训练数据有截止时间，不知道最新信息；
  2. Hallucination：可能生成流畅但错误的内容；
  3. Lack of citation：能够综合推理，但不一定能指出结论来自哪个精确来源。

Search 的优势是：

  • 获取最新内容；
  • 检索企业私有数据；
  • 提供可追溯来源；
  • 支撑事实核验与 citation。

因此企业应用常见形态不是“只使用 LLM”，而是：

在医疗、金融、客服、账单查询等场景中，事实错误的成本很高，Grounding 尤其重要。

3.6 Trend 6：Enterprise Search / Assistant

企业不仅用 AI 服务外部客户，也希望提升内部员工生产力，例如：

  • 跨内部文档搜索；
  • 总结会议、邮件和工单；
  • 查询财务、库存和客户信息；
  • 辅助代码、数据分析与知识管理；
  • 在权限控制下执行内部工作流。

Enterprise Assistant 的关键不只是聊天界面，而是与企业知识、权限、业务 API 和工作流深度连接。

4. 企业级 GenAI 平台架构

课程用 Vertex AI 作为示例，但该分层思路具有普遍性。

可插入课件图（第 46 页）：适合放在本节开头，展示 Infrastructure、Model Garden、Model Builder 与 Agent Builder 的分层。

4.1 Infrastructure

底层提供 GPU、TPU、存储、数据平台与分布式系统。企业需要在成本、性能、资源供应和地域合规之间做选择。

4.2 Model Garden

集中提供：

  • 平台自研模型；
  • 第三方商业模型；
  • Open-source Model；
  • 不同规模、延迟和模态的模型。

Model Garden 的价值在于让企业能够针对不同任务比较模型，而不是把全部任务交给同一个最大模型。

4.3 Model Builder

负责模型层面的开发与运维：

  • Prompt Management；
  • Tuning 与 Distillation；
  • Evaluation；
  • Training Pipeline；
  • Serving、Monitoring 与版本管理。

4.4 Agent Builder

负责将模型连接成应用或 Agent：

  • Orchestration；
  • Search 与 Retrieval Engine；
  • Ranking；
  • Grounding；
  • Extensions、Connectors 与 Function Calling；
  • 文档处理与企业数据接入。

这里体现了一个重要结论：

基础模型本身很强，但只有把它放入具体业务上下文、连接数据与工具，并能可靠部署时，才真正产生企业价值。

5. 模型适配：从 Prompt 到 Fine-tuning

可插入课件图（第 52 页）：适合放在本节开头，用于比较 Prompt Design、Prompt Tuning、Full Fine-tuning 与 Distillation。图中的训练规模和成本数字是示意值，不应视为普遍结论。

5.1 Prompt Design / In-context Learning

Prompt Design 不更新模型参数，而是在输入中提供任务说明与少量示例。

  • Zero-shot：不给示例；
  • One-shot：给一个示例；
  • Few-shot：给少量示例。

优点：

  • 不需要训练；
  • 开发速度快；
  • 只需要少量示例；
  • 便于实时迭代。

缺点：

  • Prompt 会占用 Context Window；
  • 推理成本可能随示例增多而增加；
  • 对复杂、稳定、严格格式任务的控制有限。

课程观察：在企业实践中，Prompt Design 因为简单直接，被广泛采用。

5.2 Prompt Tuning

Prompt Tuning 冻结基础模型，只训练一组可学习的 Soft Prompt Embeddings：

其参数量小于 Full Fine-tuning，但课程指出，在当时的企业实践中，使用率没有预期那么高，很多团队更倾向于直接做 Few-shot Prompting 或 PEFT。

5.3 Full Fine-tuning

Full Fine-tuning 会更新模型全部或大部分参数。

优点：

  • 对特定任务的适配能力强；
  • 在数据充足时可获得较高上限。

缺点：

  • 训练数据需求高；
  • 计算和存储成本高；
  • 需要理解模型架构和训练系统；
  • 每个任务可能需要维护独立模型副本；
  • 企业实践中通常不是第一选择。

5.4 Parameter-Efficient Fine-Tuning（PEFT）

PEFT 只更新少量参数或新增少量可训练模块，从而降低训练与存储成本。

常见方法包括：

  • Adapter；
  • Prefix / Prompt Tuning；
  • LoRA；
  • QLoRA。

6. LoRA：低秩适配

LoRA 冻结预训练权重 $W_0$，只学习低秩更新矩阵：

$$ W = W_0 + \Delta W, $$

其中

$$ \Delta W = BA. $$

设

$$ W_0\in\mathbb{R}^{d\times k},\quad B\in\mathbb{R}^{d\times r},\quad A\in\mathbb{R}^{r\times k}, $$

并令

$$ r\ll \min(d,k), $$

则可训练参数量从 $dk$ 降为：

$$ r(d+k). $$

原始前向传播为：

$$ h=W_0x, $$

加入 LoRA 后：

$$ h=W_0x+BAx. $$

可插入课件图（第 59 页）：建议紧跟 LoRA 公式，用于对应矩阵维度和前向传播变化。

LoRA 的企业价值

  • 基础模型不变，只保存小型 Adapter；
  • 训练计算量与显存需求较低；
  • 可为不同客户或任务保存不同 Adapter；
  • 存储、部署和版本管理更容易；
  • Adapter 与基础权重分离，有利于数据治理与权限管理；
  • 合并权重后通常不会引入额外推理延迟。

课程认为 LoRA 和 QLoRA 是当时企业中非常常见的适配方案。

7. Distillation：把大模型能力迁移到小模型

7.1 为什么需要蒸馏？

并不是每个任务都需要最强、最大的模型。大模型通常意味着：

  • 更高 API 或 Serving Cost；
  • 更大显存与计算需求；
  • 更高延迟；
  • 更难部署到边缘或高并发场景。

Distillation 的目标是让较小的 Student Model 在特定任务上模仿 Teacher Model。

7.2 核心术语

  • Teacher Model：能力较强的大模型；
  • Student Model：需要训练的小模型；
  • Hard Label：人工标注的 one-hot ground truth；
  • Soft Label：Teacher 在温度 $T>1$ 下输出的概率分布；
  • Soft Prediction：Student 在温度 $T>1$ 下的输出；
  • Hard Prediction：Student 使用正常 Softmax（$T=1$）时的输出。

7.3 Temperature Softmax

普通 Softmax 为：

$$ p_i=\frac{\exp(z_i)}{\sum_j\exp(z_j)}. $$

引入温度 $T$：

$$ p_i^{(T)} =\frac{\exp(z_i/T)}{\sum_j\exp(z_j/T)}. $$

当 $T>1$ 时，输出分布更加平滑。Teacher 不只告诉 Student “哪个类别最可能”，还传递类别之间的相对相似性。

例如 Teacher 认为一张图片属于数字 2，但同时认为它与数字 3 比与数字 7 更相似。Soft Label 能保留这种结构信息。

常见蒸馏目标可以写成：

$$ \mathcal{L} =\alpha T^2\,D_{\mathrm{KL}} \left(p_{\text{teacher}}^{(T)}\Vert p_{\text{student}}^{(T)}\right) +(1-\alpha)\,\mathrm{CE}(y,p_{\text{student}}^{(1)}). $$

第一项让 Student 模仿 Teacher 的 Soft Distribution；第二项让模型同时学习真实标签。

可插入课件图（第 68 页）：适合放在蒸馏损失之后，展示 Teacher Soft Labels、Student Soft Predictions 与 Hard Labels 的联合训练结构。

7.4 蒸馏的实际价值

  • Teacher 可以自动为大量无标签数据生成监督信号；
  • Student 在特定任务上可以接近大模型表现；
  • 降低部署成本与延迟；
  • 适合高并发、端侧或固定领域任务。

但 Student 的能力通常更集中，未必保留 Teacher 的全部通用能力。

8. Grounding、Search 与 RAG

8.1 Grounding 要解决什么？

课程把减少幻觉归纳为三个方面：

  1. Right Context：向模型提供正确、相关的信息；
  2. Better Models：训练更重视 factuality 的模型；
  3. User Experience：展示来源、支持证据和冲突信息。

Grounding 的知识来源可以是：

  • 企业 Private Documents；
  • Web 上的 Fresh Content；
  • 第三方 Authoritative Sources；
  • 数据库、知识图谱与业务系统。

8.3 Pre-hoc 与 Post-hoc Grounding

Pre-hoc Grounding

在生成前，把检索结果加入 Prompt，让模型基于证据回答：

可以通过 Fine-tuning 或 LoRA 进一步训练模型，让它更遵循证据并生成 citation。

Post-hoc Verification

生成答案后，逐句检查答案是否被资料支持。可以使用 Natural Language Inference（NLI）判断：

  • Supported；
  • Contradicted；
  • Not enough evidence。

然后为每个陈述计算支持来源、冲突来源与 citation。

8.4 Attribution 是产品体验的一部分

企业用户不仅关心答案是否流畅，还会问：

How do you know this is true?

因此系统应当把回答拆成可验证的 claims，并显示：

  • 哪些来源支持该 claim；
  • 哪些来源与之冲突；
  • 引用覆盖率；
  • 来源权威性。

可插入课件图（第 76 页）：建议放在 Attribution 讨论后，展示一句回答中不同 claim 的支持、争议与矛盾来源。

8.5 Better Models 与事实性权衡

除了检索外，还可以通过 RLHF 或 AI Feedback 对模型进行事实性训练，对 ungrounded response 给予更大惩罚。

课程指出，事实性与开放式创造、推理能力之间有时存在 trade-off。企业在医疗、金融和客服场景中通常更偏向稳定、可验证的回答。

8.6 Dynamic Retrieval

并不是每次请求都必须调用 Search。由于搜索服务的调用成本不一定像 LLM 推理成本一样快速下降，系统需要学习：

  • 当前问题是否需要外部检索？
  • 应该搜索 Web、企业文档还是结构化数据库？
  • 检索多少内容才足够？
  • 何时停止继续搜索？

这就是 Dynamic Retrieval，也是课程明确提出的研究与创业方向之一。

9. Function Calling：让模型从“回答”走向“行动”

9.1 Generative Model 的三个限制

单纯依赖文本生成通常会遇到：

  • Inconsistent Output：难以稳定生成严格 JSON 或 Schema；
  • Frozen in Time：无法自动获得最新信息；
  • Disconnected from the World：无法直接访问 API、数据库或执行真实动作。

仅靠极端 Prompt Engineering、反复要求“必须输出合法 JSON”，通常不够稳定。

9.2 Function Calling 的基本流程

  1. Developer 定义可调用函数及参数 Schema；
  2. 将函数包装为 Tool 提供给模型；
  3. 用户发送 Prompt；
  4. 模型判断是否需要调用工具；
  5. 模型选择函数并生成参数；
  6. 应用执行真实 API 或数据库调用；
  7. 将函数结果返回给模型；
  8. 模型基于结果生成最终回答。

可插入课件图（第 87 页）：建议放在流程说明之后，展示 Prompt、Tool Selection、外部系统调用与最终回复的闭环。

9.3 Agent 的关键难点：When 与 Which

Function Calling 不只是让模型“会填参数”，更难的是两步决策：

  1. When to call：模型必须知道当前问题超出自身知识或能力，需要调用工具；
  2. Which tool to call：从大量工具中选出正确函数。

课程用“Who Wants to Be a Millionaire”中的 Call a Friend 类比：

  • 先意识到自己不知道；
  • 再选择最合适的朋友。

随着 Tool Library 从几个函数扩展到数百个函数，工具检索、路由与选择会越来越困难。

9.4 Function Calling 能做什么？

  • 查询实时金融与汇率；
  • 查询航班、酒店与库存；
  • 访问数据库和 BigQuery；
  • 读写企业文档与表格；
  • 查询客户、订单、工单和账单；
  • 返回严格 Structured Output；
  • 执行多步骤 Autonomous Workflow。

可插入课件图（第 90 页）：适合放在用例总结后，展示 Function Calling 从 Structured Output 到 Autonomous Workflow 的复杂度谱系。

10. 构建成功 Agent 还需要哪些工程能力？

Fine-tuning、Grounding 和 Function Calling 只是核心模块。生产系统还需要：

10.1 Prompt Management

  • Prompt 存储与版本控制；
  • Prompt 的实验记录；
  • 不同模型之间的 Prompt 迁移；
  • 线上 Prompt 回滚；
  • 权限和审计。

10.2 Evaluation

模型数量和版本越来越多，企业需要持续比较：

  • Quality；
  • Factuality；
  • Safety；
  • Latency；
  • Cost；
  • Tool-use Accuracy；
  • Citation Accuracy；
  • Task Success Rate。

10.3 LLM-as-a-Judge

课程提到 Auto Side-by-Side Evaluation：

  1. 让多个模型回答同一组问题；
  2. 使用能力更强的大模型比较回答；
  3. 让 Judge Model 选择更优答案并给出理由；
  4. 与人类评价进行相关性检查。

优势：

  • 比纯人工评测更容易扩展；
  • 可以快速比较大量模型和版本；
  • 缓解人工标注人员不足。

风险：

  • Judge 可能存在位置偏差、长度偏差和自我偏好；
  • 需要使用人工样本校准；
  • 不应把 Judge Score 等同于真实用户价值。

10.4 Production Management

企业系统必须关注：

  • Availability 与 Reliability；
  • Monitoring 与 Logging；
  • Data Privacy 与 Compliance；
  • Access Control；
  • Cost Quota；
  • Failover 与 Model Routing；
  • Human-in-the-loop。

企业真正需要的不是一个 Demo，而是可管理、可评估、可审计的生产系统。

11. 课程提出的研究与创业方向

11.1 Long Context 与 Search 的关系

研究问题：

  • 多长的上下文才真正有用？
  • Long Context 在复杂语义检索上能否替代独立 Retrieval System？
  • 如何比较“直接塞入上下文”和“先检索再生成”的成本与准确率？

11.2 Use-case-specific Customization

通用模型越来越强，但具体业务仍需要精确适配：

  • 如何用极少数据适配特定任务？
  • Prompt、LoRA、Distillation 和 RAG 应如何组合？
  • 如何避免适配后损失通用能力？

11.3 Dynamic Retrieval

让模型自主决定：

  • 是否检索；
  • 检索哪个数据源；
  • 使用何种 Retrieval Method；
  • 是否需要二次检索或核验。

11.4 Large Tool Library 中的 Tool Selection

当工具数量变多时，需要研究：

  • Tool Retrieval；
  • Hierarchical Routing；
  • Tool Description Learning；
  • 调用失败后的 Recovery；
  • 多工具组合与 Planning。

11.5 Scalable Evaluation 与 LLM-as-a-Judge

值得研究的问题包括：

  • 如何提高 Judge 与 Human Preference 的一致性？
  • 如何减少 Judge Bias？
  • 如何评价长工作流 Agent，而不是只评价单轮回答？
  • 如何同时衡量准确性、成本、延迟和安全性？

12. 核心概念对比

方法 是否更新模型参数 主要解决问题 优点 主要代价或局限
Prompt Design 否 快速指定任务与格式 简单、无需训练 稳定性有限，占用上下文
Prompt Tuning 是，只训练 Soft Prompt 轻量任务适配 参数少 实践使用并非总是广泛
Full Fine-tuning 是，更新全部或大部分参数 深度任务适配 上限较高 数据、算力、存储成本高
LoRA / PEFT 是，只更新少量参数 低成本模型适配 易训练、易保存、易部署 仍需高质量任务数据
Distillation 训练 Student 降低成本和延迟 小模型可专精任务 通用能力可能下降
RAG / Grounding 通常不必更新模型 最新知识、私有知识与引用 可追溯、知识可更新 Retrieval 质量决定上限
Function Calling 不一定更新模型 访问外部系统并执行动作 从聊天走向 Agent 工具选择、参数正确性与安全性
LLM-as-a-Judge 否或单独训练 Judge 扩展评测规模 速度快、成本低 存在系统性偏差

13. 一页速记

  1. Foundation Model 的爆发来自 Transformer + Massive Data + Compute + Self-supervised Objective。
  2. 企业采用 AI 加速，是因为启动所需专有数据更少、普通 Developer 也能构建应用。
  3. 模型发展方向包括：Generalist、Sparse、Multimodal、Long Context。
  4. 企业更关心完整平台，而不只是某一个榜单第一的模型。
  5. Open-source 不等于零成本，应比较完整部署和运维成本。
  6. LLM 与 Search 结合，是解决新鲜度、幻觉和 citation 的关键路径。
  7. 模型适配通常从 Prompt 开始，再考虑 LoRA、Full Fine-tuning 或 Distillation。
  8. LoRA 通过 $\Delta W=BA$ 只训练低秩矩阵，适合多任务和多租户场景。
  9. Distillation 用 Teacher 的 Soft Labels 训练小型 Student，降低延迟和成本。
  10. RAG 的核心是 Retrieve → Augment → Generate → Verify。
  11. Function Calling 的真正难点是知道 何时调用 以及 调用哪个工具。
  12. 生产级 Agent 必须具备 Evaluation、Monitoring、Security、Prompt Management 和 Human Oversight。
