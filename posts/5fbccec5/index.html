加快语言模型的生成速度：Inference 加速

1. 核心要点

语言模型的推理加速并不是“凭空变快”。分析一种加速技术时，应始终追问三个问题：

  1. 是否改变原有 Attention 的计算结果或范围？
  2. 是否需要为该方法重新训练或定制模型？
  3. 为了加速，还付出了什么代价？ 例如额外显存、额外计算、模型能力下降或算法复杂度上升。

2. 语言模型推论回顾

2.1 Autoregressive Generation

语言模型本质上在做“文字接龙”：给定前面的 token，预测下一个 token；新生成的 token 又会被放回输入，继续预测下一个 token。

若输入 token 的表示为 $x_i$，Self-Attention 首先生成 Query、Key 和 Value：

$$ q_i=W_Qx_i, \qquad k_i=W_Kx_i, \qquad v_i=W_Vx_i. $$

对于位置 $t$，Causal Self-Attention 只能关注当前位置及其以前的 token。标准 Attention score 为：

$$ a_{t,i}=\frac{q_t^{\top}k_i}{\sqrt{d_k}}, \qquad i\le t. $$

课上为了突出推论算法，常把缩放项省略，直接写成 Query 与 Key 的 dot product。

经过 Softmax 后得到归一化权重：

$$ \hat a_{t,i} = \frac{e^{a_{t,i}}} {\sum_{j=1}^{t}e^{a_{t,j}}}. $$

最终输出是 Value 的加权和：

$$ o_t = \sum_{i=1}^{t}\hat a_{t,i}v_i. $$

2.2 Prefill 与 Decode

一次完整生成可以分成两个阶段。

阶段 输入形式 主要特点 典型瓶颈
Prefill 一次输入完整 Prompt 多个 token 的计算可以高度并行 长序列 Attention、算力与显存带宽
Decode 每次新增一个 token 必须逐 token 自回归生成，时间维度难以并行 读取历史 KV Cache、内存带宽与延迟

Prefill 会为 Prompt 中所有 token 计算 Q、K、V；Decode 时，每轮只新增一个 token，但新 Query 仍需要访问前面所有 token 的 Key 和 Value。

Part I：Flash Attention

3. Flash Attention 的目标

Flash Attention 的关键特点是：

  • 结果与标准 Attention 相同，不是近似算法。
  • 通常不需要重新训练模型，可以直接替换 Attention 的实现。
  • 核心优化目标不是减少数学定义中的 Attention，而是减少 GPU 上的数据搬运次数。

它是一种 IO-aware algorithm：算法设计直接考虑 GPU 存储层级，而不仅仅统计 FLOPs。

4. GPU 存储层级：HBM 与 SRAM

可以用“仓库”和“工作台”来类比 GPU 的两类存储。

存储 课堂类比 特点
HBM 仓库 容量较大，但访问较慢
On-chip SRAM 工作台 容量很小，但访问和计算很快

GPU 中有大量并行 Execution Units。数据一旦被搬到 SRAM，矩阵计算通常很快；真正拖慢程序的往往是：

  • 从 HBM 读取数据到 SRAM；
  • 将中间结果从 SRAM 写回 HBM；
  • 后续步骤再次从 HBM 读取同一批中间结果。

因此，Attention 可能不是单纯的 compute-bound，而是明显受到 memory bandwidth / IO 限制。

Flash Attention 的核心问题：能否改变计算顺序，在结果不变的前提下，让 Q、K、V 和中间量少往返 HBM？

5. 标准 Attention 为什么需要反复读写

为简化说明，先考虑一个 Query $q$ 对长度为 $L$ 的 Key 和 Value 序列进行 Attention。

Attention score 为：

$$ a_i=q^{\top}k_i, \qquad i=1,\ldots,L. $$

为了避免指数运算溢出，实际 Softmax 通常减去全局最大值：

$$ a_{\max}=\max_{1\le i\le L}a_i. \qquad \hat a_i = \frac{e^{a_i-a_{\max}}} {\sum_{j=1}^{L}e^{a_j-a_{\max}}}. $$

输出为：

$$ o = \sum_{i=1}^{L}\hat a_i v_i. $$

当序列很长时，SRAM 无法放下与 $L$ 成比例的所有 Key、Value 或 Attention score，只能将序列切成多个 chunk。设每个 chunk 包含 $N$ 个位置，chunk 数量近似为：

$$ B=\frac{L}{N}. $$

朴素实现往往需要多次扫描数据：

  1. 分块计算所有 $a_i$，并将 score 写回 HBM。
  2. 再次读取 score，寻找全局最大值。
  3. 再次读取 score，计算 Softmax 分母。
  4. 再次读取 score，生成归一化权重并写回 HBM。
  5. 读取 Attention 权重和 Value，计算最终加权和。

问题不只是“做了多少乘法”，而是 $a_i$ 与 $\hat a_i$ 等大规模中间量被反复写入、读取。

6. Online Softmax：一次扫描更新最大值和分母

6.1 分块定义

把第 $k$ 个 chunk 的位置集合记为 $I_k$。处理完前 $k$ 个 chunk 后，维护两个常数：

  • $d_k$：目前见过的最大 score；
  • $s_k$：以当前最大值 $d_k$ 为基准的指数和。

初始化：

$$ d_0=-\infty, \qquad s_0=0. $$

读入第 $k$ 个 chunk 后，最大值更新为：

$$ d_k = \max\left( d_{k-1}, \max_{i\in I_k}a_i \right). $$

分母更新为：

$$ s_k = s_{k-1}e^{d_{k-1}-d_k} + \sum_{i\in I_k}e^{a_i-d_k}. $$

旧分母原本以 $d_{k-1}$ 为基准：

$$ s_{k-1} = \sum_{i\in I_1\cup\cdots\cup I_{k-1}} e^{a_i-d_{k-1}}. $$

当新的最大值变为 $d_k$ 时，只需乘上一个修正项：

$$ s_{k-1}e^{d_{k-1}-d_k} = \sum_{i\in I_1\cup\cdots\cup I_{k-1}} e^{a_i-d_k}. $$

因为：

$$ e^{a_i-d_{k-1}}e^{d_{k-1}-d_k} = e^{a_i-d_k}. $$

这样便不需要重新读取之前所有 chunk。处理最后一个 chunk 后：

$$ d_B=a_{\max}, \qquad s_B = \sum_{i=1}^{L}e^{a_i-a_{\max}}. $$

这就是完整 Softmax 的正确分母。

7. Flash Attention：不显式保存 Attention Matrix，直接更新输出

Online Softmax 仍可以先得到分母，再第二次读取 score 计算归一化权重；Flash Attention 更进一步：

不必先完整生成并保存 $\hat a_i$，可以在每个 chunk 内直接结合 Value，逐步得到最终输出。

令 $o_k$ 表示处理完前 $k$ 个 chunk 后，按照当前 $d_k$ 与 $s_k$ 归一化得到的局部输出。初始化：

$$ o_0=0. $$

处理第 $k$ 个 chunk 时，输出更新为：

$$ o_k = o_{k-1} \frac{s_{k-1}}{s_k} e^{d_{k-1}-d_k} + \sum_{i\in I_k} \frac{e^{a_i-d_k}}{s_k}v_i. $$

第一项是在修正过去的输出：

  • $d_{k-1}$ 变成了 $d_k$，需要指数修正；
  • 归一化分母从 $s_{k-1}$ 变成了 $s_k$，需要比例修正。

第二项加入当前 chunk 的新贡献。

处理完最后一个 chunk 后：

$$ o_B = \sum_{i=1}^{L} \frac{e^{a_i-a_{\max}}} {\sum_{j=1}^{L}e^{a_j-a_{\max}}} v_i = \sum_{i=1}^{L}\hat a_i v_i. $$

所以，尽管计算过程中没有把完整 Attention 权重矩阵写入 HBM，最终输出仍与标准 Attention 完全一致。

7.1 从单个 Query 推广到完整 Attention

上面为了便于推导只画了一个 Query，真实实现会同时把 Q、K、V 划分为二维 tiles：

  • 在 SRAM 中加载一小块 Query；
  • 依次加载多块 Key 和 Value；
  • 对每一行 Query 维护对应的 running max、running sum 和输出；
  • 完成后只把最终输出写回 HBM。

其数学思想仍是相同的 Online Softmax 与分块输出修正。

8. Flash Attention 为什么更快

Flash Attention 的主要收益来自：

  1. Tiling：只把当前需要的小块 Q、K、V 搬进 SRAM。
  2. Kernel fusion：dot product、mask、Softmax 和 Value 加权尽量在同一计算过程中完成。
  3. 不物化完整 Attention Matrix：避免把长度平方级的 score 或权重矩阵频繁写入 HBM。
  4. 减少 HBM 读写：用少量额外计算换取大幅减少的数据搬运。

8.1 代价和限制

  • 需要额外的指数修正和归一化修正，计算量可能略有增加。
  • 实现复杂，理解和编写 kernel 更困难。
  • 某些框架在 Flash Attention 模式下不会保留完整 Attention 权重，因此无法直接读取整个 Attention matrix 做可视化。
  • 序列很短时，Attention 并不是总耗时的主要部分，速度提升可能不明显。
  • Flash Attention 减少的是运算过程中的 IO，不能消除 KV Cache 对 HBM 容量的占用。

8.2 课堂实验结论

课堂示例中：

  • 对随机 Q、K、V 做 Attention，序列长度达到 4096 时，Flash Attention 相较朴素实现可达到约 8 到 9 倍加速。
  • 在真实语言模型示例中，Prompt 较短时提升有限；Prompt 约为 7 万 token 时，示例耗时约由 2.0 秒降低至 1.3 秒。
  • 当 Prompt 增加到约 73 万 token 时出现 CUDA Out of Memory，说明即使运算方式更高效，HBM 容量仍然有限。

这些数值与具体硬件、模型、数据类型和实现有关，应理解为课堂示范，而不是固定性能保证。

Part II：KV Cache

9. KV Cache 的基本机制

9.1 Prefill 阶段

Prompt 一次性进入模型后，每一层都会为各 token 计算 Q、K、V。不同位置的输出可以并行计算。

Prefill 完成后：

  • 保存每一层、每一个历史 token 的 Key 和 Value；
  • 不需要保存过去的 Query。

9.2 Decode 阶段

假设当前生成第 $t$ 个 token，只需：

  1. 为新 token 计算 $q_t$、$k_t$、$v_t$；
  2. 用 $q_t$ 与缓存中的所有历史 Key 计算 Attention；
  3. 用得到的权重对所有历史 Value 做加权和；
  4. 将新的 $k_t$、$v_t$ 追加到 Cache。

9.3 为什么保存 K、V，不保存 Q

未来生成新 token 时，需要的是“新的 Query 对过去所有 Key 进行检索，再读取过去的 Value”。过去 token 的 Query 已经完成了它当时的计算，以后不会再次使用。

因此：

  • Key 和 Value 会被未来所有 token 重复访问；
  • 旧 Query 不会再被使用。

9.4 本质上的时间—空间交换

KV Cache 避免重复计算历史 token 的 K、V，显著加快 Decode；代价是缓存量随以下因素线性增长：

  • 序列长度；
  • Transformer 层数；
  • KV Head 数量；
  • 每个 Head 的维度；
  • Batch size；
  • 数值精度所占字节数。

10. KV Cache 的显存计算

只考虑单个 token、单个 batch 时，KV Cache 大小可写成：

$$ M_{\text{token}} = 2 \times N_{\text{layer}} \times N_{\text{KV-head}} \times d_{\text{head}} \times b, $$

其中：

  • 前面的 $2$ 表示同时保存 Key 和 Value；
  • $b$ 表示每个数占用的字节数，例如 FP16 为 2 Byte。


以 Gemma 2 27B 的参数做简化估算：

$$ 46 \times 32 \times 128 \times 2 \times 2 = 753664\ \text{Bytes}. $$

约为：

$$ 736\ \text{KiB} \approx 0.72\ \text{MiB/token}. $$

若把 80 GiB 全部用于该 KV Cache，理想化上限约为：

$$ \frac{80\ \text{GiB}}{0.72\ \text{MiB/token}} \approx 114000\ \text{tokens}. $$

如果有 batch，整体 KV Cache 还要再乘 batch size：

$$ M_{\text{total}} = B_{\text{batch}} \times L_{\text{seq}} \times M_{\text{token}}. $$

11. MHA、MQA 与 GQA

KV Cache 只保存 K 和 V，因此减少 KV Head 数量可以直接降低 Cache 大小和 Decode 时的内存读取量。

方法 Query Head KV Head 核心思想 KV Cache 代价
MHA 多个 与 Query 数量相同 每个 Query Head 有独立 K、V 最大 表达能力强，但显存占用高
MQA 多个 1 组 所有 Query Head 共享同一组 K、V 最小 可能明显损伤模型能力
GQA 多个 介于 1 与 Query Head 数之间 每组 K、V 服务多个 Query Head 居中 在效果和效率之间折中

若 Query Head 数为 $H_q$，KV Head 数为 $H_{kv}$，则 KV Cache 大小与 $H_{kv}$ 近似成正比：

$$ M_{\text{KV}} \propto H_{kv}. $$

每组 KV 对应的 Query 数约为：

$$ G = \frac{H_q}{H_{kv}}. $$

重要区别：

  • 不是让多个 K、V 共享一个 Query；
  • 而是让多个 Query 共享一组 K、V；
  • 原因是只有 K、V 需要长期缓存，Query 不需要缓存。

MQA 和 GQA 改变了原始 MHA 架构，通常需要在训练模型时就采用相应设计，不能简单把任意现成模型无损改成 MQA 或 GQA。

12. Multi-Head Latent Attention（MLA）

12.1 核心思想

MHA 需要缓存多组高维 K、V。MLA 在输入与多组 K、V 之间加入低维 bottleneck，将每个 token 压缩成 latent vector $c_i$，缓存时主要保存这个低维表示。

简化表示为：

$$ c_i=W_Dx_i. $$

每个 Head 的 Key 和 Value 可由 latent vector 线性映射得到：

$$ k_i=W_Kc_i, \qquad v_i=W_Vc_i. $$

MLA 的关键不只是“压缩”，而是可以利用线性代数改变运算顺序，避免对序列中的每个 latent vector 都进行完整解压缩。

12.2 Attention score 不需要逐 token 解压 Key

原本：

$$ a_i=q^{\top}k_i. $$

代入 $k_i=W_Kc_i$：

$$ a_i =q^{\top}W_Kc_i = \left(W_K^{\top}q\right)^{\top}c_i. $$

定义压缩空间中的 Query：

$$ q'=W_K^{\top}q. $$

于是：

$$ a_i={q'}^{\top}c_i. $$

因此不需要先把每个 $c_i$ 解压成 $k_i$，只需把当前 Query 变换一次，再直接与所有缓存的 latent vectors 做 dot product。

12.3 Weighted Sum 不需要逐 token 解压 Value


标准输出为：

$$ o = \sum_i\hat a_i v_i. $$

代入 $v_i=W_Vc_i$：

$$ o = \sum_i\hat a_iW_Vc_i = W_V\left(\sum_i\hat a_i c_i\right). $$

所以可以：

  1. 先在低维 latent space 中完成加权和；
  2. 最后只对汇总后的一个向量做一次映射。

而不是对长度为 $L$ 的每个 token 分别解压。

12.4 特点

  • 大幅压缩需要保存的 KV 信息。
  • 需要模型在训练时学习压缩和映射，属于模型架构设计。
  • 相对于原始 MHA，它改变了 Attention 的参数化方式；但在该架构内部，上述换序计算本身是代数等价的，不是数值近似。

13. Sliding Window Attention

Sliding Window Attention 将每个 token 的可见范围限制为最近 $W$ 个 token：

$$ \mathcal A_t = \left\{ \max(1,t-W+1),\ldots,t \right\}. $$

每层只需保存固定窗口内的 KV，Cache 不再随序列无限增长。虽然单层只能看到局部，但通过多层堆叠，较早 token 的信息可被中间位置吸收后再传递给更高层，有效感受野随深度扩大——不过长距离信息仍可能逐层衰减或丢失。实际使用时常与 Full Attention 层交替，兼顾效率与全局信息传递。

14. StreamingLLM 与 Attention Sink

单纯只保留最近窗口在长序列下表现会明显下降。StreamingLLM 发现：额外保留序列最开头的少数 token 就能让模型稳定处理远超训练长度的流式输入：

$$ \mathcal A_t = \{1,2,\ldots,S\} \cup \{t-W+1,\ldots,t\}. $$

其原理是 Softmax 权重之和必须为 1，即使当前没有需要关注的上下文，模型也必须将多余 Attention 分配给某些位置。开头的 token 往往充当了 Attention Sink——模型把”无处可放”的注意力集中到它们身上。删除这些 sink token 会破坏已形成的 Attention 行为；保留它们后表现显著恢复。这里的要点是：开头 token 被保留不一定因为语义重要，而可能是因为它们承担了稳定 Attention 分布的结构性作用。该方法可在 Inference 阶段直接应用，也可在训练时显式加入。

15. KV Cache Pruning

Scissorhands、H2O 等工作观察到：每次生成时真正获得明显 Attention 的历史 token 只占少数（heavy hitters），大量 KV 长期存在却几乎不再被访问。因此可在显存达到预算上限时，保留最近 token 和累计 Attention 较高的 token，删除长期未使用的 KV，将显存放在固定预算内。

  • 优点：通常可直接用于已有模型，丢弃大量 KV 后性能仍可能接近完整 Cache。
  • 风险：被删 token 以后可能重新变得重要但无法恢复；困难的长距离推理、信息检索任务可能明显退化。核心难点在于判断哪些 KV 可以安全删除。

16. 跨对话的 KV Cache：Prompt Caching

前面的 KV Cache 主要是在同一个生成过程中复用历史 K、V。服务平台还可以在不同请求之间复用相同前缀的 KV Cache。

16.1 只有相同 Prefix 才能复用

在 Causal Transformer 中，一个 token 的表示不仅取决于它自身，还取决于它前面的全部 token。

例如两条序列：

公共前缀“大家好，我是”对应的 K、V 可以复用；从“大”和“小”开始，前缀已经不同，后续即使再次出现相同的“金”字，其 K、V 也不能直接复用。

原因可概括为：

$$ (k_i,v_i) = f(x_1,x_2,\ldots,x_i), $$

而不是只由当前 token $x_i$ 决定。

16.2 System Prompt 的排列原则

为了提高 Cache Hit：

  • 稳定、不常变化的内容放在前面；
  • 日期、用户输入、任务参数等易变化内容放在后面。


对于 AI Agent，常见的稳定前缀包括：

  • 可使用的工具及调用规则；
  • Agent 的固定行为规范；
  • 身份、长期目标、固定记忆检索方式；
  • 长而稳定的 system prompt。

容易变化的内容包括：

  • 当前时间；
  • 用户本轮输入；
  • 动态环境状态；
  • 临时变量和任务参数。

16.3 Prompt 改写示例

Cache Hit 较少的写法：

因为地点较早出现，前缀很快分叉。

更有利于缓存的写法：

固定模板被放在前面，变化参数被移到后面，公共前缀更长。

16.4 为什么 Prompt Caching 与成本有关

如果前缀的 KV 已经计算完成，平台无需再次对这部分 Prompt 执行完整 Prefill，只需读取已有缓存。课程以 API 对 cached input 提供更低价格为例，说明计算成本与账单成本之间的联系。

需要区分：

  • KV Cache：通常指单次会话或生成过程中的历史 KV；
  • Prompt Caching / Prefix Caching：服务系统在多个请求之间复用相同前缀的 KV。

二者保存的核心对象相同，但复用范围不同。

Part III：Speculative Decoding

17. 基本思想

Autoregressive Decoding 每次只能生成一个 token，后一个 token 必须等待前一个 token 生成完成，因此长文本生成速度较慢。

Speculative Decoding 引入一个速度很快的 draft model，类似“预言家”，先猜测接下来多个 token，再由较大的 target model 一次并行验证。

假设 draft model 预测：

$$ \tilde y_1,\tilde y_2 $$

target model 可以同时计算：

$$ P(y_1\mid x),\quad P(y_2\mid x,\tilde y_1),\quad P(y_3\mid x,\tilde y_1,\tilde y_2) $$

因此，一次大模型前向计算可能推进多个 token。

17.1 验证结果

  • 全部猜对： 接受所有候选，并可额外得到下一个 token。
  • 部分猜对： 接受最长正确前缀，在第一个错误位置使用 target model 的结果，后续候选全部丢弃。
  • 第一个就猜错： 仍能输出 target model 生成的一个 token，但基本没有加速，并浪费部分计算。

所以 draft model 不需要完全正确，只要经常能连续猜对若干 token，就可以减少 target model 的调用次数。

17.2 谁可以作为 Draft Model

draft model 的要求是：

  • 速度足够快；
  • 输出尽量接近 target model；
  • 偶尔犯错没有关系。

常见选择包括：

  • Non-autoregressive model；
  • 压缩或蒸馏后的小模型；
  • 检索系统或搜索引擎；
  • 多个 draft model 同时提出候选。

draft model 不是越小越好：模型太小虽然快，但接受率可能很低；模型太大又会增加候选生成时间。

17.3 特点与代价

优点

  • 通常不需要重新训练或修改 target model；
  • 可以作为外接式推理加速方法；
  • 一次验证可能生成多个 token；
  • 通过正确的接受、拒绝和概率修正，可以保持 target model 原本的生成分布。

代价

  • 需要额外运行 draft model；
  • 错误候选会造成无效计算；
  • 接受率低时加速效果有限；
  • 会增加部署与调度复杂度。

其本质是：

使用更多并行计算资源，换取更低的生成延迟。
