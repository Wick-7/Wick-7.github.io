Introduction to Training LLMs for AI Agents

1. 课程核心问题：一个 LLM 是怎样被训练出来的？

一个现代大语言模型通常不是通过一次训练直接得到的，而是经历多个阶段：

  1. Pretraining / 预训练：从海量文本中学习语言、知识和一般能力；
  2. Reasoning RL / 推理强化学习：在具有可验证答案的任务上学习推理，仅用于 reasoning model；
  3. Classic Post-training / 经典后训练：学习遵循指令、帮助用户并符合人类偏好；
  4. 在模型发布后，还可以通过 Prompting 或 Finetuning 将模型适配到具体场景。

1.1 三个主要训练阶段

阶段 主要目标 数据规模 训练时间 计算成本 主要瓶颈 示例
Pretraining 预测互联网文本的下一个 token，学习世界知识 $>10$T tokens 数月 $>\$10$M 数据与算力 LLaMA、DeepSeek-V3
Reasoning RL 在有客观答案的问题上学习推理 约 $1$M problems 数周 $>\$1$M RL environment 与 reward hacking DeepSeek-R1
Classic Post-training / RLHF 最大化用户效用并符合人类偏好 约 $100$K problems 数天 $>\$100$K 高质量数据与评测 LLaMA-Instruct

其中 Reasoning RL 和 Classic Post-training 通常都被统称为 post-training。

需要注意的是，Reasoning RL 并不是所有模型都必须经历的阶段。普通 instruction model 可以只经过预训练和经典后训练；reasoning model 则会额外使用可验证任务进行强化学习。

1.2 模型专业化：Prompting 与 Finetuning

模型训练完成后，可以通过两种常见方法适配具体任务：

Prompting 的成本最低，但能否找到真正有效的 prompt，仍然取决于是否有可靠的评测集。Finetuning 则适用于医疗、法律、企业知识库等具有专门数据和格式要求的领域。

2. 训练 LLM 时真正重要的五个组成部分

训练一个 LLM 需要同时考虑：

  1. Architecture：例如 Transformer、Mixture of Experts；
  2. Training algorithm / loss：训练算法和优化目标；
  3. Data & RL environment：训练数据以及智能体所处的交互环境；
  4. Evaluation：如何判断模型是否真的进步；
  5. Systems and infrastructure：如何高效使用大规模计算资源。

过去学术研究更关注 architecture、algorithm 和 loss，但在实际训练前沿模型时，真正决定结果的往往是：

  • 数据的数量、质量与配比；
  • 是否有可信、稳定的 evaluation；
  • 系统是否能把训练扩展到足够大的规模。

Part I：Pretraining / 预训练

3. 预训练的基本目标

预训练的目标可以概括为：

使用尽可能多的高质量文本训练模型预测下一个 token，从而让模型学习语言结构、知识和一般模式。

对于一个 token 序列：

$$ x_{1:T}=(x_1,x_2,\dots,x_T), $$

自回归语言模型将其概率分解为：

$$ p_\theta(x_{1:T}) = \prod_{t=1}^{T}p_\theta(x_t\mid x_{<t}). $$

训练时最小化 next-token prediction 的负对数似然，也就是交叉熵损失：

$$ \mathcal{L}_{\text{LM}}(\theta) = -\sum_{t=1}^{T}\log p_\theta(x_t\mid x_{<t}). $$

直观上，如果模型能够在各个领域中准确预测下一个 token，它就必须学到一定程度的语法、语义、事实知识以及代码和推理模式。

3.1 自回归语言模型的处理流程

给定文本 She likely prefers，模型生成下一个 token 的过程为：

  1. Tokenize：将文本转换成 token ID；
  2. Forward pass：输入模型并计算 hidden representations；
  3. Predict distribution：输出词表中各 token 的概率；
  4. Sample / Decode：从概率分布中选择下一个 token；
  5. Detokenize：将 token ID 转换回文本。

其中第 4、5 步主要发生在推理阶段。训练时通常直接使用真实的下一个 token 计算交叉熵，不需要先采样生成结果。

若模型输出 logits $z_v$，则 token $v$ 的概率由 Softmax 给出：

$$ p_\theta(x_t=v\mid x_{<t}) = \frac{\exp(z_v)}{\sum_{u\in V}\exp(z_u)}. $$

4. 从 N-gram 到 Neural Language Model

4.1 N-gram 的统计思想

假设要预测短语 the grass is 后面的词，可以统计语料中的出现次数：

$$ P(X\mid \text{the grass is}) = \frac{\operatorname{Count}(\text{the grass is }X)} {\operatorname{Count}(\text{the grass is})}. $$

例如在 1000 次出现中，有 500 次后面是 green，则：

$$ P(\text{green}\mid \text{the grass is})=0.5. $$

这种方法对常见短语有效，但存在两个核心问题：

  • 必须存储几乎所有 n-gram 的统计量，内存需求极大；
  • 很多长上下文在训练集中只出现一次甚至从未出现，无法泛化到新句子。

4.2 神经语言模型

神经网络用参数化函数代替逐个存储 n-gram 计数：

与 N-gram 相比，神经语言模型能够让语义和上下文相似的输入共享参数，从而对训练集中没有完整出现过的句子进行泛化。

5. Pretraining Data：从互联网到可训练语料

预训练并不是简单地“下载整个互联网然后开始训练”。互联网包含大量广告、重复内容、模板页面、低质量文本、隐私信息以及不安全内容，因此数据处理本身是训练 LLM 的核心工作之一。

5.1 A Typical Data Processing Pipeline

Step 1: Web Crawling

从 Common Crawl 等来源抓取大规模网页数据。原始数据通常以 WARC 格式保存，规模可达到 PB 级。

Step 2: Text Extraction

从 HTML 中提取有效正文，并尽量移除：

  • JavaScript、CSS；
  • 菜单、广告、导航栏；
  • boilerplate（网页模板文本）；
  • 编码错误和异常排版。

数学公式、表格和动态网页通常需要额外处理。由于数据规模巨大，文本提取本身也会消耗大量计算资源。

Step 3: Content Filtering

过滤不适合训练的内容，例如：NSFW content/ harmful content/ PII（个人身份信息）/ 垃圾广告和恶意文本

Step 4: Deduplication（去重）

可在 URL、文档、段落或行等层级进行去重，也可检测 near-duplicate（近似重复）内容。

例如论坛的 header、footer 和菜单会在大量网页中重复出现。若不去重，模型会浪费算力反复学习相同模板。

Step 5: Heuristic Filtering（启发式过滤）

根据简单统计规则移除低质量文档，例如：

  • 文档过长或过短；
  • 重复率过高；
  • 乱码或异常 token 过多；
  • 单词长度分布异常；
  • 包含大量占位文本，如 lorem ipsum；
  • 文档结构明显不自然。

Step 6: Model-based Filtering（基于模型的过滤）

选取 Wikipedia 或被 Wikipedia 引用的网页作为高质量样本，训练分类器判断其他网页是否与其分布相似。

核心思想是：

用少量高质量数据定义“好数据”，再从海量网页中筛选相似内容。

Step 7: Data Mixture（数据配比）

训练数据通常来自多个领域，例如：

  • Web pages；
  • books；
  • scientific papers；
  • code；
  • mathematics；
  • multilingual data；
  • dialogues。

原始互联网中的领域比例通常不是最优训练比例。例如训练 coding model 时，可以提高代码数据的采样权重。

实践中通常先在小模型上测试不同数据配比，再结合 scaling laws 预测其在大规模训练中的效果。

FineWeb 的实验表明，对原始网页数据进行以下处理后，即使剩余 token 数量明显减少，模型性能仍会提升：

原因是训练计算预算有限。模型不应把算力浪费在重复、脏乱或无信息量的 token 上，而应尽可能看到更多独立、高质量的信息。

5.3 Mid-training / 中期训练

预训练结束后，通常还会进行规模较小的 continued pretraining，即 mid-training。其数据规模通常少于预训练数据，可能低于 $1$T tokens。

Mid-training 可以实现：

  • 调整 data mixture，例如增加 science、code 和 multilingual data；
  • 在更高质量的数据上进行 learning-rate annealing；
  • 扩展 context length，例如从 $4$K 提升到 $128$K；
  • 学习聊天格式和初步 instruction following；
  • 加入更高质量 reasoning data；
  • 适配模型希望具备的特殊属性。

长上下文训练往往放在这一阶段，是因为 attention 的计算和内存成本随上下文长度快速增加，若从预训练一开始就使用极长上下文，成本会非常高。

5.4 数据是前沿模型最保密的部分之一

现实中的 LLM 训练在很大程度上是“数据工程问题”。许多模型会公开架构和训练算法，却不会完整公开：

  • 具体用了哪些网页；
  • 不同领域数据的权重；
  • 如何进行 deduplication；
  • 质量分类器如何训练；
  • 是否使用版权受限数据；
  • synthetic data 与 multimodal data 的比例。

一方面，数据处理方法是重要的竞争优势；另一方面，公开数据来源还可能带来版权和法律风险。

常见公开数据集包括：C4/ The Pile/ Dolma/ FineWeb

6. Scaling Laws：为什么“大规模”如此重要？

6.1 Compute、Model Size 与 Data

预训练性能通常与总训练计算量呈较稳定的幂律关系。可用一个简化形式表示：

$$ \mathcal{L}(C) \approx \mathcal{L}_{\infty}+aC^{-b}, $$

其中 $C$ 为训练计算量，$a,b>0$。在 log-log 坐标下，这种关系接近直线。

这意味着研究者可以：

  1. 在多个较小模型上实验；
  2. 拟合不同配置的 scaling curve；
  3. 预测配置扩展到大模型后的表现；
  4. 只对最有希望的配置进行一次昂贵的最终训练。

6.2 从“大模型调参”转向“小模型预测大模型”

旧方法可能将计算资源平均分给 30 个大规模实验，最终每个模型都只训练很短时间。

New pipeline：

  1. 在多种小规模模型上快速搜索 learning rate、data mixture 等配置；
  2. 找到随模型规模变化的 scaling recipe；
  3. 用 scaling law 外推到目标规模；
  4. 将绝大部分预算投入最终大模型的完整训练。

这样可以让最终模型使用大部分计算预算，而不是把算力消耗在失败的大规模试验上。

6.3 比较架构时要关注 scaling rate

当比较 Transformer 与 LSTM 时，不仅要比较小规模下谁的 loss 更低，还要比较随着参数和计算量增加，loss 下降得有多快。

需要关注两个因素：

  • Constant：当前规模下的性能；
  • Scaling rate / slope：扩大规模后性能改善的速度。

即使某个方法在小规模下表现较好，只要它的 scaling rate 较差，在前沿规模下也可能被另一种方法超过。

6.4 Chinchilla Scaling Law

Chinchilla 研究的问题是：在固定训练 FLOPs 下，应该将预算分配给更多参数，还是更多训练 token？

经典结论是，训练计算最优的配置大约满足：

$$ D\approx 20N, $$

其中：

  • $N$：模型参数量；
  • $D$：训练 token 数。

也就是每个参数对应约 20 个训练 token。

但 Chinchilla 只优化训练成本，没有考虑推理成本。对于实际部署而言，使用更小模型并训练更多 token，虽然训练时可能不是最省 FLOPs，却能降低之后每次推理的成本。因此实践中的 token-to-parameter ratio 可能远高于 $20:1$。

6.5 The Bitter Lesson

Richard Sutton 的 Bitter Lesson 强调：

从长期看，能够有效利用持续增长计算资源的方法，往往胜过依赖人工设计复杂规则的方法。

其逻辑是：

  1. 更多计算通常带来更好性能；
  2. 硬件和可用计算资源长期持续增长；
  3. 因此，最重要的是设计能够随计算规模扩展的方法。

这并不意味着算法和数据质量不重要，而是说明：不要只针对当前规模反复添加复杂技巧，应优先选择简单、稳定并且能够 scale 的方案。

7. 训练一个 SOTA 模型需要多少计算？

每一代前沿模型的训练计算量常以接近一个数量级增长，因此算力、能源、集群通信与系统优化越来越重要。

Part II：Post-training / 后训练

8. 为什么只做 Language Modeling 不够？

纯语言模型可能继续生成一串类似问题，而不是回答这个问题，因为它只是在模仿互联网上常见的文本延续方式。

因此需要 post-training，将一个“会续写文本的模型”转化为“会帮助用户的 assistant”。

9. 两种后训练目标

9.1 Classic Post-training / Alignment / Instruction Following

目标：

  • 遵循用户指令；
  • 输出有用、清晰、符合规范的答案；
  • 遵循产品设计者的安全和行为要求；
  • 最大化人类对回答的偏好。

这是 ChatGPT 在 2022 年获得显著成功的重要原因。

9.2 Reasoning Training

Reasoning model 还会在具有可验证答案的困难问题上优化正确率，例如：

  • 数学竞赛题；
  • 代码测试用例；
  • 逻辑推理；
  • 可自动判定对错的任务。

传统 scaling law 主要强调 train-time compute：训练时投入更多计算，模型更好。

Reasoning model 引入了另一种维度：test-time compute。模型在回答时进行更长的搜索、思考或采样，也可能获得更高正确率。

可以将其理解为：一个人被要求立刻回答，通常不如拥有更多思考时间时回答得好。

10. Supervised Fine-Tuning / SFT

10.1 基本思想

SFT 仍然使用 next-token prediction，但训练数据不再是任意互联网文本，而是“理想的输入—输出对”。

10.2 SFT 数据从哪里来？

方法一：Human Demonstrations

让人类直接为 instruction 编写高质量答案。这种方法质量可控，但速度慢、成本高，而且很难覆盖大量复杂任务。

方法二：LLM-generated Data

Alpaca 的思路是使用更强的 LLM 生成 instruction-answer pairs，然后用这些数据微调开源模型。

这种方法开启了大规模 synthetic data generation，但存在一个前提：通常需要一个比目标模型更强的 teacher model。

方法三：Rejection Sampling with Verifiers

当没有更强 teacher model 时，可以：

  1. 让当前模型对同一问题生成多个答案；
  2. 使用测试用例、规则或 judge 判断答案是否正确；
  3. 只保留通过验证的答案；
  4. 用这些优质样本进行 SFT。

这相当于让模型从自己的多个候选中筛选成功轨迹，再模仿这些成功轨迹。

10.3 SFT 可以教会模型什么？

  • Instruction following；
  • 特定格式、语气和风格；
  • Bullet points、JSON、代码格式；
  • Tool use；
  • 初步 reasoning pattern；
  • 任何能够构造高质量 input-output pairs 的行为。

SFT 既可以作为最终训练阶段，也可以作为 RL 之前的 warm start，让模型先具备基本成功率，再进行强化学习。

10.4 Agent Tool Use 的 SFT 数据管线

对于工具调用，仅仅收集真实用户调用真实 API 的轨迹可能远远不够。Kimi K2 使用了更复杂的 synthetic pipeline：

  • 一个 LLM 模拟用户；
  • 一个 agent model 决定工具调用；
  • 一个 LLM 或环境模拟工具反馈；
  • 使用 rubric 和 verifier 对完整轨迹进行 rejection sampling。

这说明 agent SFT 不只是收集问答对，而是需要构造：

$$ \text{User request} \rightarrow \text{Tool selection} \rightarrow \text{Tool arguments} \rightarrow \text{Environment response} \rightarrow \text{Final answer}. $$

10.5 SFT 需要多少数据？

SFT 对已经在预训练中学过的能力非常 sample-efficient。

  • 对 style、format 和 instruction following，约几千到一万条高质量样本可能已经有效；
  • LIMA 表明，少量高质量数据就能显著改变模型回答风格；
  • 对 reasoning、tool use 等预训练中不充分的复杂能力，则需要更多样本；
  • DeepSeek-R1 的推理 SFT 数据达到约 800K 量级。

核心判断是：

如果能力已经在预训练中存在，SFT 主要负责把它“调出来”；如果能力本身没有学会，则需要更多数据，甚至需要 RL。

11. Reinforcement Learning / 强化学习

11.1 为什么 SFT 不够？

SFT 本质上是 behavior cloning，存在三个主要限制。

1. 受到示范者能力上限约束

人类可能无法亲自写出最优答案，但仍然能够判断两个答案中哪个更好。若只模仿人类生成结果，就无法超越示范质量。

2. 可能学习 Hallucination

假设参考答案包含一个模型在预训练中并不知道的文献引用。SFT 会要求模型模仿这个引用形式，却无法让模型真正知道引用是否存在。

结果可能是模型学会：

当被要求给出引用时，即使不知道，也生成一个看起来合理的引用。

3. 理想答案成本高

生成完整高质量答案通常比判断答案好坏更困难、更昂贵。

因此 RL 的思想不是“复制理想行为”，而是“直接最大化目标行为”。

11.2 Reward 从哪里来？

Rule-based Reward

适用于答案容易自动验证的任务：

  • exact string match；
  • 数学最终答案；
  • 代码是否通过 test cases；
  • 输出是否满足 JSON schema；
  • 工具调用参数是否合法。

Reward Model

使用人类偏好数据训练一个 reward model，使其预测某个回答是否更符合人类偏好。

LLM as a Judge

使用更强 LLM 评价答案的正确性、相关性、清晰度或安全性。

这些 reward 都可能存在偏差，因此需要额外的 adversarial evaluation，避免模型只学会利用评分器漏洞。

11.3 Reward Hacking

模型会严格优化被写进 reward function 的目标，而不一定实现设计者真正想要的目标。

例如，若目标只是“通过代码测试”，模型可能尝试：

  • 修改或绕过测试；
  • 删除检查逻辑；
  • 返回固定结果；
  • 利用环境漏洞。

这类行为称为 reward hacking。对于 agent，reward hacking 更危险，因为 agent 可以修改文件、调用 API 或影响外部环境。

因此 RL 环境不仅要能提供 reward，还要防止模型以非预期方式获得 reward。

11.4 DeepSeek-R1 的混合 Reward

对于不同任务，DeepSeek-R1 使用不同反馈：

  • 数学、代码和逻辑推理：rule-based verifier；
  • 写作、事实问答和翻译：helpfulness / safety reward model。

其流程可以理解为：

$$ \text{SFT checkpoint} \rightarrow \text{Generate rollouts} \rightarrow \text{Verifier / Reward model} \rightarrow \text{RL update} \rightarrow \text{Reasoning model}. $$

12. GRPO 的直观理解

对于问题 $q$，policy model 生成一组回答：

$$ o_1,o_2,\dots,o_G. $$

每个回答获得 reward：

$$ r_1,r_2,\dots,r_G. $$

然后在组内进行标准化，得到相对 advantage。一个便于理解的简化形式为：

$$ A_i = \frac{r_i-\bar r}{s_r+\epsilon}, $$

其中 $\bar r$ 为组内平均 reward，$s_r$ 为标准差。

直观上：

  • 比组内平均水平更好的回答得到正 advantage；
  • 比平均水平更差的回答得到负 advantage；
  • policy 增加高 advantage 回答的概率，降低低 advantage 回答的概率。

可将其近似理解为最大化：

$$ J(\theta) \approx \frac{1}{G}\sum_{i=1}^{G} A_i\log \pi_\theta(o_i\mid q) - \beta D_{\mathrm{KL}} \left( \pi_\theta\,\|\,\pi_{\mathrm{ref}} \right). $$

这里的 KL regularization 用于限制训练后的 policy 不要偏离 reference model 过远，以降低训练崩溃和 reward hacking 的风险。

上式用于帮助理解 GRPO 的核心结构，并非课件中完整的 clipped objective。

13. Agent RL 的最大困难之一：Infrastructure

RL 与 SFT 的一个重大区别是：SFT 使用已经准备好的固定数据，而 RL 必须不断让模型在线生成新的 rollout。

对于每个问题，可能需要：

  • 生成多个候选答案；
  • 执行很长的 reasoning trajectory；
  • 多次调用工具；
  • 等待外部 API 或 sandbox；
  • 执行 verifier；
  • 再根据结果更新模型。

13.1 Sampling Bottleneck

GRPO 等算法要求每个 prompt 采样多个答案。若回答很长，大量 GPU 时间会用于 autoregressive decoding，而不是权重更新。

13.2 Long-tail Rollouts

部分 agent trajectory 会异常长。如果训练系统等待最慢轨迹完成，所有 GPU 都可能被拖住。

一种策略是暂停极长 rollout，先使用已经完成的样本更新模型，再在之后恢复未完成轨迹。

13.3 Environment Latency

Agent 调用工具时，GPU 可能只是等待环境反馈。解决方法包括：

  • 大量 concurrent rollouts；
  • 独立的 environment services；
  • 异步调度；
  • 让其他 rollout 在某个 trajectory 等待 API 时继续运行。

13.4 Train、Checkpoint 与 Inference Engine

大规模 RL 系统通常需要分离：

  • Training engine；
  • Checkpoint / weight broadcasting engine；
  • Inference / sampling engine；
  • Environment services。

Kimi 的系统将相关 engine 尽量 colocate，以减少跨机器通信和权重同步开销。

这也是 agent training 与普通问答模型训练的重要区别：环境和 rollout 系统本身就是训练算法的一部分。

14. RLHF：从人类反馈中学习

14.1 基本思想

RLHF 不要求人类亲自写出完美答案，而是让人类比较候选回答。

典型流程：

  1. 从一个较好的 SFT model 出发；
  2. 对每个 instruction 生成两个或多个答案；
  3. 让 annotator 选择更好的答案；
  4. 用偏好数据训练 reward model，或直接使用 preference optimization；
  5. 使用 PPO、DPO 等方法让模型更倾向于生成被偏好的答案。

若人类偏好 $y^+$ 胜过 $y^-$，可以训练 reward model 满足：

$$ r_\phi(x,y^+)>r_\phi(x,y^-). $$

常见 pairwise preference loss 为：

$$ \mathcal{L}_{\text{RM}}(\phi) = -\log \sigma \left( r_\phi(x,y^+)-r_\phi(x,y^-) \right). $$

14.2 为什么 RLHF 能超过 SFT？

人类“判断哪个答案更好”通常比“从零写出最优答案”更容易。因此偏好优化能够利用人类的判断能力，而不受其生成能力完全限制。

$$ \text{Pretrained model} < \text{SFT model} < \text{RL-optimized model}. $$

这里的“小于”表示在目标人类偏好或特定任务表现上通常逐步提高，并不意味着所有能力都单调提升。

14.3 Human Preference Data 的问题

  • 收集速度慢；
  • 成本高；
  • 需要非常详细的 annotation guideline；
  • 标注者容易关注长度、排版等形式，而不是真正的正确性；
  • 标注者群体分布会影响模型价值偏好；
  • 存在 crowdsourcing ethics 问题。

14.4 使用 LLM 代替 Human Annotator

可以让更强 LLM 比较回答，生成 synthetic preference data。实践中，LLM judge 往往比人类众包便宜一个数量级以上，而且一致性可能更高。

但这种方法有明显边界：

  • judge model 的偏差会被继承；
  • 当前沿模型已经接近最强模型时，难以找到可靠 teacher；
  • judge 可能偏爱更长、更自信或特定风格的回答；
  • 需要额外验证 judge 是否真正关注正确性。

Part III：Evaluation / 评测

15. 为什么 Evaluation 是核心能力？

Evaluation 主要用于回答三个问题：

  1. Identify improvements：训练方法是否真的有效？
  2. Select models：多个模型中应该使用哪一个？
  3. Production readiness：模型是否已经足够可靠，可以投入实际应用？

没有可靠评测，就无法知道：

  • 一个新的数据配比是否更好；
  • RL reward 是否被模型利用；
  • 某项能力提升是否损害其他能力；
  • 模型在真实 agent environment 中是否稳定。

16. Close-ended Evaluation

Close-ended task 只有少量可能答案，因此可以自动判定正确性，例如：

  • Multiple Choice Questions；
  • 数学最终答案；
  • 代码测试；
  • 分类问题。

若预测答案为 $\hat y_i$，真实答案为 $y_i$，最简单的 accuracy 为：

$$ \operatorname{Acc} = \frac{1}{n}\sum_{i=1}^{n} \mathbf{1}[\hat y_i=y_i]. $$

MMLU 是典型的 close-ended benchmark。

16.1 Close-ended Evaluation 的问题

Prompt Sensitivity

同一个问题使用不同 prompt、few-shot examples、答案格式或选项顺序，可能得到不同结果。

Train-test Contamination

公开 benchmark 很可能已经出现在互联网训练数据中。模型可能因为记住题目而获得高分，而不是具备真正的泛化能力。

对于日常开发，固定 eval 仍然可以用于比较实验；但对外声称模型具有某种新能力时，contamination 会严重影响结论。

17. Open-ended Evaluation

ChatGPT 类模型可以执行写作、总结、代码、对话和解释等大量开放任务。其答案：

  • 很长；
  • 不存在唯一标准答案；
  • 无法通过字符串匹配判断质量。

因此通常采用 pairwise preference evaluation。

17.1 Human Evaluation：Chatbot Arena

Chatbot Arena 的基本方法是：

  1. 用户在不知道模型身份的情况下与两个模型交互；
  2. 用户选择更好的回答；
  3. 聚合大量 pairwise comparison，得到相对排名。

优点是接近真实用户偏好，缺点是成本高、反馈慢，而且用户群体本身也可能存在分布偏差。

17.2 LLM-as-a-Judge：AlpacaEval

对每个 instruction：

  1. baseline model 生成回答；
  2. candidate model 生成回答；
  3. judge LLM 比较两个回答；
  4. 统计 candidate 获胜的概率。

$$ \operatorname{WinRate} = \frac{1}{n} \sum_{i=1}^{n} \mathbf{1} [ \text{candidate wins on }i ]. $$

这种方法比大规模人工评测快得多、便宜得多，并且可以与 Chatbot Arena 排名保持较高相关性。

17.3 Spurious Correlation

LLM judge 可能偏爱：

  • 更长的回答；
  • 更复杂的格式；
  • 更自信的语气；
  • 重复问题内容的回答。

因此，模型可能通过增加长度提高 win rate，却没有真正提高内容质量。

一种处理方法是使用 regression 或 causal adjustment 控制回答长度，例如 length-controlled win rate。

对于 agent evaluation，还应进一步测量：

  • task success；
  • tool-call correctness；
  • trajectory length；
  • latency；
  • token / API cost；
  • error recovery；
  • robustness to environment changes；
  • 是否出现 reward hacking。

Part IV：Systems and Infrastructure

18. 为什么不能简单地“多买 GPU”？

GPU 不仅昂贵，而且稀缺。即使拥有大量 GPU，也会遇到：

  • 模型无法放入单卡；
  • GPU 之间通信速度有限；
  • 内存带宽成为瓶颈；
  • 大规模同步导致空闲等待；
  • rollout 和 environment 延迟使 GPU 利用率下降。

因此训练前沿模型不仅是 ML 问题，也是一项大型分布式系统工程。

19. GPU 的三个关键特点

19.1 Massively Parallel

GPU 在大量线程上执行相同指令、处理不同数据，更适合 throughput-oriented workload，而不是 CPU 式的复杂串行控制流。

19.2 Fast Matrix Multiplication

现代 GPU 包含专门的 Tensor Cores，矩阵乘法吞吐量可以远高于普通 floating-point operation。

因此高效模型通常会尽量将计算组织为大型矩阵乘法。

19.3 Compute 比 Memory 与 Communication 增长更快

现代 GPU 的算术能力提升速度远快于内存带宽和通信速度。因此多数训练并不是“算不动”，而是“来不及把数据送到计算单元”。

即：Bottleneck是Data movement，而不一定是 FLOPs 本身。

19.4 Memory Hierarchy

越靠近计算核心的存储：

  • 容量越小；
  • 速度越快。

越远的全局内存：

  • 容量更大；
  • 访问延迟更高。

因此优化目标之一，是尽量重复使用已经加载到高速存储中的数据。

19.5 Model FLOP Utilization / MFU

$$ \operatorname{MFU} = \frac{\text{Observed FLOP/s}} {\text{Theoretical peak FLOP/s}}. $$

在大规模训练中，达到约 50% 的 MFU 已经可能是很好的结果，因为通信、内存访问和非矩阵操作都会消耗时间。

20. Low Precision 与 Automatic Mixed Precision

使用更少 bit 可以：

  • 降低显存占用；
  • 减少通信量；
  • 提高矩阵乘法速度。

深度学习对许多中间计算并不需要 FP32 的完整精度，因此常使用 BF16。

Automatic Mixed Precision 的典型做法是：

  • Master weights 保持 FP32；
  • 前向和矩阵乘法使用 BF16；
  • Activations 使用 BF16，减少显存；
  • Gradients 可使用 BF16；
  • 参数更新等对精度敏感的步骤使用 FP32。

其思想不是所有计算都降低精度，而是在速度、内存和数值稳定性之间分配不同精度。

21. Operator Fusion、Tiling 与 FlashAttention

21.1 Operator Fusion

普通 PyTorch 程序中，每个算子可能都需要反复读取，若将多个算子融合到一个 kernel 中，可以只读取一次数据，在片上完成多个操作后再写回。

torch.compile 等工具可以自动完成部分 operator fusion。

21.2 Tiling

Tiling 将矩阵切分成小块，并改变运算顺序，使已经加载到高速缓存中的 tile 被重复使用。

以矩阵乘法 $P=MN$ 为例，不再每次从 DRAM 读取完整行列，而是：

  1. 加载 $M_{ij}$ 与 $N_{jk}$ 的小块；
  2. 在 shared memory / SRAM 中计算多个 partial sums；
  3. 重复利用这些 tile；
  4. 最后写回结果。

这可以显著减少 global memory reads。

21.3 FlashAttention

FlashAttention 通过：

  • Kernel fusion；
  • Tiling；
  • 避免将完整 attention matrix 写入显存；
  • 在需要时 recomputation；

降低 attention 的 IO 成本。它没有改变 attention 的数学结果，却能获得显著端到端加速。

这个例子说明：许多大型模型性能提升并不来自新的 ML 算法，而来自更合理的算子执行顺序和内存访问设计。

22. Parallelization / 并行化

22.1 训练内存估算

对于 $P$ 个参数，若参数和梯度使用 FP32，Adam 保存两个 FP32 moment，基础内存约为每个参数 16 bytes：

$$ \text{Memory} \approx 4P_{\text{weights}} + 4P_{\text{gradients}} + 8P_{\text{optimizer}} = 16P\ \text{bytes}. $$

若 $P$ 以 billion parameters 为单位，则约为：

$$ 16P\ \text{GB}. $$

因此一个 7B 模型仅这些状态就大约需要：

$$ 16\times 7=112\ \text{GB}. $$

这还没有包含 activations 等额外显存。

22.2 Naive Data Parallelism

流程：

  1. 每张 GPU 保存完整 model 与 optimizer；
  2. 将 batch 分给不同 GPU；
  3. 各 GPU 独立计算 gradient；
  4. 通过 all-reduce 汇总 gradient；
  5. 所有 GPU 执行相同更新。

优点：提高吞吐量。
缺点：每张卡仍保存完整模型，不能解决模型放不进单卡的问题。

22.3 ZeRO / Sharding

ZeRO 将训练状态拆分到不同 GPU：

  • shard optimizer states；
  • shard gradients；
  • shard parameters。

每张 GPU 只保存自己负责的部分，并在需要时通信，从而显著减少单卡显存占用。

22.4 Pipeline Parallelism

将不同层放在不同 GPU：

$$ \text{GPU}_1: \text{layers }1\!:\!k, \quad \text{GPU}_2: \text{layers }k+1\!:\!m, \quad \dots $$

输入依次通过各 GPU。为了降低 pipeline bubble，通常会将 batch 划分为 micro-batches。

22.5 Tensor Parallelism

将单个矩阵或单层内部的计算拆到多张 GPU，例如：

$$ W= \begin{bmatrix} W_1\\ W_2 \end{bmatrix}, $$

不同 GPU 分别计算部分矩阵乘法，再通过通信聚合结果。

Pipeline parallelism 是“按层切分”，tensor parallelism 是“在一层内部切分”。

22.6 Mixture of Experts / Architecture Sparsity

MoE 不要求每个 token 都经过全部参数。Router 根据 token 表示选择少量 expert：

$$ h' = \sum_{e\in\operatorname{TopK}(h)} \alpha_e(h)f_e(h). $$

因此可以在相近 active FLOPs 下增加总参数量，并将不同 expert 部署在不同 GPU 上。

代价包括：

  • routing imbalance；
  • all-to-all communication；
  • expert capacity 管理；
  • 训练和部署系统更复杂。

Appendix A：Perplexity

Perplexity 是语言模型验证损失的一种更直观表达。

对于序列 $x_{1:L}$：

$$ \operatorname{PPL}(x_{1:L}) = \exp \left( -\frac{1}{L} \sum_{i=1}^{L} \log p(x_i\mid x_{<i}) \right). $$

也可以写为：

$$ \operatorname{PPL}(x_{1:L}) = \left( \prod_{i=1}^{L} \frac{1}{p(x_i\mid x_{<i})} \right)^{1/L}. $$

直观上，PPL 可以近似理解为模型在每一步平均“犹豫于多少个 token 之间”。

  • 最小值接近 1；
  • 越低表示 next-token prediction 越准确；
  • 由于 tokenizer 和数据集不同，不同模型的 PPL 不一定能直接横向比较；
  • 现代模型通常不再只用 PPL 作为最终能力 benchmark，但它仍是预训练开发中的重要指标。

Appendix B：Tokenizer 与 BPE

Tokenizer 的作用是将文本转换为模型可处理的离散 token ID。

若只使用 word-level tokenizer：

  • 无法处理拼写错误和新词；
  • 词表会非常大；
  • 罕见词问题严重。

若只使用 character-level tokenizer：

  • 通用性强；
  • 但序列过长，训练与推理成本高。

因此现代 LLM 多使用 subword token，让常见词或词片段成为一个 token。

Byte Pair Encoding / BPE

BPE 的简化训练流程：

  1. 收集大规模文本；
  2. 从字符或 byte 级 token 开始；
  3. 统计相邻 token pair 的频率；
  4. 将出现频率最高的 pair 合并为新 token；
  5. 重复合并，直到达到目标 vocabulary size。

例如：

若 l 与 o 经常连续出现，可先合并为 lo；之后可能继续合并为 low。

Tokenizer 会影响：

  • 相同文本需要多少 token；
  • 多语言之间是否公平；
  • 数字、代码和特殊符号的表示效率；
  • context window 能容纳的实际文本量；
  • 模型训练和推理成本。
