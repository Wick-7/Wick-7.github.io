LLM Reasoning：关键思想、方法与局限

1. Reasoning 在 LLM Agent 中的位置

传统 LLM 的基本交互形式是：

$$ \text{Text Input} \rightarrow \text{LLM} \rightarrow \text{Text Output}. $$

LLM Agent 则将 LLM 作为负责 Reasoning & Planning 的核心模块，并进一步接入：

  • 外部环境及其反馈；
  • Tool Use；
  • Retrieval / Knowledge Base；
  • Working Memory 与 Long-term Memory；
  • 多模态感知和具身行动；
  • 其他智能体与人类。

因此，Agent 需要完成的并不是一次性的文本生成，而是一个动态闭环：

$$ \text{Observe} \rightarrow \text{Reason / Plan} \rightarrow \text{Act} \rightarrow \text{Receive Feedback} \rightarrow \text{Update}. $$

真实任务通常包含试错、任务分解、工具调用、错误恢复和长程决策。Reasoning 是这一闭环中的基础能力：如果模型无法可靠地分解问题、维持中间状态并根据反馈修正计划，那么工具和记忆模块也难以被有效利用。

2. 为什么需要推理能力

2.1 对 AI 的一个基本期待：从少量样例中学习

Denny Zhou 对 AI 的一个“较小但关键”的期待是：

AI 应该像人类一样，仅通过少量样例就能理解一个新任务。

传统机器学习长期研究数据高效学习，包括：

  • Semi-supervised Learning；
  • Bayesian Nonparametrics；
  • Kernel Machines；
  • Sparsity / Low Rank；
  • Active Learning。

但对于一些人类看一两个例子就能掌握的简单规则，普通监督学习仍可能需要大量标注数据。讲者认为，其中缺少的关键能力是 Reasoning：

人类能够从少量样例中学习，不只是因为统计归纳能力，更因为人类可以识别规则并执行推理。

3. 核心思想：通过中间步骤得到最终答案

3.1 LLM 的基本生成方式

LLM 是自回归 next-token prediction 模型。

推理过程同样可以被写成一个 token 序列。因此，只要训练数据、微调数据或提示示例中包含中间步骤，模型就可以学习生成类似的步骤。

3.2 中间步骤方法的发展脉络

1. Training with Intermediate Steps

2. Fine-tuning with Intermediate Steps

3. Prompting with Intermediate Steps

Chain-of-Thought Prompting 不再要求重新训练或微调模型，而是在 few-shot demonstration 中直接展示推理过程，引导大模型继续生成类似的多步推理。

3.3 真正关键的不是采用哪种接口

Intermediate Steps 很重要。

只要模型接触到带有中间步骤的示例，它就更可能在回答新问题时生成中间步骤。

4. Chain-of-Thought Prompting

4.1 基本形式

普通 prompting：

CoT prompting：

其目标不是让模型只模仿答案格式，而是让模型模仿一个逐步计算或推导的过程。

4.2 CoT 的主要作用

CoT 特别适用于需要串行计算或组合多个局部结论的任务，例如：

  • Arithmetic Reasoning；
  • Multi-hop Question Answering；
  • Commonsense Reasoning；
  • Symbolic Reasoning；
  • Planning 与 Tool-use Decision。

从 Agent 的角度看，中间步骤还可以自然地对应：

$$ \text{Task Decomposition} \rightarrow \text{Subgoal Selection} \rightarrow \text{Tool Call} \rightarrow \text{Observation Integration}. $$

4.3 需要注意：CoT 并不等于“模型像人一样思考”

LLM 本质上仍然是概率式 next-token generator。生成一段语言上连贯的 reasoning path，不保证：

  • 每一步都真实反映模型内部计算；
  • 推理过程与最终答案一致；
  • 推理链没有遗漏或虚构；
  • 模型能可靠判断自己的答案是否正确。

因此，CoT 应被视为一种改善推理计算和可观测性的技术，而不能直接等同于可靠、忠实的人类式思考。

5. Least-to-Most Prompting：从简单到复杂

仅仅要求“逐步思考”还不够，还可以在 demonstration 中加入更加明确的 reasoning strategy。

5.1 核心策略：Decomposition

Least-to-Most Prompting 将复杂问题拆解为一组由易到难的子问题：

$$ Q \rightarrow (q_1,q_2,\ldots,q_m) \rightarrow (a_1,a_2,\ldots,a_m) \rightarrow A. $$

其中，后续子问题可以利用前面已经得到的答案：

$$ P(a_i\mid q_i,a_{<i}). $$

例子：

Elsa 有 3 个苹果，Anna 比 Elsa 多 2 个。两人一共有多少苹果？

先分解为：

  1. Anna 有多少个苹果？
  2. Elsa 和 Anna 一共有多少个苹果？

再依次求解：

$$ 3+2=5, $$

$$ 3+5=8. $$

5.2 为什么能支持 Easy-to-Hard Generalization

普通 few-shot CoT 通常直接展示完整问题和完整推理，而 Least-to-Most 额外教给模型一个可复用的元策略：

先识别依赖关系，再把复杂任务转化为若干局部可解的问题。

这种方式对于 compositional generalization 特别有效：测试问题可以比 demonstration 更长、更复杂，只要它仍由已学会的基本操作组合而成。

5.3 实验结果

  • 在 SCAN compositional generalization 任务中，仅使用约 $0.1\%$ 的 demonstration examples，Least-to-Most 取得接近完美的泛化表现。
  • 在 CFQ Text-to-Code 任务中，Dynamic Least-to-Most 仅使用约 $1\%$ 数据，超过许多使用完整训练集和专门架构的方法。

Dynamic Least-to-Most 的关键是：不固定使用同一组分解示例，而是根据当前问题动态检索或构造更相关的子问题与示例。


6. 为什么中间步骤有效

6.1 串行问题需要串行计算

许多 reasoning tasks 是 inherently serial 的：后一步必须依赖前一步结果，例如多位数加法、长链逻辑推导和多跳规划。

课程引用的理论工作指出：

  • 固定深度的 Transformer，如果允许生成足够长的 intermediate reasoning steps，可以解决广泛的串行问题；
  • 如果要求模型直接输出最终答案，则可能需要非常深的网络，甚至无法完成某些任务。

6.2 直观理解：把输出长度转化为额外计算时间

Transformer 的一次前向传播深度有限，但自回归生成允许模型反复执行：

$$ \text{Read current state} \rightarrow \text{produce next intermediate state}. $$

因此，生成 $T$ 个中间 token 可以被直观理解为模型获得了额外的串行计算步数。中间输出既是答案的一部分，也是后续计算可以读取的外部工作记忆。

这解释了为什么以下方法可能有效：

  • 让模型 think longer；
  • 增加更细粒度的中间步骤；
  • 将复杂搜索交给 MCTS、Search 或其他外部工具；
  • 在 Agent 中把 reasoning path 展开为多个 action-observation cycles。

6.3 不能无限延长推理链

更长的推理不一定始终更好。推理链过长会带来：

  • 错误累积；
  • 无关分支增加；
  • 推理成本与延迟上升；
  • 生成概率与真正的答案概率不一致。

因此，还需要选择、验证或聚合多条推理路径。

7. Zero-shot CoT 与 Analogical Reasoning

7.1 Zero-shot CoT

Kojima et al. 发现，只需在问题后添加：

就可能触发模型生成分步推理，而不需要 few-shot demonstrations。

优点：

  • 使用简单；
  • 不需要人工准备示例；
  • 可以快速测试模型是否具备潜在推理能力。

局限：

  • 通常弱于高质量 few-shot CoT；
  • 提示本身没有提供具体的任务策略或相关知识；
  • 对不同模型、任务和措辞较敏感。

7.2 LLMs as Analogical Reasoners

人类解决新问题时，往往会先问：

我是否见过一个相关问题？它使用了什么知识或方法？

Analogical Reasoning 不直接使用一组固定人工示例，而是让模型针对当前问题，自适应生成：

  • related problems；
  • relevant knowledge；
  • useful solution strategies；
  • solved exemplars。

然后再利用这些内容解决原问题。

其基本流程可表示为：

$$ \text{Target Problem} \rightarrow \text{Generate Related Examples / Knowledge} \rightarrow \text{Solve Examples} \rightarrow \text{Transfer Strategy to Target}. $$

与固定 few-shot CoT 相比，它的优势是每个测试问题都可以获得更加相关的上下文，而不是被迫依赖同一组 demonstrations。

7.3 实验观察

课程中的结果显示，自生成 exemplars 在 GSM8K、MATH、BIG-bench 和 Codeforces 等任务上，整体优于或匹配普通 zero-shot / few-shot CoT。加入 self-generated knowledge 后，部分代码与复杂推理任务还能进一步提升。

这说明 prompt 中示例的价值不仅取决于数量，还取决于：

  • 与目标问题的相关性；
  • 是否包含正确的知识；
  • 是否展示了可迁移的解题结构。

8. 不依赖显式提示的 CoT Decoding

8.1 核心问题

即使不添加 $Let's think step by step$，预训练 LLM 的概率分布中是否已经存在带推理过程的回答？

课程给出的答案是：存在，但 greedy decoding 未必会选中它。

8.2 Greedy Decoding 的局限

Greedy decoding 每一步都选择当前概率最高的 token：

$$ \hat y_t = \arg\max_{y_t}P(y_t\mid x,y_{<t}). $$

如果答案开头概率最高的 token 直接对应一个短答案，模型可能立即结束，而不会进入一条更长但更可靠的 reasoning path。

8.3 Chain-of-Thought Decoding

课程中的方法从第一个生成位置的 top-$k$ token 分别展开：

  1. 枚举若干高概率起始 token；
  2. 从每个起始 token 继续解码；
  3. 得到多条直接回答或推理路径；
  4. 比较路径末尾最终答案的置信度，选择更可信的路径。

例子中，直接生成 $Even.$ 或 $Odd.$ 时，最终答案 token 的置信度并不高；若路径先生成出生年份，再判断奇偶，最终答案的置信度会明显升高。

8.4 关键结论

  1. 预训练模型内部已经包含逐步推理路径，但标准 greedy decoding 可能无法触发它们。
  2. 有推理过程的路径上，模型对最终答案可能具有更高置信度。
  3. Prompting 不是激发推理能力的唯一方式，Decoding Strategy 同样重要。

课程展示的结果中，CoT decoding 在 GSM8K 和 MultiArith 上显著优于 greedy decoding，且在较小的预训练模型上提升尤其明显。

9. Self-Consistency：从概率角度理解推理

9.1 单条最优推理路径与最优答案并不一致

设：

  • $x$：问题；
  • $r$：reasoning path；
  • $a$：final answer。

普通解码往往近似寻找概率最大的完整序列：

$$ (\hat r,\hat a) = \arg\max_{r,a}P(r,a\mid x). $$

但我们真正关心的是概率最大的最终答案：

$$ \hat a = \arg\max_a P(a\mid x). $$

两者并不等价。某一条完整 reasoning path 可能最常见，但多个不同路径可能共同支持另一个答案。

9.2 将 reasoning path 视为 Latent Variable

最终答案的概率应当对所有可能的推理路径做边缘化：

$$ P(a\mid x) = \sum_r P(r,a\mid x). $$

直接精确求和不可行，因此可以用 sampling 近似。

9.3 Self-Consistency 的做法

从模型中采样 $N$ 条推理结果：

$$ (r_i,a_i)\sim P(r,a\mid x), \qquad i=1,\ldots,N. $$

然后选择出现次数最多的 最终答案：

$$ \hat a_{\mathrm{SC}} = \arg\max_a \sum_{i=1}^{N}\mathbf{1}(a_i=a). $$

注意：聚合的是 final answer，而不是整条 reasoning path。因为不同的正确推理过程可以得到相同答案。

9.4 为什么有效

如果多条相互不同的推理路径都独立得到同一答案，那么该答案的边缘概率往往更高。散点图显示：answer consistency 越高，实际 accuracy 通常也越高；当一致性很高时，答案几乎总是正确。

Self-Consistency 可以理解为一种 Monte Carlo maximum marginal inference。

9.5 两个容易混淆的问题

问题一：直接答案也需要反复采样并多数投票吗？

对于只输出一个 token 的直接答案，greedy decoding 已经选择了单步概率最大的 token，因此重复采样再多数投票通常不会带来 CoT self-consistency 的同类收益。

Self-Consistency 的关键在于存在多个不同的 latent reasoning paths，可以通过它们对最终答案进行边缘化。

问题二：让模型一次生成多个回答，再自称选择“多数答案”是否等价？

不等价。Self-Consistency 依赖从模型分布中进行多次相对独立的随机采样；让同一次生成中的多个答案相互依赖，不能可靠近似相同的概率边缘化过程。

9.6 Universal Self-Consistency

对于开放式文本回答，答案可能语义相同但表面形式不同，无法直接进行字符串计数。

Universal Self-Consistency（USC）让 LLM 比较多个候选回答，识别语义上最一致、最具代表性的答案。它将多数投票从 exact match 扩展到 free-form generation。

10. LLM 推理的三类重要局限

10.1 容易被无关上下文干扰

在一道简单算术题中加入一条与问题无关的信息，模型可能错误地把它纳入计算。

例如：

正确计算应为：

$$ 65+15-4=76. $$

但模型可能继续减去无关的 $10$。

实验结论：

  • 在 GSM8K 中加入无关上下文，可造成超过 20 个百分点的性能下降；
  • 加入 $Ignore irrelevant context$ 可以恢复部分性能；
  • 当无关内容大量增长时，性能仍会明显下降。

这说明模型没有稳定地区分：

$$ \text{Available Information} \neq \text{Relevant Information}. $$

对 Agent 而言，这一点尤其危险，因为检索结果、网页内容和工具返回中常包含大量噪声。

10.2 LLM 尚不能可靠地自我纠错

当初始答案错误时，提示模型 review 可能使其找到错误并修正；但当初始答案正确时，同样的 review 提示也可能诱导模型虚构问题，将正确答案改错。

因此，单个成功案例不能证明 self-correction 有效。必须同时考虑：

  • 错误答案被改对的比例；
  • 正确答案被改错的比例；
  • 总体准确率是否真正提高；
  • 是否使用了额外 oracle 信息。

Oracle Feedback

一些工作只在已知模型答案错误时才要求模型修正，相当于隐含使用了 oracle：

$$ \text{Oracle tells model whether correction is needed}. $$

真实部署中，模型通常并不知道自己何时出错。因此，“知道答案错了以后能改”与“能自主判断并纠错”是两个不同能力。

Multi-LLM Debate

多模型辩论可能生成更多候选答案，但当控制总生成次数后，课程中的实验表明它未能超过更简单的 Self-Consistency。提升可能主要来自更多 samples，而不是 debate 本身。

Self-Debug：利用可验证反馈

代码任务中可以运行 unit tests，将测试结果作为外部 oracle：

$$ \text{Generate Code} \rightarrow \text{Run Tests} \rightarrow \text{Receive Concrete Error} \rightarrow \text{Revise}. $$

这类 self-debug 比纯语言式反思更可靠，因为模型获得了可验证的环境反馈。

10.3 Premise Order Matters

对同一个问题，仅改变前提或句子的排列顺序，LLM 的推理性能就可能明显下降。

GSM8K 重排

原题按事件发生顺序描述：取钱、丢失钞票、支付、资金翻倍、换零钱。若把“丢失钞票”移到句末，模型往往按照文本顺序机械执行，将本应较早发生的事件错误地放到最后计算。

课程结果显示，重排 GSM8K 句子后，多种 frontier LLM 的求解率下降约 10 个百分点。

逻辑规则重排

在由随机词构成的纯逻辑推理任务中：

  • 当规则按推导时的使用顺序排列，模型表现较好；
  • 仅随机打乱与 query 相关的规则，就可造成超过 30 个百分点的性能下降。

这说明模型经常依赖局部的顺序启发式，而不是先构建一个与输入顺序无关的完整逻辑结构。

对于 RAG 和 Agent，不能假定只要所有必要信息都出现在 context 中，模型就会自动正确组织它们。信息的排序、分组和呈现方式也会影响结果。

11. 对 LLM Agent 的启示

11.1 Reasoning Trace 可以成为 Agent 的工作空间

Agent 可以将复杂任务转化为：

$$ \text{Goal} \rightarrow \text{Subgoals} \rightarrow \text{Actions} \rightarrow \text{Observations} \rightarrow \text{Updated Plan}. $$

显式中间状态有助于：

  • 任务分解；
  • 工具选择；
  • 保存阶段性结果；
  • 处理长程依赖；
  • 在多个 specialized agents 之间分配子任务。

11.2 Tool Use 可以扩展中间计算

当纯语言推理过长或不可靠时，可以调用：

  • Calculator；
  • Search Engine；
  • Code Interpreter；
  • Database / Retriever；
  • SAT Solver / Theorem Prover；
  • Simulator 或 MCTS。

工具调用相当于将某些 intermediate steps 交给更适合的模块执行。

11.3 验证器比单纯反思更重要

可靠的 agentic loop 不应只依赖：

更理想的是使用可验证反馈：

  • Unit tests；
  • Compiler / Runtime errors；
  • Environment reward；
  • Constraint checker；
  • Independent verifier；
  • Multiple samples + answer aggregation。

11.4 Context Engineering 是推理系统的一部分

由于 irrelevant context 和 premise order 会影响推理，构建 Agent 时需要：

  • 过滤无关检索结果；
  • 对证据进行排序和分组；
  • 将依赖关系显式化；
  • 避免把大量原始内容直接塞入 context；
  • 对长上下文进行结构化摘要；
  • 区分 facts、rules、observations 与 hypotheses。

11.5 不应把语言上自然的过程误认为可靠推理

评估 Agent 时应分别测量：

  1. 最终答案或任务成功率；
  2. 中间步骤是否可执行；
  3. 工具调用是否正确；
  4. 对噪声、重排和长上下文是否鲁棒；
  5. 在没有 oracle 时能否识别错误；
  6. 增加 test-time compute 后是否真正提升，而非仅增加成本。

本节课最重要的结论

  1. Intermediate steps 是提升 LLM reasoning 的核心变量。 它们可以出现在训练、微调、提示或解码阶段。
  2. Reasoning strategy 也很重要。 Decomposition 和 analogy 往往比泛化的“逐步思考”提示更有效。
  3. 推理链是一种额外计算。 对 inherently serial problems，生成中间状态可以弥补固定网络深度的限制。
  4. 一条最可能的推理链不等于最可能的答案。 Self-Consistency 通过采样和边缘化多个路径提升最终答案准确率。
  5. LLM 仍是概率式 token generator，而不是可靠的自主推理者。 无关上下文、自我纠错和前提顺序都会显著影响表现。
  6. 外部可验证反馈非常关键。 Unit tests、环境反馈和 verifiers 比单纯语言反思更适合构建可靠 Agent。
  7. 研究应从正确的问题与第一性原理出发。 不要只堆叠复杂 prompt 或 agent workflow，而应明确失败来自模型、上下文、解码、搜索还是验证环节。

长期目标

自主学会何时分解、何时类比、何时延长推理、何时搜索、何时调用工具、何时验证，同时克服无关上下文、自我纠错和前提顺序等局限。
