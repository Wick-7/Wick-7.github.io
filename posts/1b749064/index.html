Practical Lessons from Deploying Real-World AI Agents

1. 课程概览

这节课并不重点讨论某一种新的 Agent 算法，而是从 Sierra 部署大量 customer-facing agents（面向客户的智能体）的经验出发，讨论 Agent 真正接触现实世界后会遇到的问题。

整节课可以概括为一条主线：

$$ \text{Agent Demo} \rightarrow \text{Agent System} \rightarrow \text{Reliable Product} $$

一个基础 Agent 看起来可能只需要：

  • LLM；
  • RAG；
  • Tool Use；
  • 一个简单的 reasoning / orchestration loop。

但在生产环境中，还需要处理测试、监控、延迟、安全、版本管理、合规、多渠道部署、故障切换、用户身份验证等大量工程问题。

构建 Agent 的难点并不只是让模型“会做事”，而是让它在复杂、开放、非确定性的真实环境中持续、可靠、安全地做对事情。

2. Agent 的三种主要形态

讲者将当前 Agent 大致分为三类。

类型 典型例子 主要作用
Personal Agent ChatGPT、Gemini 作为个人数字助手，为用户完成通用任务
Role-based / Persona-based Agent Coding Agent、Legal Agent 围绕某种职业或角色完成专业工作
Customer-facing Agent 企业客服、销售、售后 Agent 代表企业直接与客户交互并执行业务流程

Sierra 主要关注第三类，即面向客户的 Agent。它们不仅回答问题，还需要真正执行操作，例如：

  • 退换商品；
  • 排查设备故障；
  • 修改订阅计划；
  • 恢复账户；
  • 查询订单；
  • 更新客户信息；
  • 触发业务系统中的真实操作。

这意味着 customer-facing agents 不只是对话系统，而是连接用户、企业知识和业务系统的执行层。

3. AI 的应用层：从 Website、App 到 Agent

讲者将 Agent 类比为新一代技术平台上的核心应用形态：

$$ \text{Internet} \rightarrow \text{Websites},\qquad \text{Mobile} \rightarrow \text{Apps},\qquad \text{AI} \rightarrow \text{Agents} $$

LLM 更像底层能力，Agent 则是用户实际使用 AI 的产品形态。

未来企业可以只定义一次 Agent 的知识、工具和规则，再将其部署到电话、Chat、Email 等不同渠道：

$$ \text{Define Once} \rightarrow \text{Deploy Everywhere} $$

自然语言对话由此成为新的用户界面。Agent 的价值不只是回答问题，而是通过对话直接完成过去需要多层 UI 或人工客服处理的任务。

4. Outcome-based Pricing

软件商业模式大致经历：

$$ \text{一次性购买} \rightarrow \text{SaaS 订阅} \rightarrow \text{按使用量付费} \rightarrow \text{按结果付费} $$

Sierra 采用 outcomes-based pricing：只有 Agent 成功完成退货、换货、销售或客户挽留等目标时，企业才付费。

这种模式要求 Agent 的结果必须可验证，而不能只依赖回答“看起来不错”。

5. Agent Iceberg：真正困难的部分在水面之下

表面上，一个 Agent 似乎只需要：

  • LLM；
  • RAG；
  • Tool Use。

但生产系统还必须处理：

  • 测试、监控与 Trace；
  • 版本控制和发布管理；
  • Prompt Injection、防泄漏与合规；
  • 多渠道接入和人工转接；
  • 延迟优化与模型 Failover；
  • 权限控制、身份验证和业务系统访问。

因此，当前 Agent 生态类似 1997 年的 Web：基础能力已经出现，但成熟的工程栈和标准仍不完善。

行业正在经历：

$$ \text{Agents as Technology} \rightarrow \text{Agents as Product} $$

核心原则是：

Simple, not simplistic.

产品可以易用，但不能牺牲可靠性、安全性和可控性。

6. 从 Transaction 到 Relationship：Agent Memory

当前很多 Agent 是 transactional 的：每次会话相互独立，用户再次联系时需要从头说明问题。

更好的 Agent 应当：

  1. 跨时间、跨渠道持续交互；
  2. 记住过去的重要信息；
  3. 整合企业已有的客户数据；
  4. 从被动响应转向主动服务。

Agent Memory 的关键价值是解决 Warm Start：新会话不再从零开始，而是基于历史交互继续服务。

7. Agent Development Life Cycle

Agent 具有非确定性，因此也需要独立的开发生命周期：

$$ \text{Build} \rightarrow \text{Test} \rightarrow \text{Optimize} $$

构建平台通常同时提供：

  • Code-based SDK：适合工程团队接入 Git、版本管理和发布流程；
  • No-code / Low-code 工具：适合业务团队配置知识、工具和 Policy；
  • Declarative Configuration：描述 Agent 应该做什么，而不是手工编排每次模型调用。

Sierra 强调的不是简单的 Build or Buy，而是 Build with：平台提供底层生产能力，企业保留业务定制空间。

8. Voice Agent：Demo 容易，Production 很难

生产级语音 Agent 通常采用：

$$ \text{Audio} \rightarrow \text{Speech-to-Text} \rightarrow \text{LLM / Orchestration} \rightarrow \text{Text-to-Speech} \rightarrow \text{Audio} $$

这种模块化架构比端到端 audio-to-audio 更容易控制、调试和替换。

语音系统需要同时解决：

  • 低延迟；
  • 背景噪声和多说话人；
  • 打断检测；
  • 专有名词与数字发音；
  • 语气、节奏和情绪；
  • 模型故障切换。

8.1 Latency

真正重要的是用户停止说话到 Agent 开始回答之间的时间：

$$ L_{\text{perceived}} = t_{\text{agent starts speaking}} - t_{\text{user stops speaking}} $$

常见优化方法包括：

  • 并行执行多个步骤；
  • 同时发出多个推理请求，使用最先返回的结果；
  • 用自然的 filler phrase 缓解等待感；
  • 在模型异常时自动 Failover。

这些方法能降低延迟，但会增加计算成本和系统复杂度。

8.2 打断检测

系统需要区分：

$$ \text{Acknowledgement} \neq \text{Semantic Interruption} $$

“嗯嗯”“好的”通常只是附和，而“等等，我不是这个意思”才是真正的打断。普通 VAD 只能检测声音，无法判断语义，因此需要专门的 interruption classifier。

8.3 Transcription 与 Synthesis

传统转录指标：

$$ WER = \frac{S+D+I}{N} $$

但 Voice Agent 更关心：

  • 是否识别主要说话人；
  • 是否忽略背景电视和次要说话人；
  • 是否识别关键实体；
  • 错误是否影响任务完成。

因此，WER 不一定等同于真实体验。

语音合成还要处理：

  • 地址、电话、姓名等实体发音；
  • Prosody 和自然停顿；
  • 简短、口语化的表达；
  • 与用户情绪和企业品牌匹配的声音。

即：

$$ \text{Good Chat Response} \not\Rightarrow \text{Good Voice Response} $$

9. 为什么 Agent 需要新的测试方法

Agent 的同一输入可能产生不同的推理过程、工具调用和对话轨迹，因此传统 Unit Test 和 Integration Test 不够。

测试需要尽量还原真实环境：

  • 多轮对话；
  • 用户情绪与口语表达；
  • 工具调用；
  • 数据库状态变化；
  • Policy 与 Guardrail；
  • 噪声、口音和打断；
  • 用户与 Agent 共同改变环境。

Sierra 早期曾让 user agent 与 business agent 自动对话，但由于没有终止条件，两个 Agent 陷入无限的“感谢—不客气”循环。这说明测试必须观察完整 trajectory，并设置明确的终止和成功条件。

10. $\tau$-Bench：面向真实交互的 Agent Evaluation

$\tau$ 表示 Tool-Agent-User。它评估 Agent 能否在动态、多轮、受政策约束的环境中完成任务。

核心要求包括：

  1. Seamless Interaction：与用户和工具顺畅交互；
  2. Policy Adherence：遵守业务规则；
  3. Consistency at Scale：多次运行仍保持稳定。

10.1 环境组成

每个任务包含：

  • 数据库和可读写 API；
  • Domain Policy；
  • 明确的用户目标；
  • LLM-based User Simulator；
  • 可验证的最终状态。

环境是 dual-control 的：

$$ s_{t+1}=T(s_t,a_t^{agent},a_t^{user}) $$

即用户和 Agent 都可能改变世界状态。

10.2 Objective Evaluation

相比只使用 LLM-as-a-Judge，$\tau$-Bench 更关注最终状态：

$$ \text{Success} = \mathbb{1} \left[ s_{\text{final}}=s_{\text{target}} \land \text{Policy Satisfied} \right] $$

例如检查订单是否真正退款、航班是否修改、身份验证是否完成，而不是只判断回答是否自然。

11. Pass@k 与 Pass$^k$

  • Pass@k：$k$ 次尝试中至少一次成功，强调“能不能做到”；
  • Pass$^k$：连续 $k$ 次全部成功，强调“能否稳定做到”。

若单次成功率为 $p$，则：

$$ P(\text{all }k\text{ runs succeed})=p^k $$

例如：

$$ 0.95^{20}\approx0.358 $$

因此，生产 Agent 不能只追求 best-case capability，还要关注 repeated reliability。

12. Production Simulation 与 Traces

$\tau$-Bench 提供通用评测框架，生产环境还需要持续运行的 Simulation Suite：

$$ \text{Agent Change} \rightarrow \text{Simulation} \rightarrow \text{Regression Analysis} \rightarrow \text{Release / Reject} $$

Simulation 可以覆盖企业自己的数据库、工具、用户 persona、语音噪声和业务 Policy。

Traces 则记录一次响应中的完整执行过程，例如：

  • Agent 推理；
  • Retrieval；
  • Tool Call；
  • Supervisor；
  • 数据库请求；
  • Safety Check；
  • TTS。

它主要用于：

  • Debugging；
  • Latency Optimization；
  • Audit；
  • Root Cause Analysis。

13. Use AI to Improve AI

Primary Agent 周围可以配置多个专门的 Supervisor：

  • Task Supervisor；
  • Grounding Supervisor；
  • Compliance / Safety Supervisor；
  • Tool Supervisor；
  • Prompt Injection Detector。

这种设计能提高安全性和可控性，但也会增加延迟、成本和系统复杂度。

系统还可以利用真实交互持续改进：

$$ \text{Escalated Cases} \rightarrow \text{Human Expert Solutions} \rightarrow \text{Synthesized Knowledge} \rightarrow \text{Improved Agent} $$

即从人工客服成功处理的案例中补充 Agent 的知识缺口。

14. Prompt Injection 与 Agent Security

Agent 能访问知识、调用工具并修改真实业务系统，因此 Prompt Injection 可能直接造成现实风险。

防护应采用分层架构：

$$ \text{Input Guardrails} \rightarrow \text{Rules / Policies / Supervisors} \rightarrow \text{Output Guardrails} $$

关键原则包括：

  • 输入阶段检测 Prompt Injection、Context Poisoning 和异常请求；
  • Agent 内部明确 Rules、Policies 和 Tool Permissions；
  • 输出前检查 Prompt 泄露、PII 和违规内容；
  • 高风险规则由确定性软件强制执行；
  • LLM 不直接无限制访问 CRM 或交易数据库。

正确的工具访问路径是：

$$ \text{LLM Intent} \rightarrow \text{Deterministic Tool Interface} \rightarrow \text{Authentication + Validation} \rightarrow \text{Business System} $$

生产 Agent 同时面临两类风险：

$$ \text{Classic Software Vulnerabilities} + \text{AI-specific Vulnerabilities} $$

因此，Agent Security 必须同时采用传统软件安全、AI Guardrail 和持续 Red Teaming。

15. 整体架构总结

一个生产级 customer-facing Agent 可以抽象为：

在系统外围还需要：

  • Simulation；
  • Regression Testing；
  • Traces；
  • Monitoring；
  • Release Management；
  • Model Failover；
  • Human Handoff；
  • Continuous Optimization。

16. 本节课最值得记住的实践原则

原则一：不要把 Agent 等同于 LLM + RAG + Tools

真正的生产 Agent 是一个完整的软件系统。

原则二：能力只是起点，可靠性才决定能否部署

模型偶尔成功，不代表系统能够服务数百万用户。

原则三：指标必须贴近最终用户体验与业务目标

WER、LLM-as-a-Judge 等通用指标不一定适合具体产品。

原则四：测试真实 trajectory，而不只是单轮答案

需要模拟用户、工具、环境状态、政策约束和多轮变化。

原则五：对可执行任务，应优先使用可验证的最终状态

数据库是否被正确修改，比回答是否“听起来合理”更重要。

原则六：高风险操作必须由确定性软件约束

不要让 LLM 直接、无限制地访问企业核心系统。

原则七：Voice Agent 是完整实时系统问题

语音体验由延迟、识别、打断、表达、情绪和故障切换共同决定。

原则八：Agent 需要 Build–Test–Optimize 的持续闭环

Agent 不是一次性完成的模型应用，而是需要长期迭代的产品。

Post-Training Verifiable Agents传统 Chat Model 的主要目标是生成“人类喜欢的回答”；Agentic Model 在此基础上，还需要在环境中采取行动、调用工具，并最终完成一个能够被检查的任务。因此，Agent 后训练的核心不只是语言风格对齐，而是： $$ \text{理解环境状态} \rightarrow \text{选择并调用工具} \rightarrow \text{改变或查询环境} \rightarrow \text{完成任务} \rightarrow \text{由 Verifier 检查结果} $$ 本节重点关注 Verifiable Agent 的三个关键： Good Training Data：构造多样且高质量的环境、工具、任务和验证器。 Good Evaluations：定义什么是 Agent Intelligence，并用可靠评测量化它。 Good Training Recipes：通过 SFT 与 RL，让模型能够训练更久、学习更难任务并探索更多有效轨迹。 最后进一步提出实践问题：如何把环境、验证器、评测和算法整合成...
