AI Agents to Automate Scientific Discoveries

本节课虽然以 AI for Science 为主要场景，但真正值得关注的是其中展示的 Agent 系统设计趋势：如何让大模型组织专业角色、调用工具、管理长期状态、验证结果，并把静态知识转化为可执行能力。

1. 一句话概括

AI Agent 正在从“回答问题或调用一个工具”，发展为能够围绕开放式目标，持续完成以下循环的复杂系统：

核心变化不是模型单次回答得更聪明，而是 模型开始成为工具、知识、记忆与协作流程的组织者。

2. 这节课的核心观点

2.1 从 AI Tool 到 Agentic System

传统 AI 通常解决边界清晰的单一任务，例如分类、预测或生成。

Agent 系统面对的则是开放式、长周期任务。它需要自己决定：

  • 下一步做什么；
  • 需要调用哪些工具；
  • 是否要搜索新知识；
  • 如何保存中间结果；
  • 什么时候检查、回退或请求人类介入。

因此，Agent 的能力不只取决于底层 LLM，还取决于外围系统是否具备：

2.2 Multi-Agent 的价值在于专业分工，而不是简单增加模型数量

Virtual Lab 使用类似真实团队的结构：

  • Manager / PI Agent：理解目标、拆解任务、协调其他 Agent；
  • Specialist Agents：承担不同专业角色；
  • Coding / Tool Agent：执行代码和调用工具；
  • Critic Agent：检查假设、方法和结果；
  • Human：提供目标、约束、关键审核和现实验证。

多个 Agent 的潜在优势包括：

  • 从不同角度探索方案；
  • 将复杂任务拆给不同专家；
  • 通过讨论和批评发现错误；
  • 并行生成多个候选方案，再由 Manager 汇总。

但 Multi-Agent 并不天然比单 Agent 可靠。多个 Agent 可能共享同一种模型偏差，也可能对错误形成伪共识。因此，真正关键的是角色差异、独立采样、验证机制和明确的聚合方法。

2.3 当前 Agent 的“创新”主要是组合创新

课程案例显示，Agent 更擅长：

  • 发现适合当前问题的已有工具；
  • 修改部分 scoring function 或参数；
  • 将多个工具连接成新的 pipeline；
  • 把已有方法迁移到新的数据或任务。

这种能力可以称为 combinatorial creativity。

它并不等同于从零发明一个全新的基础模型，但已经能够显著降低复杂工作流的设计、实现和试错成本。

2.4 可靠 Agent 必须管理长期状态

开放式任务不能只依赖不断增长的聊天记录。Virtual Lab 使用了分层状态管理：

其中：

  • External Memory 保存历史讨论和中间产物；
  • Manager Agent 周期性总结“已经完成什么、当前状态、下一步是什么”；
  • Critic Agent 检查逐轮累积的错误；
  • Human 在关键节点纠正方向。

这说明长程 Agent 的重点不是无限扩大 Context Window，而是维护一个持续更新、大小受控的 working state，并按需访问外部历史。

3. 课程展示的前沿进展

3.1 动态生成和组织专业 Agent

系统不再固定所有 Agent，而是由 Manager 根据任务动态判断需要哪些专业能力，并创建相应角色。

值得研究的问题包括：

  • 什么任务需要多 Agent；
  • 应创建哪些角色；
  • 如何分配任务和通信预算；
  • 什么时候合并、替换或终止 Agent。

3.2 Agent 自我专业化

Agent School 展示了一条从通用模型到专业 Agent 的路线：

其核心思想是：不同 Agent 可以从同一 Base Model 出发，通过不同数据和工具训练形成 specialization。

不过，参数级持续学习仍面临成本、错误知识写入、灾难性遗忘和能力评估等问题。实践中，RAG、工具说明、Memory 与轻量微调可能会组合使用。

3.3 从静态文档到 Executable Knowledge

Paper2Agent 的重要思想不是“让 Agent 总结论文”，而是将文档、代码、数据、环境和使用流程封装成可调用接口：

这使知识从 可阅读 进一步变为 可执行、可复用和可组合。

相较于每次都让通用 Coding Agent 从头阅读文档、安装环境并理解代码，预先构建并验证好的工具接口通常更稳定，也更节省后续调用成本。

3.4 Agent-to-Agent Collaboration

当不同知识来源都被封装成 Agent 或工具后，一个方法 Agent 可以与一个数据 Agent 自动协作：

这一思路可以推广到软件工程、数据分析和企业流程：不同系统不再只提供 API，而是提供能够解释、规划并协作的 Agent 接口。

3.5 Agent Evaluation 从最终答案走向过程评估

课程中的 AI 审稿实验表明，只看最终文本远远不够。Agent 系统还需要评估：

  • 引用和外部事实是否真实；
  • 中间步骤能否复现；
  • 工具是否被正确调用；
  • Reviewer 或 Critic 是否存在固定偏差；
  • 需要多少人类干预；
  • 长任务中是否发生 topic drift；
  • 最终结果是否经过现实或外部验证。

这也是当前 Agent Evaluation 的重要方向：从静态 QA Accuracy，转向 trajectory、tool use、robustness、cost 与 human effort 的综合评估。

4. 当前比较现实的应用

4.1 复杂研究与分析工作流

Agent 可以帮助完成：

  • 文献和资料检索；
  • 跨来源信息整理；
  • 研究方案或分析计划生成；
  • 代码实现和计算实验；
  • 数据分析与候选方案筛选；
  • 自动整理阶段性状态和报告。

其价值主要在于提高探索速度和减少重复劳动，关键结论仍需要人类审核。

4.2 将论文、代码库或内部文档封装成专业 Agent

一个知识库不再只支持问答，还可以进一步提供：

  • 正确的工具调用；
  • 环境配置；
  • 标准化 workflow；
  • 输入检查；
  • 中间结果和错误诊断。

这可用于：

  • 开源项目使用助手；
  • 企业内部数据分析 Agent；
  • 模型与算法的交互式文档；
  • 面向非技术用户的专业软件接口。

5. 当前能力边界

5.1 更擅长使用和组合已有能力

Agent 目前更擅长调用、修改和连接现有工具，而不是从零创造突破性的基础方法。

5.2 广度较强，深度仍依赖专家

Agent 可以快速连接多个领域，但在非常细分的问题上，容易缺少隐性知识、边界条件和真正深入的机制理解。

5.3 Context 不完整会导致错误假设

仅提供数据或任务描述往往不够，还需要提供：

否则 Agent 可能在错误前提下完成一套看似合理的分析。

5.4 Multi-Agent 可能放大成本与共同偏差

增加 Agent 会带来更多 Token、通信和延迟。若角色设计不合理，多 Agent 只是重复生成相似答案，并不会产生真正的多样性。

5.5 外部验证仍不可缺少

无论是代码测试、数据核验、模拟环境还是现实实验，Agent 输出都需要与外部环境形成闭环。纯语言层面的“互相同意”不能证明结果正确。

6. 对 Agent 研究最值得关注的启发

这节课最值得保留的并不是具体 AI4S 案例，而是下面四个研究问题：

  1. Dynamic Agent Organization
    如何根据任务动态决定使用单 Agent、多 Agent或哪些专业角色？

  2. Bounded Working State + External Memory
    如何用固定大小的当前状态维持长程任务，同时把细节保存在可检索的外部 Memory 中？

  3. Verification-aware Agent
    如何让 Agent 主动调用测试、Critic、环境反馈和真实数据，而不是依靠语言上的自我反思？

  4. Executable Knowledge
    如何把论文、代码库和文档转换成经过验证、能够被 Agent 可靠调用的工具与工作流？

7. 最终总结

Agent 的前沿不只是让 LLM 进行更长的推理，而是构建一个能够组织角色、调用工具、维护状态、验证结果和复用知识的完整系统。

当前最现实的方向不是 Fully Autonomous Agent，而是：

从更一般的 Agent 研究角度看，这节课展示了三条重要趋势：

  • 从单 Agent 到动态组织的协作系统；
  • 从长对话到分层 Memory 和 bounded working state；
  • 从静态知识检索到可执行、可验证的知识接口。
