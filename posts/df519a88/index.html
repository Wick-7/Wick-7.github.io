Compound AI Systems & DSPy Framework

1. 课程主线

现在使用大语言模型快速搭建一个令人印象深刻的 Demo 已经非常容易，但将一个单体语言模型（Monolithic LM）变成可靠、可控、可调试、可持续改进的 AI 系统仍然很困难。

这节课的核心观点是：

与其把所有任务都交给一个黑盒 LM，不如把系统写成一个由多个专门模块组成的程序，并进一步让这些模块的提示词、示例乃至模型权重根据任务指标自动优化。

这对应两个重要概念：

  1. Compound AI System：使用 LM 作为专门组件的模块化程序。
  2. DSPy：以声明式方式编写 LM 程序，并通过 Optimizer 将程序编译为适配具体模型和任务的高质量实现。

Part I：为什么需要 Compound AI Systems

2. 单体 LM 的问题

语言模型的流畅性既是优势，也是风险来源。模型即使产生了错误答案或有 Bug 的代码，也可能表现得非常自信，使错误难以被发现。

所有 AI 系统都会犯错，真正的问题在于：单体 LM 的内部行为高度耦合，因此很难完成以下工作：

  • Control：约束系统以规定方式执行任务；
  • Debug：定位错误来自知识、推理、检索还是工具调用；
  • Improve：只修改系统中的一个局部组件；
  • Attribute：解释答案依据了什么信息；
  • Scale computation：在测试时系统化地投入更多计算。

如果系统只是：

开发者通常只能不断修改一整段 Prompt，或者等待更强的新模型发布。

3. Compound AI System 的定义

Compound AI Systems 是将 LM 作为专门组件使用的模块化程序：

其中，LM 不再直接等价于整个应用，而可能只负责：

  • 生成检索 Query；
  • 对问题进行分解；
  • 根据证据生成答案；
  • 调用工具；
  • 检查事实；
  • 对多个候选结果排序；
  • 反思并修正答案。

系统能力来自模型能力与程序结构的组合。

Compound AI Systems 还允许在推理阶段投入更多计算，例如：

  7. 迭代得到最终答案。

这里的关键不是单次调用的模型突然更强，而是系统能够有组织地搜索更好的输出。

Part II：现有 Prompt-based 系统的问题

6. “Stringly-Typed” Compound AI Systems

许多系统在架构图上非常模块化，但在代码内部仍依赖几千甚至上万 Token 的手写 Prompt。

一个 Prompt 往往同时耦合了五种本应独立的职责：

角色 作用
Signature 定义模块的核心输入到输出行为
Predictor 定义实现该行为所采用的推理策略
Adapter 负责输入格式化、输出解析和类型转换
Metrics / Assertions 定义任务目标、约束和正确性标准
Optimizer 找到适合某个模型和任务的指令、示例或权重

例如，一段 Prompt 可能同时描述：

  • 输入字段是什么；
  • 输出格式是什么；
  • 应该逐步思考；
  • 可以使用哪些工具；
  • 不允许产生哪些内容；
  • 必须如何引用；
  • 遇到格式错误时如何重试；
  • 给出哪些 few-shot examples；
  • 使用什么措辞才能让某个特定模型表现更好。

这种实现被称为过于 stringly-typed：

系统的核心架构与具体 Prompt 措辞、模型习惯、数据集细节和格式约定混在了一起。

后果包括：

  • 更换模型后，原 Prompt 可能失效；
  • 增加一个模块后，需要重新调整整个 Prompt Chain；
  • 更换任务指标后，旧 Prompt 不再合适；
  • 论文中的 Prompt 难以迁移到真实业务；
  • 开发过程依赖大量不可复现的手工试错。

7. 从 Prompt Engineering 转向 Programming

课程提出一个类比：

  • 高级程序语言描述“程序要做什么”；
  • 编译器根据底层硬件生成具体机器代码；
  • 开发者不需要为每种 CPU 手写汇编。

同理，LM 系统也可以：

  1. 用高层抽象描述系统结构与模块行为；
  2. 通过 Optimizer 针对具体 LM、数据与指标生成合适的 Prompt 或权重；
  3. 在更换模型时重新编译，而不是重写整个系统。

这就是 DSPy 的基本愿景：

将 Compound AI Systems 表示为普通程序，其中包含具有模糊自然语言类型、能够从数据中学习行为的模块。

Part III：DSPy 的核心抽象

8. 什么是 LM Program

设一个 LM 程序为：

$$ \Phi:\mathcal{X}\rightarrow\mathcal{Y}, $$

其中输入与输出都可以是自然语言对象。

程序执行过程中会调用若干 LM 模块：

$$ M_1,M_2,\ldots,M_{|M|}. $$

每个模块可以写成：

$$ M_i:\mathcal{X}_i\rightarrow\mathcal{Y}_i, $$

并通过自然语言定义：

  • 可选的子任务描述；
  • 输入字段及其语义；
  • 输出字段及其语义。

与普通 Python 函数不同，这些模块并没有完全确定的实现。其具体行为需要由 Adapter、Predictor 和 Optimizer 转化为 LM 调用。

  • Signature 类似输入输出维度；
  • Predict、ChainOfThought 等类似可组合的层类型；
  • Optimizer 则负责为这些模块寻找具体可执行参数。

11. Adapter：在抽象与具体模型之间转换

Adapter 负责把 Signature 和 Predictor 转换为具体模型能够处理的输入格式，并将模型输出解析回结构化字段。

例如：

Adapter 负责的内容包括：

  • 选择 Chat 或 Instruct 格式；
  • 排列输入字段；
  • 告诉模型输出哪些字段；
  • 解析结构化输出；
  • 转换类型；
  • 必要时处理格式错误。

这个基础 Prompt 不一定已经最优，但可以作为编译和优化的起点。

12. Metric 与 Assertion

DSPy 需要一个系统级目标判断程序表现是否良好。

Metric 可以是：

  • Exact Match；
  • F1；
  • 检索是否包含目标文档；
  • 代码是否通过测试；
  • 输出是否满足格式；
  • 基于 rubric 的 LM Judge；
  • 另一个 DSPy 程序给出的评分。

Assertion 则可用于表达系统行为约束，例如：

  • 答案必须有引用；
  • Query 不得为空；
  • 生成 SQL 必须能够执行；
  • 输出不能包含不存在的页面。

重要的是：DSPy 不要求为每个中间模块提供标签。通常只需要对最终系统输出进行评分。

13. Optimizer：编译 LM Program

它可能优化：

  • 每个模块的 Instruction；
  • 每个模块的 few-shot demonstrations；
  • 模型权重；
  • 多个模块之间的组合配置。

因此，DSPy 中的“编译”不是传统意义上的机器代码生成，而是：

根据任务数据与评价指标，为声明式 LM 程序寻找更合适的执行参数。

Part V：DSPy 的优化问题

16. 形式化目标

对每个模块 $M_i$，需要确定：

  1. 具体 Prompt 字符串 $\Pi_i$；
  2. 可选的模型权重 $\Theta_i$。

设训练集为：

$$ \mathcal{X} = \left\{ (x_1,m_1),\ldots, (x_{|\mathcal{X}|},m_{|\mathcal{X}|}) \right\}, $$

其中：

  • $x$ 是程序输入；
  • $m$ 是标签、提示或其他评价所需的 metadata；
  • $\mu:\mathcal{Y}\times\mathcal{M}\rightarrow\mathbb{R}$ 是评分函数。

DSPy 希望求解：

$$ \arg\max_{\Theta,\Pi} \frac{1}{|\mathcal{X}|} \sum_{(x,m)\in\mathcal{X}} \mu\left( \Phi_{\Theta,\Pi}(x),m \right). $$

这里优化的是整个程序的最终指标，而不是要求每个中间模块都有监督信号。

17. 为什么这个问题困难

17.1 没有端到端梯度

程序中可能包含：

  • 检索器；
  • 搜索引擎；
  • Python 控制流；
  • 外部 API；
  • 代码执行；
  • Calculator；
  • 非可微分工具。

因此不能简单地像训练普通神经网络那样反向传播。

17.2 没有中间标签

开发者通常只有最终答案标签，却不知道：

  • 第一次 Query 应该是什么；
  • 第二次 Query 应该是什么；
  • 哪条 Passage 最关键；
  • 中间 Reasoning 应该怎样写。

如果每增加一个模块都需要重新人工标注中间数据，模块化开发的优势就会消失。

17.3 Prompt 是巨大离散空间

所有可能字符串构成的搜索空间几乎不可穷举，且不同模块的 Prompt 会相互影响。

17.4 Credit Assignment

当程序最终得分变化时，需要判断：

  • 是 Query Generator 改善了；
  • 是 Answer Generator 改善了；
  • 是两个修改的组合有效；
  • 还是一个局部改进破坏了下游模块的假设。

22. 为什么叫 Bootstrapping

它并不是从人工标注的中间步骤开始，而是：

  1. 用当前不完美的程序运行任务；
  2. 从其中筛出成功案例；
  3. 用成功案例改善程序；
  4. 改进后的程序又可能产生更好的案例。

这是一种“系统从自身成功行为中学习”的过程。

23. Random Search

不同成功 demonstrations 的组合效果可能差异很大。

因此 BootstrapFewShotWithRandomSearch 会：

  1. 生成多个 demonstration candidates；
  2. 为每个模块随机选择不同的示例子集；
  3. 将组合后的程序在 Validation Set 上评估；
  4. 保存得分最高的配置。

注意，单个示例能够带来的效果不是固定的。示例会与：

  • 其他 demonstrations；
  • 模块 Instruction；
  • 上下游模块；
  • 具体 LM

发生交互。

24. Bootstrap Few-Shot 的优缺点

优点

  • 不需要中间人工标签；
  • 实现简单；
  • 在很多任务中收益非常明显；
  • 直接利用整个系统的成功行为；
  • 可与更强 Teacher Model、Ensemble 或后续微调组合。

局限

  • 成功轨迹不代表每个中间步骤都真正正确；
  • demonstrations 的选择非常敏感；
  • 搜索过程可能对小 Validation Set 过拟合；
  • 少数示例不一定能表达复杂的条件规则。

课程实验的一个重要结论是：

在许多任务中，自动构造并选择好的 demonstrations 是最稳定、最关键的优化手段之一，即 “Show, don’t tell”。

Part VIII：OPRO 与 Instruction Optimization

25. OPRO：Optimization through Prompting

OPRO 的思想是让另一个 LM 扮演 Proposer LM：

  1. 给它已有 Instructions 及对应分数；
  2. 让它归纳哪些指令效果更好；
  3. 提议新的 Instruction；
  4. 评估新 Instruction；
  5. 将结果加入历史并继续迭代。

可以将其理解为让 LM 在离散 Prompt 空间中进行迭代搜索。

26. 将 OPRO 扩展到多模块程序

26.1 Coordinate-Ascent OPRO

最直接的方法是坐标上升：

  1. 固定其他模块；
  2. 只优化模块 1；
  3. 选择模块 1 的最佳指令并固定；
  4. 再优化模块 2；
  5. 循环执行。

问题在于：

  • 每评估一个局部候选，都必须运行整个程序；
  • 计算开销高；
  • 贪心地固定局部最优不一定得到全局最优；
  • 模块之间存在强交互。

课件将其复杂度概括为较昂贵的：

$$ O(ND^2M), $$

其中具体变量含义取决于搜索设置，但核心含义是：模块数、候选数和评估成本会快速放大总体开销。

26.2 Module-Level OPRO

另一种做法是放弃显式逐模块 Credit Assignment，直接并行或联合修改多个模块的 Instructions。

优点：

  • 比坐标上升便宜；
  • 能尝试联合改变多个模块。

缺点：

  • Proposer LM 面临更复杂的提案任务；
  • 不容易知道最终提升应归因于哪个模块；
  • 如果缺少上下文，模型可能提出与程序角色不匹配的指令。

Part IX：MIPRO

28. MIPRO 的目标

MIPRO：Multi-prompt Instruction PRoposal Optimizer。

它试图同时解决：

  1. Prompt Proposal：如何提出高质量 Instructions；
  2. Credit Assignment：如何判断跨模块配置中的哪些选择有效；
  3. Joint Optimization：如何联合优化 Instructions 与 Demonstrations。

29. MIPRO 的三个步骤

Step 1：Bootstrap Task Demonstrations

运行当前程序，通过 Rejection Sampling 收集成功轨迹，并构造各模块的 demonstration candidates。

Step 2：Propose Instruction Candidates

使用内部 LM Program 生成多个 Instruction 候选，并利用：

  • Dataset Summary；
  • Program Summary；
  • Bootstrapped Demonstrations；
  • 历史 Instructions 与分数；
  • Proposal Tips

对候选生成过程进行 Grounding。

Step 3：Bayesian Optimization

每个模块都有多个离散选项：

所有组合构成巨大的离散搜索空间，不能全部穷举。

MIPRO 使用 Bayesian Surrogate Model 学习：

然后利用 Acquisition Function 选择更值得尝试的配置。

30. Bayesian Search 的迭代过程

为了节省成本：

  • 多数 Trial 只在小型 Minibatch 上评估；
  • 每隔若干 Trial 在完整 Validation Set 上评估；
  • 最终返回完整验证得分最好的程序配置。

Surrogate Model 随着 Trial 增多逐步学习哪些模块组合更可能取得高分，因此通常比完全随机搜索更高效。

31. MIPRO 与前两种方法的关系

方法 优化对象 Proposal Credit Assignment
Bootstrap Few-Shot Demonstrations 从成功轨迹中构造 通过系统级 Metric 和搜索间接完成
OPRO 扩展 Instructions Proposer LM 坐标上升或弱化显式归因
MIPRO Instructions + Demonstrations Grounded LM Program Bayesian Surrogate Model

MIPRO 不是只让 LM 猜一个“更好的 Prompt”，而是把候选生成与组合搜索分开：

  • LM 擅长提出自然语言候选；
  • Bayesian Optimization 擅长在有限预算下搜索离散组合。

33. LangProBe 中的主要发现

33.1 Instruction Optimization 通常有效

相比 Adapter 自动生成的基础 Signature Prompt，优化 Instructions 经常能够提升结果。

但：

  • 并不存在对所有任务都最好的 Instruction Proposal 方法；
  • 在数据很少时可能过拟合；
  • 只优化 Instructions 的收益并不总是稳定。

33.2 Bootstrapped Demonstrations 非常关键

优化 demonstrations 往往带来更明显、更稳定的提升。

不同 demonstration sets 即使来自相同的 Bootstrapping 流程，效果也可能差异巨大：

  • 某些组合低于 Zero-shot；
  • 某些组合显著超过基础系统。

为什么不同 demonstrations 会产生如此大的差异，仍是值得研究的问题。

Part XII：课程总结

37. Natural Language Programming

  1. 程序化系统往往比单独使用模型更准确、更可控、更透明，有时也更高效。
  2. 开发者应该描述声明式程序，而不是过早绑定具体实现细节。
  3. 高层 Optimizer 可以自动构造 Prompt、Demonstrations 或模型权重。
  4. 系统开发的主要精力应放在：
    • 模块是否合理；
    • 模块如何连接；
    • Metric 是否真正对应目标；
    • 程序架构是否适合任务。

38. Natural Language Optimization

  1. 在许多任务上，自动 Bootstrap 高质量 Demonstrations 非常重要：

    Show, don’t tell.

  2. 在 Demonstrations 基础上生成更好的 Instructions 是可行的；
  3. 当任务包含大量条件规则时，Instructions 尤其重要；
  4. 高质量 Instruction Proposal 需要充分 Grounding；
  5. 多模块优化需要明确、有效的 Credit Assignment；
  6. 当前尚不存在适用于所有任务的最佳优化算法。

39. 开放研究愿景

课程最后强调，AI 进步不应只依赖：

  • 更大但更不透明的单体模型；
  • 针对某个版本模型的临时 Prompt 技巧；
  • 很快失效的 ad-hoc synthetic data 方法。

另一条开放研究路线是：

  1. 设计边界明确的 Compound AI Programs；
  2. 研究可复用的 inference-time strategies；
  3. 研究通用 Predictor 和 Adapter；
  4. 研究新的自然语言程序 Optimizer；
  5. 通过模块化贡献，让不同研究成果可以组合。

这种模块化研究方式类似于神经网络发展过程中，Attention、Transformer、Convolution 等组件由不同研究者分别提出，最后共同推动整个领域前进。
