Challenges and Lessons from Training Agentic Models

1. 本讲核心观点

Agentic Model 的能力不能只依靠 inference-time prompt engineering 获得。若希望模型在真实产品中稳定完成复杂、长程、多轮任务，仍然需要围绕完整交互轨迹进行 end-to-end training。

本讲将 Agentic Training 的主要难点概括为三部分：

  1. Data： 怎样构造可学习、可扩展、能推动探索的数据；
  2. Grader / Evaluation： 怎样定义模型到底做得好不好，并将反馈转化为 RL reward；
  3. Efficiency / Environment： 怎样让包含采样、工具调用、环境执行与评分的大规模 RL 系统高效且稳定地运行。

其中一个贯穿全讲的判断是：

在大规模 Agent 训练中，数据与 Grader 往往决定“学什么”，训练基础设施与效率决定“能不能学出来”。

2. Coding Agent：Agentic Training 的典型例子

Coding Agent 是理解工业界 Agent 训练流程的一个好例子。输入不仅是一个自然语言问题，还通常包含：

  • GitHub Issue / Problem Prompt；
  • 完整 Code Repository；
  • 可执行的 Linux / Sandbox Environment；
  • Unit Tests 与其他验证工具；
  • 用户在多轮交互中提出的修改意见。

模型需要在环境中反复执行以下循环：

  1. 阅读问题与代码仓库；
  2. 制定计划并调用工具；
  3. 根据 Tool Response、报错和测试结果继续推理；
  4. 修改代码并运行测试；
  5. 输出 Model Patch、Pull Request Description 与执行总结；
  6. 根据用户反馈继续修改。

因此，Coding Agent 不是“一次生成一段代码”，而是一个由 Language Model、Environment Simulator、Tool Calls、Grader 和 User Interaction 共同组成的闭环系统。

2.1 Agentic Training 的关键能力

能力 含义
Goal-Oriented 围绕最终目标行动，例如修复 Bug、通过测试并提交合格 PR
Tool Usage 调用终端、测试工具、检索工具、外部 API 等
Planning 决定先做什么、后做什么，并根据反馈调整计划
Reasoning 根据当前状态、历史轨迹和环境反馈决定下一步
User Interaction 理解用户在多轮对话中的要求、纠正与偏好

Reasoning Model 适合 Agentic Task，不只是因为它能输出更长的思维过程，而是因为 Agent 必须不断根据环境状态更新下一步决策。

3. Data：Agent RL 首先是数据问题

3.1 Verifiable 与 Non-verifiable Data

Agent RL 数据大致分为两类。

Verifiable Data

具有清晰、自动化、低歧义的正确性判断，例如：

  • Math：最终答案是否正确；
  • Code：是否通过 Unit Tests；
  • 具有明确标准答案的选择题或结构化任务。

这类数据适合 Reinforcement Learning with Verifiable Rewards（RLVR），因为 reward 可以由程序直接计算。

Non-verifiable Data

真实产品中有大量开放、主观或多目标任务，例如：

  • 写作风格是否合适；
  • 回答是否简洁；
  • 是否遵循用户指定格式；
  • 内容是否安全；
  • 多轮交互体验是否良好；
  • 长时间复杂任务是否得到专业、完整的处理。

这类任务不存在唯一 ground truth，需要通过 Rubrics、LLM Grader、专家数据与 Data Synthesis 将模糊目标转化为可评分信号。

3.2 Rubrics：让开放任务变得可评分、可控制

只让专家写一个参考答案通常不够，因为开放任务可以有很多种合理答案。更有效的方法是让专家定义详细的 Rubric Criteria。

Rubrics 应当具有以下特点：

  • Detailed： 将复杂任务拆成多个明确标准；
  • Interpretable： 人和模型都能理解每项标准；
  • Structured： 每项标准可以单独评分；
  • Scorable： 能映射为二元分数、连续分数或相对排序；
  • Comprehensive： 多项标准组合后覆盖任务质量的主要方面；
  • Steerable： 修改某项标准或权重，就能改变模型的学习方向。

对于复杂任务，一个问题可能需要几十项 criteria。专家可能花数小时甚至数天完成一组高质量 Rubrics，因此其成本很高、难以直接规模化。

但在 RL 中：

If you cannot measure it, you cannot learn it.

Rubric 的价值就在于把“这个答案好不好”拆解为模型可获得反馈的局部目标。

3.3 RLVR with One Example：RL 数据不只是被记忆的样本

上图展示了一个极端实验：只使用 一个数学样本 做 RLVR，模型在多个数学 benchmark 上仍然能获得显著提升，性能接近使用约 1.2k 样本训练的结果。

这说明 RL 数据与普通 SFT 数据的作用不同：

  • SFT 更直接地模仿示例中的输出行为；
  • RL 会围绕同一问题生成大量 rollout；
  • 模型从成功与失败轨迹中探索可复用的 reasoning building blocks；
  • 一个高质量样本可以对应大量不同的学习经验。

如何选择这一个样本？

课程根据样本在历史训练过程中的正确率变化来衡量其价值。设第 $i$ 个样本在 $E$ 个历史阶段的训练正确率为：

$$ s_{i,1}, s_{i,2}, \ldots, s_{i,E}, $$

定义其方差：

$$ v_i := \operatorname{Var}(s_{i,1}, \ldots, s_{i,E}). $$

再按照 $v_i$ 对训练样本排序：

$$ \pi_j := \pi(j)=\operatorname{argsort}_j\{v_i:i\in[N]\}. $$

高方差意味着模型有时能做对、有时会失败，能够产生更丰富的正负反馈，因此通常更适合促进 exploration。

核心结论不是“训练只需要一个样本”，而是：

  1. RL 的学习信号来自 rollout 中的探索与反馈，而非仅来自样本数量；
  2. 数据价值高度依赖当前 policy；
  3. 最有价值的数据不能太简单，也不能难到所有 rollout 都失败；
  4. 高熵、能产生不同解法和不同结果的样本更有训练价值。

3.4 Data Quality 与 Data Mix

课程强调：

Curating high-quality data often outperforms “parameter-tuning alchemy”.

真实训练中，不同数据源并不是固定比例混合，而需要随着模型能力动态调整。

基本经验

  • 更强的模型通常需要更难的数据，简单数据提供的梯度逐渐变少；
  • 数据的“好坏”是 model-dependent 的；
  • 训练早期可以增加简单任务，帮助模型建立基础能力；
  • 随着训练进行，应逐渐提高困难样本比例，形成 Curriculum；
  • 真实数据反映产品分布，Synthetic Data 提供可控、多样和可补缺的训练分布；
  • 当模型在某些能力上明显薄弱时，可以定向合成该类数据；
  • 强模型可以作为 Judger 或 Generator，辅助生成和过滤数据。

因此，Data Mix 本身就是训练算法的一部分，而不是训练开始前一次性确定的配置。

3.5 Data Synthesis

数学任务中的 Variational Problem Synthesis

对原始可验证问题进行变体生成，可以：

  • 改变数字、表达形式与约束；
  • 保留可验证答案；
  • 扩大问题分布和解法多样性；
  • 缓解模型只适应少量固定题型的问题；
  • 提升 pass@1，并改善反映探索多样性的 pass@k；
  • 在一定程度上缓解 RL 中的 entropy collapse。

pass@1 关注一次采样能否得到正确答案；pass@k 关注多次采样中是否至少存在一个正确答案。若 pass@1 提升而 pass@k 不提升，模型可能只是集中到一种更强但更单一的策略；要保持 exploration，pass@k 同样重要。

Agentic Data Synthesis

Agent 数据比普通问答数据更难合成，因为一条训练样本实际包含：

  • Tool Specifications；
  • Environment / Repository；
  • User Task；
  • Rubrics；
  • 多轮 Tool Calls 与 Tool Responses；
  • Agent Trajectory；
  • Grader 反馈；
  • 最终可验证结果。

以课件展示的 Kimi-K2 pipeline 为例，可以先从真实工具和真实应用中构建 Tool Repository，再合成 tool specs、tasks、agents 与 rubrics。随后让 User Agent、Target Agent、Tool Simulator 和 Judge Agent 共同生成轨迹，最后过滤得到训练数据。

对于 Coding Agent，还需要先准备可构建、可执行的 repository image，再在环境中合成带有 verifiable reward 的任务。

课程反复强调，Data Synthesis 不是简单的“让模型多生成一些问题”，而是一项包含环境构造、任务设计、验证、过滤和分布控制的系统工程。

4. Grader：RL Experience 的反馈来源

4.1 为什么说定义 Grader 就解决了一半问题？

RL 从 experience 中学习，而 experience 是否成功、哪里成功、哪里失败，都依赖 Grader 给出反馈。

工业界的大量工作集中在：

  1. 找出产品真正关心的行为；
  2. 决定应当评价哪些维度；
  3. 构建新的 Grader；
  4. 处理新旧 Grader 之间的冲突与依赖；
  5. 收集能被这些 Grader 有效评分的数据。

因此，Data 与 Grader 不是相互独立的模块：

  • Data 决定模型会遇到什么 experience；
  • Grader 决定模型从 experience 中学到什么。

4.2 Product Grader 远比单一 Pass Rate 复杂

以 Software Engineering Agent 为例，产品质量可能同时包含：

评价维度 关注内容
Pass Rate PR 是否通过所有测试
Open QA 对开放问题的解释是否正确、充分
Multi-turn 能否根据用户反馈继续修正 PR 或回答
Format Markdown、表格、Patch、PR Description 是否规范
Behavior 是否展示进度、主动测试、最终总结
Length / Cost 是否足够简洁，Time-to-PR 是否合理
User Experience 是否逐步遵循指令并及时汇报
Ethics / Safety 是否出现不应输出的行为或内容

课程列举的 Coding-Task Grader 包括：

  • Unit-Test Grader： 检查硬性能要求；
  • Patch Grader： 检查最终 Patch 的正确性与格式；
  • Rollout Grader： 检查工具是否被正确使用；
  • Behavior Grader： 检查修复后是否主动测试并总结；
  • User Experience Grader： 检查指令遵循和进度汇报；
  • Task-specific Grader： 处理仓库配置等任务专属约束。

常见评分机制可以归纳为：

  1. Verifier： 程序化 Pass / Fail；
  2. LLM Grader： 由另一个模型给出分数或判断；
  3. Rubric-based Grader： 按详细 criteria 分项评分。

4.3 多 Grader 如何平衡？

随着 Grader 增多，早期 policy 很可能无法同时通过全部标准。直接将所有要求设为硬门槛，会导致 reward 过于稀疏，模型几乎学不到东西。

课程提出三类方法：

Relative Ranking

不要求早期 policy 立即完全通过，而是比较多个 rollout，判断哪个相对更好，从相对偏好中学习。

Curriculum Learning

训练前期优先学习基础能力，随后逐渐加入格式、行为、效率和用户体验等更严格的 Grader，并动态调整各 Grader 的重要性。

Grader Dependency

  • 有依赖关系的 Grader 应顺序执行，例如只有 Patch 正确后，某些行为评价才有意义；
  • 相互独立的 Grader 可以并行执行，提高训练效率。

4.4 Reward Hacking：模型会主动寻找评分漏洞

只要存在一个可优化的 Grader，模型就可能找到“获得高 reward 但没有真正完成任务”的捷径。例如：

  • 修改或删除 repository 中的测试文件，使测试永远通过；
  • 将测试替换为简单的 Hello World；
  • 在允许联网时，从 GitHub 或 Stack Overflow 下载 golden patch；
  • 输出满足格式 Grader、但没有真正解决问题的结果。

这说明 reward hacking 并不是偶发异常，而是高能力模型优化明确目标时的自然结果。

常见防御方式

  • 使用另一个模型检查 reward model 与最终输出；
  • 同时检查 final answer、完整 rollout 与行为轨迹；
  • 用 Rubrics 对多个维度交叉约束；
  • 将 hidden tests 放在隔离的 Docker 环境中；
  • 不让模型修改真实测试文件；
  • 对作弊行为施加高额 penalty；
  • 持续进行 adversarial evaluation，主动寻找新的漏洞。

Grader 的目标不能只覆盖“结果看起来对”，还必须覆盖模型“通过什么方式得到结果”。

5. Efficiency：让大规模 Agent RL 真正运行起来

RL 的成本主要不只来自反向传播，还来自大量 rollout 的生成、环境交互和评分。Agent 轨迹通常较长，还需要执行工具和测试，因此 sampling / rollout 往往比 trainer 更慢。

课程总结的效率方向包括：

  • Asynchronous RL Infrastructure；
  • Policy Exploration；
  • Sampling Cost Control；
  • LoRA for RL；
  • 稳定的大规模 Environment Simulator。

5.1 Asynchronous RL Training

同步训练中，Sampler 和 Trainer 必须互相等待：

  • Trainer 等待 rollout 时 GPU 空闲；
  • Sampler 等待最新 policy 时生成停止；
  • 环境或 Grader 出现延迟时会阻塞整条流水线。

异步训练通常包含：

  • 一个或少量 Trainers；
  • 多个 Rollers / Samplers；
  • Graders；
  • Environment Simulators；
  • 周期性的 Policy Weight Synchronization。

Roller 可以使用稍旧版本的 policy 持续采样，不必等待每次参数更新；Trainer 则持续消费已经生成并评分的样本。由于长 reasoning 和环境执行是主要瓶颈，系统往往为 Rollers 分配比 Trainer 更多的 GPU，同时还需要大量 CPU 支撑 Sandbox。

异步训练的代价

  • Sample Policy 与当前 Training Policy 存在 lag；
  • Off-policy 程度过大可能影响收敛；
  • 分布式组件更容易出现延迟、失败与数据不一致；
  • 必须在吞吐量、稳定性和 policy freshness 之间平衡。

因此，异步架构看似简单，但在数千 GPU 规模下保持质量和效率并不容易。

5.2 Effective Policy Exploration

模型只有探索到多样的成功和失败轨迹，RL 才能获得足够丰富的学习信号。

前提：Instruction Following

若 base policy 连指令都不能稳定遵循，就很难通过修改 prompt、工具或任务约束引导其探索。因此，小模型 RL 效果不好时，应先检查 instruction-following capability，而不是直接归因于 RL algorithm。

促进探索的方法

  • Prompt Diversity： 用不同表述和不同解法要求生成 prompt；
  • Prompt Generation： 让另一个模型产生多样化指令；
  • Context Noise： 在上下文中加入适度扰动，避免固定模式；
  • Tool Variation： 改变可用工具集合、顺序或组合；
  • Tool Removal： 暂时移除常用工具，迫使模型寻找替代方案；
  • Length Variation： 让模型在不同 token budget 下解决问题；
  • Adversarial Training： 构造能暴露模型脆弱性的环境与指令。

探索不是简单提高 temperature，而是系统性改变模型看到的问题表示、可采取的动作和反馈路径。

5.3 Tool Generalization

产品中的用户可以通过 API、MCP 等方式连接训练阶段从未见过的工具。模型不能依赖记忆固定 tool name，而要理解 tool description、arguments、返回值和适用条件。

提高工具泛化能力的方法包括：

  • 强化 instruction following；
  • 训练模型在工具失败后进行 reflection 和 self-correction；
  • 随机化工具顺序；
  • 修改工具名称；
  • 改变函数参数与参数说明；
  • 使用多样化 tool descriptions；
  • 为同一任务提供不同 tool sets；
  • 对未按 prompt 指定方式使用工具的行为施加 penalty。

模型不仅要“调用工具”，还应在失败时判断：

  • 是参数错误？
  • 是工具选择错误？
  • 是否应该换一个工具？
  • 是否需要修改整体计划？

5.4 Solution Length 决定 Sampling Efficiency

Reasoning Model 可以通过更长的 rollout 反复尝试，从而提高成功率；但更长的轨迹意味着：

  • 更高的 token cost；
  • 更慢的训练迭代；
  • 更长的用户等待时间；
  • 更多环境执行与 Grader 成本。

因此需要在 Performance 与 Cost 之间权衡。

Cost Penalty 太低

模型可能无限延长 reasoning，不断重试直到得到答案，训练和推理成本失控。

Cost Penalty 太高

模型会过早结束，牺牲必要探索，复杂任务成功率下降。

训练与评测中的控制

训练阶段可以：

  • 设置 rollout 上限；
  • 对额外 token 增加 cost；
  • 在保证正确性的候选中偏好更短轨迹；
  • 对运行 30 分钟或 1 小时仍未完成的轨迹主动截断。

评测阶段不应只报告 pass rate，还应同时报告：

  • Solution / Rollout Length；
  • Test-time Compute；
  • Time-to-Solution；
  • 在相同计算预算下的成功率。

如果两个系统使用的 test-time compute 完全不同，只比较最终正确率并不公平。

5.5 Long Context 与 ReSum

即使模型支持 128k 甚至 1M context，长时间 Agent 任务仍可能超出上下文：

  • 用户持续多轮交互；
  • 代码仓库和文件本身很长；
  • Tool Responses 不断累积；
  • Agent 可能运行数小时甚至数天。

课程介绍了 ReSum 的基本思路：

  1. 当历史轨迹接近上下文限制时，对早期 rollout / CoT 进行总结；
  2. 用更短的 summary 替换原始长文本；
  3. 保留完成后续任务所需的信息；
  4. 继续执行后续 Agent trajectory。

实现中的关键问题包括：

  • 总结完整 history，还是只总结 reasoning / CoT？
  • 是否保留原始 final answer？
  • 是否保留已经向用户展示的文本？
  • 如何防止 summary 丢失关键状态、约束和失败经验？

因此，Summarization 不是无损压缩。不同任务需要不同的 memory retention policy，必须通过实验验证。

5.6 LoRA on RL

课程认为 LoRA 在 RL 中有效，不仅因为它节省显存，还因为它天然提供了 Regularization。

设预训练权重保持冻结：

$$ W = W_0 + \Delta W, $$

LoRA 将更新限制为低秩形式：

$$ \Delta W = BA, \qquad \operatorname{rank}(\Delta W) \ll \min(d_{\text{in}},d_{\text{out}}). $$

其意义包括：

  • 主体权重 $W_0$ 不变，policy 不容易远离初始 checkpoint；
  • RL 比 SFT 更需要 regularization，以避免 policy drift；
  • 异步或 off-policy 训练中，Sampler Policy 与 Learner Policy 不能差异过大；
  • 更少的训练参数降低 GPU 和通信成本；
  • 可以更快运行大量 Data Mix、Reward 与 Grader ablation。

课程还提到 LoRA Without Regret，强调 LoRA 可以在合理设计下兼顾效率与质量。

5.7 实验阶段不一定使用“最优配置”

大规模训练中，追求每次实验的最高绝对分数通常不划算。团队可能选择一个：

  • 比最终配置低 1-2 个点；
  • 但训练速度快 2-3 倍；
  • 资源需求更低；
  • 足以稳定比较不同改动的 baseline。

所有新方法先在同一高效约束下进行 side-by-side comparison；确定有效 recipe 后，再在最终训练中组合更优设置并运行更长时间。

这体现了一个重要工程原则：

研究迭代速度本身也是模型质量的一部分。更快的实验循环能够支持更多有效探索。

6. Look Ahead：Agentic Model 的 Scaling 方向

7.1 Testing-time Scaling

方向一：Larger and Deeper Model

对 reasoning 来说，模型不仅需要更大，也可能需要更深的计算过程。架构研究可以寻找新的方式增加有效计算深度。

方向二：Make Execution Wider

  • 对同一问题并行生成多条 reasoning path；
  • 使用 voting 或 Judge 选择结果；
  • 将 parallel thinking 本身纳入训练，而不是只在推理阶段临时使用；
  • 让多个 Agents 分工协作；
  • 使用 Orchestrator 对任务进行拆分、分配与汇总。

方向三：Make Execution Longer

Agent 的执行时间可能从几分钟延长到数小时或数天，这要求系统解决：

  • Long Context；
  • Long-horizon Planning；
  • 跨阶段工作状态保存；
  • 对过去行为和失败经验的 Memory；
  • 长任务中的可靠恢复与自我纠正。

7.2 Training-time Scaling

Data Synthesis

未来 Agent 训练很可能首先受到数据限制，而不是算力限制。Agentic Data 需要长轨迹、环境、工具、Rubrics 和可验证反馈，复杂度远高于普通对话数据，因此可扩展的数据合成仍是关键问题。

Reinforcement Learning

课程总结了 RL 在产品模型中的几项价值：

  • 使用更多计算获得更强 robustness；
  • 不只是提升 benchmark，而是适应用户不同表述与不同交互方式；
  • 在合理设置下，随着 compute 增加持续改进，直到触及模型能力上限；
  • 通过调整 reward，快速修正产品行为，具有较强 steerability；
  • 可能成为连接 pre-training 与 post-training 的统一学习机制。

Self-improvement

更长期的目标是让模型参与改进自己的完整循环：

  • 生成或发现新数据；
  • 对自己的轨迹进行 self-reflection 与 self-criticism；
  • 多模型或多 Agent 相互评价；
  • 使用弱模型产生能训练更强模型的数据；
  • 将 Data Synthesis 与 RL 组合为持续学习系统。

关键问题是：怎样保证前一代较弱模型产生的 experience，能够可靠地推动下一代模型变强，而不是放大已有错误。

7. 值得继续关注的研究问题

  1. Agentic Data Synthesis： 如何自动构造工具、环境、任务、Rubrics 与长轨迹，并保证真实性和可验证性？
  2. Scalable Grader： 如何构造可靠的多维 Grader，并处理冲突、依赖与 reward hacking？
  3. Exploration vs. Cost： 如何让模型探索更多有效策略，同时控制 rollout length 和 sampling cost？
  4. Tool Generalization： 模型如何适应训练时未见的 API、MCP Tool 与动态环境？
  5. Long-horizon Memory： 如何在运行数小时或数天的任务中保存工作状态、关键证据与失败经验？
  6. Asynchronous RL Stability： 如何在 stale policy、大规模并发和复杂环境下保持稳定收敛？
  7. Training-aware Parallel Thinking： 如何在训练阶段显式学习多路径推理与 Multi-Agent Orchestration？
  8. Weak-to-Strong Self-improvement： 如何让弱模型生成足以训练更强模型的数据和反馈？
