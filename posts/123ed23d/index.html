LLM Agent Evaluations & Project Overview

0. 本节课要点

  1. 理解 Agent Evaluation 的基本分类
  2. 理解什么是“好的评测”
  3. 把 Benchmark 落地为 Green Agent

1. 为什么需要 Evaluation？

1.1 Evaluation 的定义

Evaluation 是对模型或 Agent 进行系统化、可重复测量的过程。它通过统一的 Benchmark、环境和指标，帮助我们回答：

  • 当前模型具备什么能力？
  • 相比其他模型是否有真实提升？
  • 模型在哪些场景中容易失败？
  • 是否能够安全、可靠地部署到真实环境？

评测的价值不仅在于给模型排名，还包括：

  • 用可复现证据衡量领域进展；
  • 识别模型能力边界和潜在风险；
  • 支持模型选择、部署与论文比较；
  • 推动整个研究领域持续进步。

1.2 Evaluation 贯穿模型开发全过程

不同开发阶段对评测系统的要求并不完全相同：

阶段 评测的主要作用 更关注的性质
Train 提供可优化的训练目标与 Validation Signal 极快、便宜、最好可微，不能存在明显 shortcut
Develop 快速比较设计和定位问题 快、便宜，同时避免被模型轻易投机
Model Selection 从多个候选模型中选择更好的版本 快、成本可控、区分度足够
Deploy 判断模型能否在真实任务中可靠运行 Trustworthy、task-specific、能反映绝对能力
Publish 让外部研究者能够公平比较与复现 标准化、可复现、覆盖广、易于使用

1.3 Benchmark 为什么能推动研究进步？

Benchmark 为不同模型提供了共同参照系。没有稳定评测时，模型“变好”往往只是主观印象；有了统一 Benchmark 后，研究者才能：

  • 在相同任务上公平比较；
  • 识别当前模型仍未解决的问题；
  • 判断某种训练或 Agent 架构是否带来真实提升；
  • 持续追踪领域能力边界。

但 Benchmark 也会反过来塑造研究方向：大家会优先优化能够被现有指标测量的能力。因此，Benchmark 的设计决定了我们奖励什么，也在一定程度上决定了领域向哪里发展。

2. 从 LLM Evaluation 到 LLM Agent Evaluation

传统 LLM 通常可以近似看作静态的 text-to-text 系统：输入文本，输出文本。

Agent 则在 LLM 之外加入了更多组件和交互过程：

  • Planning
  • Tool Use / Function Calling
  • Memory
  • Multi-step Reasoning
  • Environment Interaction
  • 多轮观察、行动与反馈

因此，Agent Evaluation 不只需要判断最终回答文本，还可能需要检查：

  • 是否调用了正确工具；
  • 是否使用了正确参数；
  • 环境状态是否被正确修改；
  • 多轮交互是否遵守规则；
  • 任务是否稳定完成，而非偶然成功；
  • 成本、步数、延迟和资源使用是否合理。

这使得 Agent Evaluation 比普通 LLM Evaluation 更接近一个完整的实验系统或软件测试系统。

3. Evaluation 的基本类型

3.1 Close-Ended vs. Open-Ended

3.1.1 Close-Ended Evaluation

封闭式任务的候选输出和正确答案数量有限，例如：

  • Multiple Choice
  • Yes / No
  • Sentiment Analysis
  • Natural Language Inference
  • Named Entity Recognition
  • Part-of-Speech Tagging

这类任务容易自动评测，常见指标包括：

$$ \text{Accuracy}=\frac{\text{正确预测数量}}{\text{样本总数}} $$

$$ \text{Precision}=\frac{TP}{TP+FP},\qquad \text{Recall}=\frac{TP}{TP+FN} $$

$$ F_1=\frac{2\cdot \text{Precision}\cdot \text{Recall}} {\text{Precision}+\text{Recall}} $$

优点：自动、快速、便宜、可复现。
局限：只能覆盖答案空间较小的任务，无法表达许多真实世界的复杂目标。

3.1.2 Open-Ended Evaluation

开放式任务允许模型生成自由形式输出，正确答案可能很多，也不一定只有“对或错”两档。

典型任务包括：

  • Summarization
  • Translation
  • Storytelling
  • Writing Style Adaptation
  • Instruction Following
  • 复杂知识工作与文档生成

对于开放式输出，更合理的问题通常是:

“答案在正确性、完整性、风格、逻辑和实用性等维度上有多好？”

因此，需要进一步区分 Verifiable 和 Non-Verifiable 任务。

3.2 Verifiable vs. Non-Verifiable

3.2.1 Verifiable Tasks

可验证任务具有明确的 Oracle 或客观成功条件。

例如：

  • 数学证明可以由 Proof Checker 检查；
  • 代码生成可以运行 Unit Tests；
  • 数据库操作可以比较最终 DB State；
  • 网页或应用操作可以检查目标状态是否达成。

此时评测的核心是实现一个忠实遵循成功条件的 Evaluator：

$$ \text{Score}=\mathbb{I}[\text{Outcome satisfies success criteria}] $$

这类任务虽然输出形式可能开放，但最终结果能够被程序检查，因此通常更适合构建稳定 Benchmark。

3.2.2 Non-Verifiable Tasks

不可验证任务缺少唯一、客观的 Ground Truth，例如：

  • 故事是否精彩；
  • 写作风格是否自然；
  • 回答是否足够有帮助；
  • 文档是否专业、美观且完整。

这些任务往往需要：

  1. Human Evaluation
  2. LLM-as-a-Judge

3.3 Human Evaluation

Human Evaluation 可以对输出整体评分，也可以按照多个 Rubric 分别评分;

eg:测评文本生成的质量

  • Fluency
  • Coherence / Consistency
  • Factuality / Correctness
  • Commonsense
  • Style / Formality
  • Grammaticality
  • Redundancy

人类评测通常被视为 Gold Standard，但存在明显问题：

  • 慢且昂贵；
  • 不同标注者之间存在 Inter-Annotator Disagreement；
  • 同一标注者在不同时间也可能产生 Intra-Annotator Disagreement；
  • 难以完全复现；
  • 主观偏好可能掩盖真实任务目标。

因此，Human Evaluation 更适合作为高质量抽查、校准或最终验证，而不一定适合作为所有样本的日常自动评测。

3.4 LLM-as-a-Judge

LLM-as-a-Judge 使用另一个 LLM 对候选输出进行打分、比较或排序。

优点

  • 比人工评测便宜、快速；
  • 能理解语义等价、内容完整性和风格差异；
  • 高能力模型的判断通常与人类偏好具有较强一致性；
  • 适合大规模开放式评测。

主要问题与改进

问题 改进方式
Judge 不够可靠 使用小规模 Human Eval 进行 Cross-check
输出具有随机性 同一 Judge 重复多次，取平均分或多数票
不同 LLM 有不同偏差 使用多个不同 Judge Model 进行 Ensemble / Majority Vote
单一分数难解释 使用连续分数，并按多个维度设置 Rubric
对 Prompt 敏感 给出详细评分标准、正反例和明确输出格式
容易受到 Prompt Injection 隔离待评文本与系统指令，设计 Adversarial Tests
可能出现 Reward Hacking 验证评分与真实任务成功的一致性

但增加 Judge 数量并不能自动解决根本问题。首先仍需确认：Judge 给出的高分是否真的意味着任务完成得好。

3.5 Static vs. Dynamic Benchmark

类型 定义 优点 局限
Static Benchmark 测试样本和指标固定 易复现、易比较、适合论文与排行榜 容易数据污染、过拟合和被针对性优化
Dynamic Benchmark 样本持续更新或周期性重新生成 更贴近真实数据分布，更难记忆和投机 不同时期结果不完全可直接比较，维护成本更高

动态 Benchmark 特别适合容易快速变化的领域，例如实时编码问题、网页环境或持续更新的知识任务。

4. LLM Agent Evaluation 的任务分类

Agent Benchmark 可以按“评测对象”分成三类。

4.1 评测某一种 Agent Capability

关注 Agent 的单项能力，不限定具体应用领域：

能力 代表性 Benchmark
Planning / Multi-step Reasoning GSM8K、MATH、MINT、PlanBench、Natural Plan 等
Function Calling / Tool Use BFCL、ToolBench、APIBench、API-Bank、ToolSandbox 等
Self-Reflection LLF-Bench、LLM-Evolve、Reflection-Bench
Memory NarrativeQA、MemGPT、LoCoMo、A-MEM、Long-term Memory Benchmark 等

4.2 评测某一种具体应用

关注 Agent 在特定真实任务中的表现：

能力 代表性 Benchmark
Web Agent WebArena、VisualWebArena、WorkArena 等
Software Engineering Agent SWE-bench、SWE-bench Verified 等
Scientific Agent ScienceWorld、ScienceAgentBench、CORE-Bench、MLGym-Bench 等
Safety / Cybersecurity RedCode、CyberGym、BountyBench 等
Legal LegalAgentBench 等
Healthcare MedAgentBench、HealthBench 等
Finance Finance Agent Benchmark 等

4.3 Generalist Agent Evaluation

评测 Agent 跨多个应用领域的综合能力，例如：

  • GAIA
  • AgentBench
  • OSWorld
  • AppWorld
  • OmniACT
  • TheAgentCompany

这类 Benchmark 更接近 Generalist Agent 的综合能力测试，但课程项目通常更适合选择边界更清晰的具体能力或具体应用。

5. 什么是一个好的 Evaluation System？

5.1 Task Validity 与 Outcome Validity

Task Validity

任务成功是否真的说明 Agent 具备目标能力？

例如，要评测“长程规划能力”，任务不能仅靠一个显而易见的单步操作完成，否则即使成功，也不能证明 Agent 具有长程规划能力。

Task Validity 主要取决于：

  • 任务设计；
  • 环境设计；
  • 工具和信息可见性；
  • 难度与能力要求是否匹配。

Outcome Validity

评测结果是否忠实反映任务是否成功？

例如，Agent 已正确完成数据库修改，但 Evaluator 只做字符串匹配，最终判为失败，这就是 Outcome Validity 不足。

对于 Green Agent，Outcome Validity 尤其关键，因为它直接决定 Green Agent 输出的分数是否可信。

5.2 文本结果的 Outcome Validity

String / Substring Matching

使用字符串匹配时，需要处理：

  • 与 Ground Truth 语义相同但表述不同的答案；
  • 不影响正确性的冗余文本；
  • 否定词和双重否定；
  • 大小写、空格、格式和标点差异；
  • Agent 在答案中枚举所有可能选项以“碰中”答案；
  • Ground Truth 过于简单，导致随机猜测即可得分。

因此，Exact Match 适合输出格式高度规范的任务；若答案具有多种语义等价表达，则需要更灵活的规则或 LLM Judge。

LLM-as-a-Judge

需要验证：

  • Judge 与人类判断是否一致；
  • Judge 多次运行是否稳定；
  • Judge 是否容易被待评文本中的恶意指令操控；
  • Judge 是否会奖励冗长、讨好式表达，而不是真正正确的答案；
  • Rubric 是否清晰覆盖任务真正关注的维度。

5.3 Code Generation 的 Outcome Validity

Unit Test / End-to-End Test

  • 测试本身必须正确；
  • 覆盖正常输入、边界条件与错误输入；
  • 不只检查示例用例；
  • 测试结果应尽量客观；
  • 可补充 Code Coverage、复杂度或安全检查；
  • End-to-End Test 应覆盖真正相关的完整执行路径；
  • 避免 Flaky Test 和环境随机性导致误判。

Fuzz Testing

Fuzz Testing 应覆盖：

  • 不同数据类型；
  • 极端数值范围；
  • 特殊字符和异常格式；
  • 大小规模输入；
  • 程序最敏感、最可能出错的输入区域。

评测代码生成时，测试不足会带来 False Positive，测试或环境错误则可能造成 False Negative。

5.4 Environment State Change 的 Outcome Validity

许多 Agent 任务的真正结果不是一段文字，而是环境状态发生变化，例如：

  • 数据库中创建了正确订单；
  • Calendar 中添加了正确会议；
  • 文件系统中生成了目标文件；
  • 网页应用完成了指定操作。

State Matching 需要注意：

  • Ground Truth 应包含所有合法成功状态，而不是只允许唯一实现路径；
  • 只检查与任务有关的状态字段；
  • 同时检查不应被修改的状态，防止 Agent 破坏其他数据；
  • 避免 Agent 通过无意义状态修改骗取分数；
  • 对多步骤任务，可以同时检查最终状态和关键中间状态。

5.5 Multi-Step Reasoning 的 Outcome Validity

多步推理评测需要避免：

  • 输出格式模糊，导致解析器误判；
  • 答案空间过小，随机猜测也容易成功；
  • 只奖励最终答案，而完全忽略过程中的关键约束；
  • 设计可被简单 shortcut 或 reward hacking 利用的指标。

最终答案正确并不一定说明推理过程可靠；但强制评测完整 Chain-of-Thought 也可能带来新的隐私、可解释性与标准化问题。因此，应根据任务选择：

  • Final Answer Matching；
  • Intermediate State / Tool Trace Checking；
  • Constraint Satisfaction；
  • Progress Metric；
  • 多次运行的一致性。

6. Evaluation 常见失败方式

6.1 Data is noisy or biased

  • 样本标注错误；
  • 任务描述含糊；
  • 不同群体或场景覆盖不均；
  • 难度分布不合理；
  • 测试数据与目标真实场景不一致。

6.2 Not Practical

Benchmark 可能在形式上很复杂，却只测试 Toy Problem。好的评测应首先回答：

这个能力是否对应实际使用者的真实需要？

6.3 Shortcut - Eval can be gamed

Agent 可能利用评测漏洞获得高分，而没有真正完成任务，例如：

  • 在输出中列出全部候选答案；
  • 使用固定关键词骗过 Substring Matching；
  • 对 LLM Judge 进行 Prompt Injection；
  • 只修改 Evaluator 检查的字段；
  • 利用测试环境泄露 Ground Truth。

6.4 Not challenging enough

如果大部分模型很快达到接近 100%，Benchmark 将失去区分能力。

好的 Benchmark 应：

  • 包含多个难度层级；
  • 能区分当前系统与未来更强系统；
  • 不容易被少量模板或专门调参攻破；
  • 保留足够 Hard Cases。

7. 好 Benchmark 的构建原则

一个完整 Agent Benchmark 至少应说明：

  1. Goal：要评测什么能力？
  2. Task：Agent 具体需要完成什么？
  3. Environment：Agent 在哪里运行？能使用哪些工具？
  4. Data Pipeline：测试样本如何生成、清洗和验证？
  5. Evaluator：如何判断任务成功？
  6. Metrics：报告哪些性能、可靠性和效率指标？

此外，应尽量满足：

  • Real-world relevance；
  • 多个难度等级；
  • 不容易被 Training Data Contamination；
  • 不容易快速 Saturation；
  • 结果可复现；
  • 能分析失败原因，而不只是输出一个总分。

8. Case Studies：优秀 Agent Benchmark 如何设计

8.1 CyberGym

Goal

评测 Agent 是否能够在大规模真实漏洞上复现网络安全漏洞。

Task

给定：

  • 漏洞描述；
  • Pre-patch Codebase；
  • 对应可执行程序。

Agent 需要生成一个 Proof-of-Concept（PoC），触发未修复代码中的目标漏洞。

Environment

Containerized Sandbox，用于构建、运行程序并安全检测崩溃或漏洞触发情况。

Data Pipeline

  • 基于 ARVO 和 OSS-Fuzz 发现的历史真实漏洞；
  • 重建漏洞修复前后的 Commit 和 Executable；
  • 保留 Ground-truth PoC；
  • 使用 LLM 将原始信息改写成简洁漏洞描述；
  • 通过人工检查保证任务质量。

Evaluation

在 Pre-patch 和 Post-patch 两个版本上运行 PoC：

  • 若只在 Pre-patch 触发目标漏洞，则说明复现成功；
  • 运行时 Sanitizer 根据 Crash 和 Stack Trace 自动检测；
  • 使用总体 Success Rate；
  • 通过评测模型知识截止日期之后发现的漏洞，分析数据污染风险。

启示

CyberGym 的优势是任务来自真实漏洞，Evaluator 由程序运行和 Sanitizer 完成，不依赖主观文本判断。

8.2 $\tau$-bench

Goal

评测 Agent 是否能够在多轮用户交互中：

  • 正确调用 API；
  • 完成用户目标；
  • 始终遵守复杂的领域 Policy。

Task 与 Environment

典型场景包括零售和航空客服。每个 Domain 提供：

  • API Tools；
  • Policy Document；
  • 数据库；
  • LLM-powered User Simulator。

Agent 需要通过多轮对话理解用户目标，并使用工具完成退货、改签等任务。

Data Pipeline

  • 人工设计 Schema、API 和 Policy；
  • GPT-4 辅助生成 Sampling Code 和 Synthetic Data；
  • 人工修订场景；
  • 多轮 Agent Run 验证；
  • 确保每个任务具有明确、唯一的目标 End State。

Evaluation

比较 Agent 执行后的 Final Database State 与 Annotated Goal State，属于程序化、可验证评测。

课程中使用的稳定性指标含义为：

$$ \text{pass@1} =\frac{1}{N}\sum_{i=1}^{N}\mathbb{I}[S_i=1] $$

若每个任务独立运行 $k$ 次，并要求 $k$ 次全部成功：

$$ \text{pass@}k =\frac{1}{N}\sum_{i=1}^{N} \prod_{r=1}^{k}\mathbb{I}[S_{i,r}=1] $$

这里的 pass@k 强调持续可靠性：Agent 偶尔成功不够，必须在重复运行中稳定成功。

注意：这一含义与代码生成领域常见的“$k$ 个候选中至少一个成功”的 pass@k 定义不同，应结合具体 Benchmark 文档理解。

8.3 $\tau^2$-bench

$\tau^2$-bench 将 $\tau$-bench 的单边控制扩展为 Dual-Control：Agent 和 User 都能通过自己的工具影响共享世界。

Data Pipeline

  • LLM 起草 Product Requirements Document；
  • 生成代码、Mock DB、Unit Tests、User DB 与 User Tools；
  • 从 Atomic Subtasks 组合复杂任务；
  • 使用初始化状态、解法和 Assertion 进行自动验证。

Evaluation

不仅检查最终数据库状态，还加入：

  • Environment Assertions；
  • Communication Assertions；
  • Natural Language Assertions；
  • Action Assertions。

这能够区分“最终结果碰巧正确”和“交互过程真正正确”。

8.4 GDPval

Goal

评测模型完成具有经济价值的真实 Knowledge Work 的能力，并将 AI Deliverable 与行业专家成果比较。

Task

每个任务类似真实工作委托，包含参考文件、数据和素材。模型需要一次性生成：

  • Document
  • Slide Deck
  • Spreadsheet
  • Diagram
  • Media 等工作成果。

Data Pipeline

  • 由经过筛选的行业专家编写任务，平均约 14 年经验；
  • 经过多轮审核和模型辅助验证；
  • Prompt 模拟日常工作要求；
  • 专家自己的成果作为 Reference Deliverable。

Evaluation

同职业领域的专家在 Blind Setting 下比较 AI 与 Human Deliverable：

  • Better
  • As good as
  • Worse

同时比较时间与成本。

启示

GDPval 的任务创建成本很高，但更贴近真实工作、不容易污染，也不容易因简单模板优化而迅速饱和。

8.5 CRMArena

Goal

评测 Agent 在真实企业 CRM 工作流中的能力。

Task

覆盖三类 Persona：

  • Service Agent
  • Analyst
  • Manager

代表性任务包括：

  • New Case Routing
  • Knowledge QA
  • Top Issue Identification
  • Monthly Trend Analysis

Environment

使用 Salesforce 的 Live Sandbox（Simple Demo Org），支持：

  • UI；
  • API；
  • SOQL / SOSL；
  • Function Calling；
  • 包含多种 Enterprise Object 的复杂 Schema。

Data Pipeline

  • 基于 Salesforce Service Cloud Schema 进行 LLM Synthetic Data Generation；
  • 引入 Agent Skill、Customer Shopping Habit 等 Latent Variables，形成更真实的因果模式；
  • Mini-batch Prompting；
  • String Matching 去重；
  • Format 与 Content 双重验证；
  • 使用 LLM Paraphrasing 增加 Query Diversity。

Evaluation

  • Knowledge QA：F1；
  • 其他任务：Ground-truth ID 的 Exact Match；
  • 通过 pass@k 衡量重复运行可靠性；
  • 同时报告 Turns、Tokens 和 Dollar Cost。

启示

评测 Agent 不应只看是否成功，还应衡量：

  • 交互次数；
  • Token 消耗；
  • 成本；
  • 稳定性。

8.6 LegalAgentBench

LegalAgentBench 面向真实中文法律工作流，任务需要 Multi-hop Retrieval、Reasoning、Legal Writing 和 Tool Use。

Environment

  • 17 个语料库；
  • 14 个表格数据库 + 3 个检索语料库；
  • 37 个工具，包括数据库、数学计算、文本检索和 Finish Tool。

Data Pipeline

  1. 根据 Tool Dependency 构建 Planning Tree；
  2. 选择 1–5 Hop 的串行或并行路径；
  3. 筛选能够成功执行的实体；
  4. 使用 GPT-4 改写问题，隐藏明显的解题路径；
  5. 通过 Toolchain 程序化生成答案；
  6. 对问题、路径和答案进行人工验证。

Evaluation

  • Success Rate：与 Gold Key Answer 的关键词匹配；
  • Process / Progress Rate：检查关键中间结果，奖励部分进展；
  • BERTScore；
  • Token Count。

该案例说明：对于复杂多步任务，可以同时评测最终结果和中间进度，而不是只有二元成功率。

9. Green Agent：课程项目中的评测 Agent

9.1 Green Agent 与 White Agent

可以将 Green Agent 类比为“出题和监考的老师”：

  • Green Agent：定义评测、分配任务、管理环境并评分；
  • White Agent：参加测试并尝试完成任务的 Participant Agent。

颜色命名来自项目早期对 Red Team、Blue Team 等安全评测角色的讨论，后来 Green Agent 固化为 Hosting / Evaluating Agent，其他参与者统称 White Agent。

9.2 Green Agent 的职责

Green Agent 负责：

  1. 准备并验证环境；
  2. 加载测试数据；
  3. 向 White Agent 分发任务；
  4. 管理工具、权限和运行流程；
  5. 收集 Agent 输出与执行结果；
  6. 计算指标；
  7. 将结果和运行状态报告给平台。

它不是一个单纯的 Judge Prompt，而是整个评测流程的 Orchestrator。

9.3 完整 Assessment Flow

  1. AgentBeats 确认所有 Agent 在线并重置状态；
  2. 平台向 Green Agent 发送任务和待测 White Agent 的 URL；
  3. Green Agent 分配任务并监督运行；
  4. Green Agent 管理工具和环境，持续上报进度；
  5. Green Agent 根据自身实现的规则计算 Metrics；
  6. 最终将结果提交给平台。

关键点：

由 Green Agent 决定测量什么指标，也由 Green Agent 执行得到这些指标的完整过程。

9.4 Green Agent 的组成

一个典型 Green Agent 包含：

  • 测试任务 Dataset；
  • 预定义 Testing Process；
  • Environment；
  • White Agent 可使用的 Tools；
  • MCP Modules 或其他外部服务；
  • 给 White Agent 的操作说明；
  • Evaluator 和 Metrics；
  • 运行 Trace 与结果上报逻辑。

9.5 Prompt-Based 与 Coding-Based

Prompt-Based Green Agent

适合：

  • 快速原型；
  • 流程较简单；
  • 主要依赖 Prompt 和 LLM Judge 的任务。

Coding-Based Green Agent

适合：

  • 严格工作流；
  • 自定义环境；
  • 数据库或容器状态检查；
  • 复杂 Tool Use；
  • 对可复现性和自动化要求高的任务。

课程中，2/3/4 Units 的项目通常预期以 Coding-Based Green Agent 为主；1 Unit 可以更多使用 Prompt-Based Toolkit。

只要遵循 A2A Protocol，任何符合协议的 Web Service 都可以作为 Green Agent。

10. 课程项目的两种类型

10.1 Type 1：Integrating Existing Benchmarks

目标

将已有 Benchmark 适配并集成到 AgentBeats，封装成 Green Agent，例如：

  • SWE-bench Verified
  • Terminal-Bench
  • 其他已有 Tool / Web / Domain Benchmark

已有 Benchmark 的 Metrics 和 Rubrics 可以大体复用，但不能只完成“代码搬运”。还需要分析 Benchmark 的质量和局限。

Step 1：Integration

  • 将原始问题格式转换成 AgentBeats / A2A 可处理的格式；
  • 实现 Dataset Loader 和 Interface；
  • 连接工具与环境；
  • 实现 Evaluator；
  • 检查运行正确性与可复现性。

Step 2：Benchmark Quality Analysis

Manual Validation

抽样检查：

  • 数据是否正确；
  • Task Description 是否清晰；
  • 难度是否合理；
  • 环境是否能够稳定运行。

Evaluator Check

验证原 Benchmark 的 Metric 或 Judge 是否与真实任务成功一致，即检查 Outcome Validity。

Bias & Limitation Analysis

分析：

  • 覆盖范围不足；
  • 数据分布偏差；
  • 污染风险；
  • 难度失衡；
  • 不公平测试；
  • 可能的 Shortcut。

Step 3：Correction and Expansion

  • 修复已发现的错误；
  • 删除或修正低质量样本；
  • 增加缺失的难度层级和边界场景；
  • 扩展规模、领域和样本多样性；
  • 改进 Evaluator 和测试环境。

10.2 SWE-bench Verified：Correction and Expansion 的例子

这个案例说明：

更困难不等于更高质量。若任务描述或 Evaluator 本身有问题，低分可能只是评测错误，而不是 Agent 能力不足。

10.3 Type 2：Building New Benchmarks

目标

从零设计新的 Agent Benchmark，覆盖现有评测尚未充分涉及的能力空间。

任务要求

  • 对应真实、有用的日常或专业场景；
  • 必须体现 Agentic Reasoning，而不是普通单轮问答；
  • 适合在课程项目规模内稳定评测；
  • 优先使用自动评测或轻量人工检查。

可能的任务：

  • Organize Calendar；
  • Schedule Meetings；
  • Manage To-dos；
  • 多系统信息协调；
  • 需要多角色协作的任务。

Scope by Units

  • 1 Unit：重点在 Prompt Design 和简单 Data Generation；
  • More than 1 Unit：需要更完善的 Rubric、多轮交互、数据生成流程和稳健评测。

Multi-Agent Benchmark

课程鼓励考虑多 Agent 任务，例如：

  • 多个 Worker 分别提出方案；
  • Synthesizer 汇总结果；
  • Analyzer 检查冲突、错误和遗漏；
  • 不同 Agent 负责不同子任务。

但多 Agent 不应只是形式上的角色数量增加，而应当让任务真正需要信息分工、协作或相互校验。

11. 构建 Green Agent 的 Step-by-Step Checklist

Step 1：Choose the Task

首先明确：

  • 要测试哪一种能力？
  • 是集成已有 Benchmark，还是提出新 Benchmark？
  • 为什么该任务需要 Agent？
  • Human 在相同信息与工具下是否能够完成？

Step 2：Design the Environment

明确：

  • Agent 能看到哪些信息；
  • 可以使用哪些 Tools；
  • 能执行哪些 Actions；
  • 每次行动后环境返回什么 Observation；
  • 环境如何 Reset；
  • 是否存在副作用和权限限制。

以 Ticket Booking 为例：

  • Tools：Browser、Ticket App 或 API；
  • Actions：Click、Typing、API Call；
  • Feedback：网页变化、搜索结果、订单状态或错误信息。

Step 3：Design Metrics Early

不要等系统实现完成后才临时决定指标。

Ticket Booking 不应只检查“是否订到票”，还可以考虑：

  • 是否满足时间、地点和座位要求；
  • 价格是否合理；
  • 是否产生重复订单；
  • 是否遵守用户预算；
  • 使用了多少步骤和成本。

一个多维评分示例：

$$ \text{Score} =w_s\cdot \text{Success} +w_q\cdot \text{RequirementMatch} -w_c\cdot \text{NormalizedCost} -w_t\cdot \text{NormalizedTurns} $$

其中各权重应根据任务目标设定，并通过测试样本验证是否会产生意外激励。

Step 4：Design Test Cases

测试样本不仅要覆盖正常成功，还应覆盖不同失败方式：

  • 正确完成任务；
  • 完成了错误任务；
  • 结果符合部分要求但违反关键约束；
  • 成功但成本过高；
  • 无法找到正确工具；
  • 工具调用参数错误；
  • 环境异常或部分不可用；
  • Prompt Injection 或恶意输入；
  • 各类 Edge Cases。

每个 Test Case 应尽量配套：

  • 初始状态；
  • Task Description；
  • 合法成功状态；
  • 不合法状态；
  • Ground-truth Eval Result；
  • 失败原因标签。

然后使用这些“已知结果”的案例反过来测试 Green Agent 本身，确认 Evaluator 不会误判。

12. Project Grading Rubric

12.1 New Benchmark Track

维度 核心问题
Goal & Novelty 目标是否重要？是否覆盖了新的能力空间？
Scope & Scale 样本量和多样性是否足以支持可靠结论？
Evaluator Quality 指标是否清晰？Judge 是否稳定、一致且高质量？
Validation 是否进行人工抽查或 Spot Validation？
Reliability Green Agent 是否能在 AgentBeats 上稳定运行？
Quality Assurance 是否检查 Bias、Contamination 和评测漏洞？
Realism 是否反映真实工作负载，而不是 Toy Setting？
Impact 是否可复用、文档完整、表达清晰？

12.2 Existing Benchmark Track

维度 核心问题
Analysis 是否识别原 Benchmark 的质量问题与缺陷？
Faithfulness 除明确修复部分外，是否忠实复现原结果？
Correction & Expansion 是否修复缺陷并扩展原 Benchmark 覆盖？
Evaluator Quality 指标和 Judge 是否清晰、一致、可信？
Validation 是否人工检查 Green Agent 的评测输出？
Reliability 脚本与 Agent 是否能稳定、可复现地运行？
Bias / Contamination QA 是否检查偏差、污染和 Shortcut？
Impact 是否可复用、文档完整、展示清晰？

核心区别：

  • New Benchmark 更重视 Novelty；
  • Existing Benchmark 更重视 Faithfulness、Analysis 与 Improvement。

13. Coding Example：将 $\tau$-bench 集成到 AgentBeats

课程最后给出了一个从已有 Benchmark 到 Green Agent 的实现思路。

13.1 Sort Out the Interface

两个基本原则：

  1. Human 在相同任务和信息条件下应当能够完成；
  2. 任务接口应尽量 Agent-friendly，让 Agent 能够自然使用自身的推理和工具能力。

例如：

  • Web Agent：只给 URL，还是提供结构化 Browser Actions？
  • Coding Agent：给 Repository 并要求 Patch，还是提供完整 Coding Environment？
  • Game Agent：通过自然语言确认投票，还是提供明确 Vote Tool？

实现前需要：

  • 阅读论文，理解原始 Task Formulation；
  • 阅读代码库，理解数据和评测接口；
  • 用尽量少的代码侵入，将相同信息转换为 A2A 格式。

13.2 两个关键挑战

Challenge 1：Cross-Agent Tool Use

原始 Benchmark 往往将 Tool 直接传给内部 Completion Interface，但标准化 White Agent 可能运行在远端。

可能方案：

  1. 为 White Agent 提供特殊 Tool Access；
    • 实现简单，但标准化较弱。
  2. 在 Prompt 中说明工具，让 White Agent返回 Tool Name 和 Arguments；
    • Green Agent 代理执行，但无法完整利用 White Agent 内部原生 Tool Calling。
  3. 提供 MCP Server；
    • 更标准化，但需要动态发现和连接工具。

Challenge 2：Migrate Evaluation

原始 Benchmark 的 Evaluator 可能直接访问内部 Tool Trace，但远程 Green Agent 未必能看到 White Agent 内部轨迹。

因此需要决定：

  • 是否让 Green Agent 代理所有 Tool Call；
  • 是否只通过最终环境状态评测；
  • 是否要求 White Agent 回传结构化 Trace；
  • 如何保证 Trace 不可伪造。

13.3 Design the Workflow

Kickoff Script

  • 向 Green Agent 发起测试；
  • 传入 White Agent URL、任务配置和运行参数；
  • 约定 Message Format。

Green Agent

  • Coding-Based；
  • 导入或复用原 Benchmark 逻辑；
  • 初始化任务和环境；
  • 与 White Agent 交互；
  • 执行最终评分；
  • 返回 Metrics 和 Traces。

White Agent

  • 可以是 Prompt-Based 或 LLM Workflow；
  • 选择适合的 SDK；
  • 使用有助于理解工具和 Policy 的 Prompt；
  • 按标准协议接收任务和返回行动。

13.4 从“能够运行”到“可复现、可公开使用”

完成 Green Agent、White Agent 和 Kickoff Script 后，只能算主要逻辑基本完成。进一步集成 AgentBeats 还需解决：

  1. 如何获得远程 Agent URL / MCP Server URL；
  2. 如何安全访问 LLM API；
  3. 如何报告 Metrics 与 Traces；
  4. 如何打包 Repository 供平台托管；
  5. 如何固定依赖、环境和随机种子；
  6. 如何让其他人重复运行得到一致结果。
