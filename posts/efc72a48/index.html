Post-Training Verifiable Agents

传统 Chat Model 的主要目标是生成“人类喜欢的回答”；Agentic Model 在此基础上，还需要在环境中采取行动、调用工具，并最终完成一个能够被检查的任务。因此，Agent 后训练的核心不只是语言风格对齐，而是：

$$ \text{理解环境状态} \rightarrow \text{选择并调用工具} \rightarrow \text{改变或查询环境} \rightarrow \text{完成任务} \rightarrow \text{由 Verifier 检查结果} $$

本节重点关注 Verifiable Agent 的三个关键：

  1. Good Training Data：构造多样且高质量的环境、工具、任务和验证器。
  2. Good Evaluations：定义什么是 Agent Intelligence，并用可靠评测量化它。
  3. Good Training Recipes：通过 SFT 与 RL，让模型能够训练更久、学习更难任务并探索更多有效轨迹。

最后进一步提出实践问题：如何把环境、验证器、评测和算法整合成开放、可扩展的基础设施。

1. Agentic Model 与传统 LLM 的区别

1.1 两类对齐目标

维度 Earlier Chat Models Agentic Models
主要对齐信号 Human Preference Environment Feedback + Human Preference
核心目标 生成让人满意、自然、有帮助的回复 在环境中正确完成任务并获得可验证奖励
主要交互对象 用户 用户、环境、工具、软件系统等多类实体
输出评价 通常没有唯一正确答案，需要偏好建模 经常存在明确正确、错误或部分正确的结果
常用反馈 Reward Model、人工排序 Unit Test、Ground Truth、状态检查、规则验证器等

传统 RLHF 流程通常是：收集 Human–Assistant 对话，对多个回答进行人工偏好排序，训练 Reward Model，再通过 SFT、PPO、GRPO 等方法优化模型，使其更符合人类偏好。


Agentic Model 的目标则更接近：

$$ \max_{\theta} \mathbb{E}_{\tau\sim \pi_\theta} \left[R_{\text{verifier}}(\tau)\right], $$

其中 $\tau$ 是模型在环境中执行的一条完整 trajectory，可能包括推理、工具调用、环境反馈和最终回答。

1.2 为什么只优化“对话质量”不够

自然对话往往不存在唯一正确答案。例如“你觉得这部电影怎么样”主要需要偏好建模；但在代码修复、数学求解、浏览器操作、数据库更新等任务中，错误结果可能直接导致整个系统失败。

Agent 还需要处理：

  • 用户目标可能并不清晰，需要主动澄清或分解需求；
  • 环境中存在大量状态，而非只有用户的一句话；
  • 必须选择正确的工具和调用格式；
  • 操作可能改变环境状态；
  • 最终结果必须能够被明确检查；
  • 除正确性外，还可能关心成本、步数、延迟和输出格式。

因此，一个可靠 Agent 不能只依靠外层软件工程“修补”一个能力不足的 LLM。模型本身必须具备较强的任务理解、工具使用和纠错能力。

2. Verifiable Agent 的基本组成

一条 Agent 轨迹可写作：

$$ \tau=(o_0,a_0,o_1,a_1,\ldots,o_T), $$

其中：

  • $o_t$：用户意图、工具返回值和当前环境状态等 observation；
  • $a_t$：模型生成的自然语言、工具调用或环境操作；
  • $o_{t+1}$：环境执行动作后的新反馈；
  • $R(\tau)$：Verifier 对完整轨迹或最终状态给出的奖励。

课件将可验证 Agent 数据的组成分为三个核心部分。

2.1 Environment：环境

Environment 表示任务发生的完整上下文和状态，例如：

  • 一个包含多个文件与依赖关系的代码仓库；
  • 一个网页或浏览器当前 DOM 状态；
  • 企业数据库、CRM 或销售系统；
  • 数学问题及其辅助计算环境；
  • 能够被 Agent 修改的文件系统或软件沙箱。

模型需要消费描述环境状态的 token，并理解用户当前真正希望实现的目标。

2.2 Tools：工具

工具是模型能够调用的外部软件接口。模型生成特定 token 或结构化参数，软件执行后把结果返回给模型。

工具大致有两类：

  1. 获取信息：天气 API、搜索引擎、数据库查询、文件读取、Python 执行等；
  2. 改变状态：更新数据库、修改代码、提交表单、执行 shell 命令等。

一个强 Agent 需要完成：

$$ \text{目标分解} \rightarrow \text{选择工具} \rightarrow \text{构造调用参数} \rightarrow \text{理解返回结果} \rightarrow \text{决定下一步行动}. $$

2.3 Verifier：验证器

Verifier 判断 Agent 是否真正完成任务，例如：

  • 代码任务中的 unit tests；
  • 数学题的答案检查器；
  • 形式化证明中的 proof checker；
  • 网页任务中的 DOM/state 检查脚本；
  • 数据库任务中的 ground-truth state checker；
  • JSON schema、格式约束和指令遵循测试。

Verifier 不一定只是一个二元的“对/错”信号。更完整的反馈可以写成向量：

$$ \mathbf r(\tau)= \left[ r_{\text{correct}}, r_{\text{format}}, r_{\text{cost}}, r_{\text{latency}}, r_{\text{safety}},\ldots \right]. $$

训练时可以根据任务需求把多个维度组合为标量奖励，例如：

$$ R(\tau) = R_{\text{task}}(\tau) -\lambda_{c}C(\tau) -\lambda_{l}L(\tau), $$

其中 $C(\tau)$ 可以表示工具或计算成本，$L(\tau)$ 可以表示不必要的轨迹长度。

3. Step 1：构造高质量训练数据

3.1 多样性比单一规模更重要

好的 Agent 数据不能只覆盖少数任务，而应同时覆盖：

  • 多种 Environment；
  • 多种 Tool 和 action space；
  • 多种 Verifier；
  • 同一任务下的不同工具组合；
  • 不同难度、不同失败方式和不同正确路径。

例如，一个数学环境可以连接 Python 工具和 Math Verifier；代码环境可以连接 shell、文件编辑器和 unit tests；销售助手可以连接天气 API、新闻 API、数据库和 ground-truth state verifier。

关键结论是：

Environment diversity 与 Verifier diversity 是 Agent 泛化能力的基础。

如果训练只覆盖少数固定环境或固定工具，模型很容易学会表面模板，而不能在新系统、新 API 或新交互方式中工作。

3.2 Verifier Quality：验证器质量

高质量 Verifier 必须同时减少：

  • False Positive：错误答案被误判为正确；
  • False Negative：合法答案被误判为错误。

例如：

  • 问“1 美元买 0.5 美元饮料，应找回多少”，$\frac12$ 与 $\frac24$ 在数值意义上都正确；
  • 若额外要求“给出最简形式”，则只能接受 $\frac12$，不能接受 $\frac24$。

Verifier 必须理解任务约束中的细微差别。错误验证器尤其会伤害困难样本：模型可能已经产生了有效的新解法，但由于验证器覆盖不足，正确轨迹无法获得奖励；反过来，若错误轨迹被奖励，模型会学会利用 verifier loophole。

3.3 训练数据中的 trajectory

Agent 训练数据不应只有最终答案，还需要包含过程：

  • 如何理解任务；
  • 为什么选择某个工具；
  • 工具调用参数；
  • 如何处理工具结果；
  • 遇到错误后如何修正；
  • 何时停止并提交最终结果。

因此，Agent 的 demonstration 更接近一条完整轨迹，而非普通问答对。

4. Step 2：构造高质量评测

有了训练数据，并不代表模型真的具有通用 Agent 能力。如果训练集与评测集高度相似，模型可能在已有 benchmark 上表现很好，却在真实环境中迅速失效。

4.1 Agent 评测需要覆盖的五类能力

  1. 能够使用多种工具；
  2. 能够处理多种 use case；
  3. 能够理解模糊或不完整指令；
  4. 能够适配不同 Agent Harness 和软件系统；
  5. 对人类输入、工具错误和环境异常具有鲁棒性。

这些能力都需要被独立、定量地追踪，而不能只看一个总分。

4.2 Benchmark、Harness 与 Unit-Test Evaluation

Off-the-Shelf Benchmarks

Agent Harness

Harness 是模型与 benchmark 环境之间的运行框架，它决定：

  • 模型能看到什么 observation；
  • 可以使用哪些工具；
  • 工具调用格式是什么；
  • 如何维护上下文与环境状态；
  • 何时终止任务。

同一 SWE-bench 任务可以在 OpenHands、MiniSWE、OpenSWE、Aider-SWEBench 等不同 harness 下运行。同一个模型若只在某个固定 harness 上表现好，可能只是适应了该工具接口，而不是掌握了真正的软件工程能力。

因此应比较：

$$ \text{Accuracy}(\text{Task},\text{Harness}_1), \quad \text{Accuracy}(\text{Task},\text{Harness}_2), \quad \ldots $$

若替换 harness 后性能大幅下降，通常说明模型对某种 action space 或交互模板存在过拟合。

Focused Unit-Test Evaluations

针对具体能力进行局部评测，例如：

  • Structured Output Adherence：能否严格生成 JSON 或指定 schema；
  • Conditional Instruction Following：能否处理带条件和例外的指令；
  • Long Trajectory：长时间、多步骤执行中是否保持目标一致性。

课件提到 structEval、IFBench、NexusBench 等例子。

4.3 Holistic Measurement：整体测量

Agent Intelligence 不能由单一 benchmark 定义。合理评测应构造任务与 harness 的组合矩阵。

例如，数学任务既可以在无工具条件下测试，也可以连接搜索或 Python；SWE-bench 可以在不同 coding harness 下测试。

选择什么 benchmark suite，本质上就是在定义“什么是智能”。因此评测必须保持 holistic，而不能只追逐某一榜单分数。

4.4 如何判断 Benchmark 本身是否可靠

主要看三个维度：

Hardness：难度

  • 太容易：强模型和弱模型都接近满分，benchmark 已饱和；
  • 太困难：所有模型都接近零分，无法提供有效学习或比较信号；
  • 合理：能够随着模型能力变化持续反映进步。

例如，当大量模型在 GSM8K 上已经接近高分时，继续提升少量分数不一定代表通用推理能力真正增强，还可能来自数据污染或训练相似性。

Separability：区分度

一个有效 benchmark 应让更强模型整体表现更好、较弱模型表现更差。如果不同能力模型取得几乎相同分数，benchmark 就难以区分模型能力。

可将区分度理解为：

$$ \operatorname{Sep}(B) \propto \operatorname{Var}_{m} \left[S_B(m)\right], $$

但高方差本身还不够，排序还应与独立认知中的模型能力大体一致。

Diversity：多样性

若 benchmark 只有少量高度相似的 prompt，即使得分很高，也不能说明模型能处理真实世界的复杂情况。应覆盖不同任务、环境、工具、输入风格和失败模式。

4.5 防止 Benchmark Overfitting

公开 benchmark 可能出现：

  • 训练数据污染；
  • 针对固定题集反复调参；
  • Harness 特化；
  • Verifier 漏洞；
  • benchmark 饱和。

私有 benchmark 能暂时缓解问题，但静态私有测试也可能逐渐失效。更根本的方案是持续构造大量、多样的新 benchmark，使“对所有 benchmark 过拟合”越来越接近真正掌握广泛能力。

5. Step 3：Agent 的训练范式

5.1 训练 Agent 到底意味着什么

给定任务和环境，模型会产生多条不同尝试：

$$ \tau_1,\tau_2,\ldots,\tau_K. $$

Verifier 对每条轨迹给出正确性与成本反馈。训练目标是：

  • 提高高质量轨迹的概率；
  • 降低错误轨迹的概率；
  • 保留足够的探索多样性；
  • 避免大量毫无意义、几乎不可能成功的昂贵 rollout。

这里存在一个核心张力：

  • 所有尝试都相同，模型得不到多样反馈；
  • 所有尝试都毫无意义，计算资源被浪费；
  • 理想状态是“meaningful but diverse”：不同、合理，并且至少部分轨迹能够成功。

5.2 先 Imitate，再 Explore

课程将 Agent 后训练分为两个阶段：

  1. SFT：模仿高质量 demonstration trajectory；
  2. RL：让当前模型探索，并强化成功轨迹、抑制失败轨迹。

6. SFT：让模型先产生有意义的轨迹

给定专家轨迹 $\tau^*$，SFT 可以写为：

$$ \mathcal L_{\mathrm{SFT}}(\theta) = -\mathbb E_{(x,\tau^*)} \left[ \sum_{t=1}^{T} \log \pi_\theta(a_t^*\mid x,a_{<t}^*) \right]. $$

SFT 的主要作用不是直接“产生最终智能”，而是：

  • 避免模型在 RL 初期产生大量垃圾轨迹；
  • 学会基础工具调用格式和任务流程；
  • 将 rollout 分布移动到有可能成功的区域；
  • 降低 RL 探索所需的计算成本。

6.1 SFT demonstration 必须多样

如果所有 demonstration 都使用同一种思路、同一种工具顺序或同一种表达，模型可能只会复制固定模板，后续 RL 很难探索其他路径。

6.2 SFT 应该保持较轻

过度 SFT（over-SFT）会让模型过分确信“只有 demonstration 中的答案才正确”，从而降低输出熵与探索能力。训练数据永远不能覆盖全部真实任务，所以模型不能只记住固定解法。

课程的简化结论是：

SFT 主要负责抑制无意义和低质量尝试；RL 才负责真正强化可泛化的智能。

6.3 SFT 数据来源

  • 人工构造的 expert demonstrations；
  • 强教师模型生成的 synthetic trajectories；
  • 当前模型生成候选后，通过 Verifier 做 rejection sampling，保留正确轨迹形成 self-distilled data。

7. RL：在反馈中强化智能

RL 阶段主要包括：

  1. Rollout：针对同一任务采样多条有意义且不同的轨迹；
  2. Reward / Advantage Computation：判断轨迹是否正确、是否罕见且有价值；
  3. Backpropagation：提高好动作概率、降低坏动作概率。

一个简化的 policy gradient 形式为：

$$ \nabla_\theta J(\theta) = \mathbb E_{\tau\sim\pi_\theta} \left[ A(\tau)\nabla_\theta\log\pi_\theta(\tau) \right]. $$

其中 $A(\tau)$ 表示该轨迹相对于基线的优势。真正困难的地方是：怎样设计 rollout、reward、advantage 和更新方式，使模型下一次遇到新任务时确实产生更好的轨迹，而不是只记住当前题目。

Good RL 三个支柱：

  1. Train Longer：训练更久而不发生探索能力崩塌；
  2. Train Harder：在相关且具有学习信号的困难任务上训练；
  3. Sample Better：以有限计算获得更多高质量、多样的响应。

8. Train Longer：在奖励与探索之间保持平衡

8.1 Entropy Collapse

对状态 $s$ 下的动作或 token 分布：

  • 高熵：模型对下一步存在多种可能，能产生不同轨迹；
  • 低熵：概率集中在少数 token 上，多次采样仍得到几乎相同的轨迹。

如上图实验，随着 RL 训练步数增加，reward / performance 上升，但 policy entropy 快速下降。模型虽然短期更准确，却可能逐渐失去探索能力，最终训练饱和。

目标不是无限增大熵，而是维持 entropy balance：

$$ \text{高正确性} + \text{足够探索性} + \text{可控的生成长度}. $$

8.2 方法一：减少有偏更新，尽量使用 On-Policy Data

On-policy 表示使用当前策略 $\pi_\theta$ 自己生成的轨迹，并立即根据这些轨迹的反馈更新模型；off-policy 则使用其他模型、旧策略或分布不匹配的数据。

直观上，当前模型更容易从“自己刚刚犯的错误”中学习。如上图实验中，on-policy 更新具有：

  • 更小的策略分布偏移或 KL divergence；
  • 更好的 entropy control；
  • 在 MATH-500、AIME 2024、AIME 2025 上更好的泛化表现。

这并不代表所有 off-policy 方法都无效，而是强调：rollout policy 与 update policy 的分布失配会引入偏差，尤其可能加速熵坍塌。

8.3 方法二：使用非对称 Clipping 平衡更新强度

令当前策略与旧策略的 token 概率比为：

$$ r_{i,t}(\theta) = \frac{ \pi_\theta(o_{i,t}\mid q,o_{i,<t}) }{ \pi_{\theta_{\mathrm{old}}}(o_{i,t}\mid q,o_{i,<t}) }. $$

DAPO 类方法采用非对称 clipping：

其直觉是：为低概率但可能有价值的 token 留出更大的概率提升空间，从而鼓励探索；同时仍限制过大的策略更新，保持训练稳定。

8.4 方法三：直接加入 Entropy Bonus / Entropy Loss

若优化目标是最大化 reward，可以增加熵奖励：

$$ \mathcal J_{\mathrm{total}}(\theta) = \mathcal J_{\mathrm{RL}}(\theta) + \alpha_H \mathbb E_s \left[H(\pi_\theta(\cdot\mid s))\right]. $$

等价地，若写为最小化损失：

$$ \mathcal L_{\mathrm{total}} = \mathcal L_{\mathrm{RL}} - \alpha_H H. $$

较大的 $\alpha_H$ 会更强地抑制 entropy collapse，但过大也可能让模型过于随机。因此真正问题仍是寻找 reward、entropy 与生成长度之间的合理平衡。

9. Train Harder：选择有意义的困难任务

9.1 太容易与太困难都无效

  • 任务太容易：模型几乎每次都成功，advantage 接近零，无法获得新学习信号；
  • 任务太困难：所有 rollout 都失败，模型不知道哪种尝试更好，同样难以学习；
  • 理想任务：与目标能力相关，当前模型成功率较低但不为零。

可以将合适难度理解为：

$$ 0 < P_{\pi_\theta}(\text{success}\mid x) \ll 1. $$

也就是说，任务应让模型有机会探索到正确答案，但不能已经轻易掌握。

如上图，只用较简单的 GSM8K 训练，困难验证集表现有限；但只把极难样本全部投入训练也未必最好。混合不同难度、形成合理课程式分布，通常更有效。

9.2 置信度与奖励需要相关

理想模型应该：

  • 对正确轨迹赋予更高生成概率；
  • 对错误轨迹保持较低置信度；
  • 训练后能把概率质量逐渐移动到高 reward 轨迹上。

课件给出的一个 proxy 为：

$$ \operatorname{Cov} \left( \pi_\theta(y\mid x)A(y,x), \log\pi_\theta(y\mid x) \right). $$

其核心含义是检查模型对候选答案的生成置信度，是否与该答案的 advantage / reward 有意义地一致。对于当前模型几乎完全无法解决的题目，这种相关性可能很弱，说明即使给出奖励也难以形成稳定学习信号。

9.3 Difficulty-Aware Reward / Advantage Reweighting

一种干预方式是根据题目经验成功率调整 advantage。设同一问题采样 $K$ 条轨迹，经验正确率为：

$$ \hat p_x = \frac{1}{K} \sum_{k=1}^{K} \mathbf 1\{\tau_k\text{ correct}\}. $$

对较难但仍可解的问题，使用更大的权重：

$$ \widetilde A_k = w(\hat p_x)A_k, \qquad w(\hat p_x) \text{ 随 }\hat p_x\text{ 降低而增大}. $$

同时可以：

  • 对正确但过长的回答进行 length penalty；
  • 对错误回答给予明确负奖励；
  • 对困难题中的正确轨迹赋予更强正奖励；
  • 避免简单题的大量重复正确样本主导梯度。

关键不是简单地“越难权重越大”，而是在仍有有效成功样本时增强困难任务的学习信号。

10. Sample Better：用计算换取更好的轨迹

RL 的学习质量高度依赖 rollout。如果采样不到多样且正确的轨迹，即使 verifier 很好，模型也没有足够高质量信号可学。

10.1 Parallel Reasoning 与 GenSelect

GenSelect 的基本流程是：

  1. 对同一个问题并行采样 $K$ 个不同解答；
  2. 将所有候选再次交给模型或 selector；
  3. 比较候选的逻辑、完整性、鲁棒性和最终答案；
  4. 选择其中最优解答。

形式上：

$$ \tau_1,\ldots,\tau_K \sim \pi_\theta(\cdot\mid x), $$

$$ k^* = \arg\max_k S_\phi(x,\tau_k), \qquad \hat y=y(\tau_{k^*}). $$

它利用并行计算把原本的 pass@$K$ 能力转化为更高的单次最终正确率，并可用于生成更强的 synthetic demonstrations。

10.2 Majority Voting 的局限

普通 self-consistency 会对 $K$ 个采样答案直接多数投票：

$$ \hat y = \arg\max_y \sum_{k=1}^{K} \mathbf 1[y_k=y]. $$

问题是：一些轨迹在早期已经明显失去方向，却仍参与最终投票，可能让大量低质量样本淹没少量高质量答案。

10.3 DeepConf：基于置信度筛选和加权

DeepConf 在生成过程中跟踪 trajectory confidence：

  • 若置信度低于阈值，可提前停止该轨迹；
  • 最终投票只保留高置信度轨迹；
  • 或按照置信度对候选答案加权。

$$ \hat y = \arg\max_y \sum_{k=1}^{K} w(c_k) \mathbf 1[y_k=y], $$

其中 $c_k$ 是第 $k$ 条轨迹的置信度，$w(c_k)$ 可以在低于阈值时取 0。

这相当于把有限计算集中到更有希望的推理路径上，而不是无差别地增加采样数量。

12. 课程的核心研究判断

12.1 当前方法尚未收敛

课程反复强调，SFT、PPO、GRPO、DAPO、entropy bonus、difficulty weighting 等方法仍带有较强经验性。它们是当前观察到有效的干预手段，不一定代表最终正确的学习算法。

现有实践与机器学习理论之间仍存在明显差距：

  • 为什么某些 clipping trick 有效；
  • 怎样从有限反馈中获得真正可泛化的更新；
  • 为什么模型会快速 entropy collapse；
  • SFT 与 RL 应如何统一；
  • 如何定义并测量 Agent Intelligence。

这些问题都仍是开放研究方向。

12.2 人类学习是重要类比，但不能直接照搬

人类会：

  • 在多种环境中学习；
  • 对某些课程投入更多精力；
  • 同时向教师模仿并自行探索；
  • 通过不同考试评价能力；
  • 根据反馈在下一次任务中改进。

Agent 后训练试图复制这一闭环，但必须把自然的人类学习过程转化为明确、可执行的算法。

13. Open Questions

  1. 开放环境集合如何设计？
    如何构建可持续扩展的 environments、evaluations、verifiers 和 algorithms 集合，使模型接触足够多样的真实任务？

  2. 如何分配学习重点？
    人类会重点学习尚未掌握但可以进步的内容。模型如何自动选择训练任务，同时不破坏训练稳定性？

  3. Imitation 与 Exploration 如何平衡？
    SFT 应做到多重、做到多深？能否在同一算法中同时利用教师轨迹与模型自身探索？

  4. 如何比较不同的 Intelligence 定义？
    不同 benchmark suite 会给出不同模型排名。如何判断哪种评测更接近真实 Agent 能力？

  5. 如何构造高覆盖率 Verifier？
    如何减少 corner cases、reward hacking、false positive 与 false negative，并验证 verifier 本身的可靠性？

  6. 如何实现持续学习而不发生熵坍塌？
    模型能否像人类一样，在持续反馈中不断改进，同时保留多种解决问题的方式？
