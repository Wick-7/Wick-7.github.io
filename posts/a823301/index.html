《生成式AI时代下的机器学习 第二讲》课堂笔记

发表于2026-07-03|更新于2026-07-03|学习笔记
|浏览量:

1. 什么是 AI Agent

AI Agent 没有唯一、严格统一的定义。这里讨论的是人类主要给出 goal，而不是逐步告诉 AI 应该执行什么；Agent 需要自己决定一连串 action，并根据环境回传的 observation 持续调整，直到完成目标。

普通的 AI 使用方式通常是“一条指令对应一个回答”。Agent 面对的任务则往往需要多个步骤，环境还可能发生无法预先知道的变化，因此它不能只执行固定脚本，而要在交互过程中不断判断下一步。

最基本的 Agent loop 可以写成：人类给出 goal；Agent 接收 observation；Agent 产生 action；action 改变环境；环境返回新的 observation；循环继续。这里的重点不是某一次回答是否聪明，而是系统能否长期维持一个面向目标的闭环。

AlphaGo 可以被视为典型 Agent：goal 是赢棋，observation 是当前棋盘状态，action 是选择落子位置。这个例子也说明 Agent 的描述方式与 Reinforcement Learning 中的 agent–environment interaction 非常接近。

$$ Goal → Observation_t → Action_t → Environment → Observation_t+1 → … $$

2. 从 RL Agent 到 LLM Agent

传统 Reinforcement Learning 的做法是先把任务目标设计成 reward，再训练一个 policy 去 maximize reward。以围棋为例，可以把胜利设为 +1、失败设为 -1，然后通过大量交互让模型学习。

这种路线的主要限制是 task-specific：通常每一个任务都要重新定义环境、action space 和 reward，并训练一套新的参数。一个会下围棋的模型不会因此自然获得下西洋棋或操作网页的能力。

LLM Agent 的想法不同：直接把已有的 Large Language Model 放入 Agent loop。goal 可以用自然语言表达；observation 可以是文字、图片或其他多模态信息；模型输出一段描述 action 的文字，再由外部系统把这段文字翻译为真正可以执行的操作。

从 LLM 自己的角度看，它没有突然学会一种全新的算法。goal、observation、action、下一次 observation 会依次进入 context，模型一直在做 next-token prediction。所谓 Agent，更多是对现有 LLM 能力的一种系统化使用方式，而不是重新训练出的新模型。

这也解释了为什么强大的 LLM 仍可能在棋类等任务中违反规则：语言上的流畅与对环境状态、规则约束和长期计划的可靠维护并不是同一件事。

3. 使用 LLM 驱动 Agent 的优势与限制

第一项优势是 action space 更开放。传统 Agent 通常只能从预先定义好的有限动作中选择；LLM 可以生成近乎任意的文字，因此可以表达更多行动，也更容易扩展到新的工具和任务。

第二项优势是能够直接利用 richer feedback。传统 RL 往往把环境反馈压缩成一个 reward 数值，但数值本身信息很少，reward 设计也很困难。LLM Agent 可以直接阅读 compile error、网页提示、测试结果或用户意见，从具体错误中修改下一步行为。

第三项优势是通用性。相同的基础模型可以在不同 prompt、不同工具和不同环境包装下承担 programmer、web agent、research assistant 等角色，而不一定要为每个任务单独训练。

但开放 action space 也意味着更难验证。语言模型可能产生看似合理、实际上不可执行或不符合规则的 action；多步执行还会放大早期错误。因此，一个完整 Agent system 通常还需要 memory、tool interface、validator、planner、permission control 等外部机制。

4. Agent 的三个关键能力

第一，能否根据过去的 experience 和当前 feedback 调整行为。这里不依赖更新模型参数，而是改变模型在下一步能够看到的 context。

第二，能否使用外部 tools。模型不可能把所有知识和能力都存进参数，因此需要搜索、执行代码、调用 API，甚至向其他 AI 求助。

第三，能否 planning。Agent 不仅要产生眼前的 action，还要考虑一连串动作如何共同实现最终 goal，并在环境变化时重新规划。

这三部分彼此相连：memory 为 planning 提供过去经验，tools 扩展可执行能力，而 planning 决定何时读取 memory、何时调用 tool、何时直接行动。

5. 不更新参数，如何根据反馈改变行为

假设 programmer agent 写出的代码出现 compile error。传统学习思路会想到收集数据、更新 parameters；本课强调另一条路线：把 error message 直接加入后续输入，模型面对不同 context，自然会产生不同代码。

这类改变属于 in-context adaptation，而不是 model training。模型参数没有变化，变化的是输入中新增了什么信息。对于 LLM 来说，“第一次写错”和“看过错误日志后重写”是两个不同的续写条件。

问题在于不能把 Agent 一生中的全部记录都塞进 context。随着交互步数增加，序列会变得过长，计算成本上升，重要经验也会被大量琐碎内容淹没。课程用 Highly Superior Autobiographical Memory 的类比说明：记住全部细节不一定等于更会思考。

因此，真正的问题不是“能不能存下来”，而是“什么值得存、什么时候读取、如何把零散经历整理成有用经验”。

6. Agent Memory：Read、Write 与 Reflection

Memory 可以看作 Agent 的长期经验库。它与当前 context 不同：context 是模型这一次直接看到的内容；memory 则可以长期保存，但需要额外模块把合适的信息取出来。

Read 模块负责 retrieval。当前 observation 相当于 query，过去经历构成 database，系统从中选出 relevant experience，再把这些经验放入 prompt。技术上它与 RAG 非常相似，区别主要在于资料来源：一般 RAG 检索外部知识，Agent memory 检索自己的经历。

Write 模块决定什么值得进入长期记忆。把每一个 observation 都保存会让 memory 被“桌子还在那里”“房间没有变化”等细节塞满。Write 可以由另一个 LLM 或 Agent 自己判断事件的重要性、novelty、长期价值和失败代价。

Reflection 模块对已有记忆进行更高层次的整理，例如总结规律、抽象策略、建立事件之间的因果或实体关系。它也可以把经历组织为 knowledge graph，再用 GraphRAG、HippoRAG 一类方法检索。

Reflection 并不天然可靠。它可能从有限证据推导出错误结论，并把错误总结长期保存。因此 memory system 需要保留原始证据、标注来源和时间，并允许用户或系统修正。

ChatGPT 的 memory 可以作为直观例子：明确说“记住这件事”可能触发 Write；之后相关问题会触发 Read。但模型可能把对话重新表述后再保存，所以记忆内容不一定逐字准确。

$$ \text{Memory System} = \text{Write} + \text{Read / Retrieval} + \text{Reflection} $$

7. Tool Use 与 Function Calling

对 Agent 而言，tool 是一个只需知道 interface、无需理解内部实现的外部能力。常见工具包括 Search Engine、Python、database、calculator、浏览器，以及具有不同能力的其他 AI。

工具可以统一抽象为 function：模型需要知道函数名称、参数含义、调用格式和返回结果。所谓 tool use 或 function calling，本质上是模型决定“此时应该调用哪个 function，并提供什么 arguments”。

一个通用流程是：System Prompt 说明可用工具和调用格式；User Prompt 给出当前任务；模型生成结构化 tool call；Agent runtime 解析这段输出并真正执行函数；工具结果作为 observation 写回 context；模型继续生成最终回答或下一次调用。

必须区分“模型输出了一串调用文字”和“函数真的被执行”。LLM 只会生成 token，真正连接 API、处理权限、执行代码和捕获错误的是 Agent developer 编写的 orchestration layer。

System Prompt 通常包含开发者长期固定的规则，User Prompt 则是本次用户输入。它们在实际平台上可能具有不同优先级，但无论具体 API 如何设计，核心都是让模型清楚知道可做什么、何时调用、输出应该满足什么 schema。

$$ \text{LLM} \to \text{tool call} \to \text{runtime executes function} \to \text{tool output} \to \text{LLM} $$

8. 工具很多时如何选择，以及如何自己制造工具

当工具只有几个时，可以把全部 description 放入 prompt；当工具达到数百或数千个时，完整阅读所有说明既浪费 context，也会增加选错工具的概率。

解决办法与 memory retrieval 相似：把 tool descriptions 存入工具库，根据当前 observation 先做 tool retrieval / tool selection，只把少量候选工具的 schema 提供给决策模型。

工具选择本身可以使用 embedding retrieval、classifier、LLM reranker 或层级目录。它要同时考虑 relevance、cost、latency、权限、可靠性与数据隐私，而不只是语义相似度。

LLM 还可以 make tools。因为工具本质上可以是一段 function，模型能够为反复出现的问题编写脚本，验证成功后把它保存到工具库，未来再次调用。

这种“把成功做法封装成工具”的机制与 memory 很相似：一个保存自然语言经验，一个保存可执行程序。两者都把一次性的尝试转化为可复用资产。

9. 不要盲信工具：Internal Knowledge 与 External Knowledge

工具可能出错，检索内容也可能是玩笑、广告、过时资料或恶意信息。课程以搜索结果建议在披萨中加入胶水的例子说明：RAG 能找到资料，不代表资料可信，也不代表模型能够正确理解语境。

LLM 在使用工具时会在 internal knowledge 与 external knowledge 之间权衡。外部结果如果与模型原有信念略有差异，模型可能接受；如果数值极端不合理，模型可能回到自己的 prior 并指出工具异常。

这种判断不是稳定的事实验证器。课程示例中，100°C 的异常温度仍可能被直接接受，而 10000°C 才触发质疑，说明模型的 sanity check 有阈值、也有不一致性。

有些研究指出：外部信息与模型 prior 差距越大，越难改变模型；模型对原答案越有信心，也越不容易被检索内容说服。

当多个来源冲突时，模型可能受到不应影响内容正确性的因素左右。例如：模型可能更信任 AI-generated text、较新的发布日期，以及某些页面视觉模板；而来源标签本身未必产生预期影响。

即使工具返回的每条资料都正确，最终 synthesis 仍可能错误，例如把同名人物的信息混合。可靠 Agent 需要 cross-check、source attribution、constraint validation 和异常检测，而不是把工具输出直接复制给用户。

使用工具还存在 cost。对于 3×4 这样的简单问题，调用计算器可能比模型直接回答更慢；是否调用应由任务难度、准确性要求、latency 和成本共同决定。

10. Reactive Response 与显式 Planning

Reactive Agent 只根据当前 observation 立刻产生下一步 action。它可能在简单任务中足够有效，但容易局部贪心：每一步看似合理，组合起来却无法完成长期目标。

显式 planning 要求模型先生成一串可能的 actions，再把 plan 放回 context，后续动作按照计划执行。Plan-and-Solve 一类 prompting 方法体现了“先整体分解，再逐步执行”的思想。

计划不能被视为不可修改的脚本。action 之后的 observation 由环境决定，对手可能走出意外棋步，网页可能弹出广告，API 可能失败。Agent 应在关键 observation 后重新检查 plan，并产生 plan′。

理想流程不是“每一步都推翻全部计划”，也不是“无论发生什么都坚持原计划”，而是持续比较预期状态与真实状态，只在偏差足以影响目标时 replan。

$$ \begin{aligned} \text{plan}_t &= P(\text{goal}, \text{observation}_t, \text{memory}) \\ \text{action}_t &= \pi(\text{observation}_t, \text{plan}_t) \end{aligned} $$

11. 强化规划：Tree Search、World Model 与 Reasoning

一种加强 planning 的方法是 search：从当前状态尝试多个 action，观察后继状态，再继续展开，寻找能到达目标的路径。由于完整搜索代价巨大，需要用 value estimate 或 LLM judge 提前剪掉“看起来没有希望”的分支。

Tree Search for Language Model Agents 展示了这种思想在 web agent 中的应用：模型探索候选路径，对成功概率打分，保留更有希望的路线。它比单次 reactive decision 更像真正的规划。

真实环境搜索存在 irreversible action：披萨已经下单后，不能因为另一分支更好就当作从未发生。因此，很多搜索不能直接在现实中反复试错。

解决思路是把探索放进内部 simulation。World Model 根据假想 action 预测可能出现的 observation，Agent 在“脑内小剧场”中比较分支，只把选中的 action 真正执行。Web agent 研究可以让 LLM 用文字模拟点击某个按钮后的页面变化。

Reasoning model 的长链思考也可被理解为内部 rollout：它尝试不同步骤、模拟后果、找到一个 plan，再执行第一步。课程中的积木例子展示了这种效果。

但更长 reasoning 并不总是更好。The Danger of Overthinking 指出，一些 Agent 会在可以低风险试一下时持续思考，或在尚未行动前就因为想象困难而放弃。有效 Agent 要在 thinking 与 acting 之间做资源分配。
