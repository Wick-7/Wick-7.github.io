Evolution of System Designs from an AI Engineer Perspective

从 AI Engineer 的视角观察：

  1. 模型算法为什么仍在持续进步；
  2. 什么样的 AI 应用能够真正创造价值；
  3. 为什么 AI Infra 正在成为企业 IT 的第三大支柱；
  4. 为什么新的系统设计正在重新走向过去 mainframe / HPC 的某些思路。

1. Demystify “LLM” and “AGI”

1.1 Chinese Typewriter Problem

英文打字机可以直接将字母映射到键盘按键，而常用汉字有数千个，无法直接放入普通 QWERTY 键盘中。早期中文打字机需要操作者移动机械结构，从巨大的字盘中寻找对应汉字。为了优化布局，人们统计常见的 bigram、n-gram，将经常连续出现的汉字放在相近位置，从而减少机械移动距离。

早期 n-gram 只能使用很短的历史，因为随着上下文长度增加，组合数量会迅速爆炸；现代 Transformer 则能够利用长上下文和大规模参数，在更大范围内建模 token 之间的关系。

1.2 BERT 与 GPT 的区别

  • BERT：利用目标位置前后的双向上下文，预测被遮盖的 token；
  • GPT：只使用当前位置之前的历史信息，自回归地预测下一个 token。

GPT 的生成过程与输入法或打字过程更相似：每次根据已有序列生成下一个 token，再将新 token 加入上下文继续预测。

2. 本节课的四条主线

讲者希望传达四个核心判断：

  1. New algorithms continue to drive LLM models
    模型能力仍未明显停滞，新的算法和训练范式继续推动性能提升。

  2. ToC apps thrive; ToB apps are hopeful and nascent
    消费级应用已经非常活跃，企业级应用潜力巨大，但仍处在相对早期阶段。

  3. AI Infra is the third pillar in IT strategy
    AI 基础设施不再只是传统云中的一个附属模块，而正在形成独立的基础设施体系。

  4. History repeats itself
    AI 系统在软件、硬件与集群设计上，正在重新采用许多 HPC 和 mainframe 时代的思路。

3. 新算法持续推动 LLM 演进

3.1 Closed-source 与 Open-weight 模型竞争

当前模型生态中存在两条快速发展的路线：

  • Closed-source models：GPT、Gemini、Claude、Grok 等；
  • Open-weight models：DeepSeek、Qwen、Llama、GPT-OSS 等。

讲者的观察是：

  • 闭源模型在绝对能力、推理质量和新能力上通常仍然领先；
  • 开放权重模型的追赶速度越来越快；
  • 新算法一旦被验证，开放社区通常可以在较短时间内复现和扩展；
  • 两条路线之间的竞争有助于整个生态持续进步。

判断 AI 是否只是“泡沫”，不能只看模型发布数量，也应观察真实使用量。课件以 OpenRouter 的 token consumption 为例，指出模型调用量持续快速增长，说明需求不只来自模型训练，也来自真实应用中的推理消费。

3.2 四个阶段的算法演进

课件用历史类比概括了 LLM 的几个重要阶段。

阶段 代表性变化 核心思想 历史类比
2022 前后 GPT / ChatGPT 结构与规模带来能力跃迁 AlexNet
2023 前后 Mixture of Experts 稀疏激活，提高参数与计算利用率 Ensemble、Inception
2024 前后 Test-time Scaling 在推理阶段投入更多计算 Multi-instance、空间上的重复预测
2025 前后 Reinforcement Learning 根据最终任务结果优化行为 经典 RL、长期回报优化

3.2.1 GPT：结构性创新

GPT / ChatGPT 的意义类似于 AlexNet：

  • AlexNet 使计算机视觉从 handcrafted features 转向端到端深度学习；
  • GPT 使自然语言任务从大量专用 pipeline 转向统一的预训练生成模型。

真正重要的不是某个单独 benchmark，而是整个问题求解范式发生了变化。

3.2.2 Mixture of Experts

MoE 并不是让每个 token 都经过所有参数，而是通过 gating network 为输入选择少量 expert。

这样可以：

  • 扩大模型总参数量；
  • 避免每次推理都使用全部参数；
  • 在保持计算成本相对可控的同时提升模型容量。

它与 Ensemble Learning、Inception 中的多分支结构具有相似思想：不是依赖一个单一、稠密的大模块，而是让不同子结构承担不同模式。

3.2.3 Test-time Scaling

过去主要通过扩大训练数据、参数和训练计算提升能力；Test-time Scaling 则强调：

在推理阶段允许模型进行更长的思考、搜索、验证和反思。

模型可以：

  • 产生中间推理步骤；
  • 对候选答案进行比较；
  • 检查前面的错误；
  • 重新规划后续步骤；
  • 用更多 inference compute 换取更高的准确率。

因此，模型能力不仅取决于参数本身，还取决于推理时分配了多少计算预算。

3.2.4 Reinforcement Learning

传统语言模型训练直接优化 token-level prediction，但 token 预测准确并不等价于最终任务完成得好。

强化学习允许根据完整 rollout 的最终结果提供反馈。

它解决的核心问题是：

当真正关心的目标很难写成可微 loss 时，如何利用最终结果训练模型。

对于 reasoning、coding、tool use 和 agent tasks，最终答案是否正确、程序是否通过测试、任务是否成功，往往比局部 token 是否符合参考文本更重要。

3.3 Hype Cycle：技术热词会变化，基础能力会沉淀

模型生态中的关注热点不断迁移：

  • ModelOps；
  • Prompt Engineering；
  • RAG；
  • Agentic AI；
  • Multi-agent systems；
  • Reasoning models。

一个技术从热点位置下降，并不意味着它失去价值。很多技术只是从“显眼的新概念”变成了默认基础设施。

RAG 并没有过时

  • RAG 仍然是降低 hallucination、接入企业知识和获得最新信息的重要方法；
  • 只是它逐渐被产品内部吸收，用户不一定直接看到“RAG”这个词；
  • 就像今天很少有人把“使用数据库”当作产品卖点，但几乎所有产品都依赖数据库。

一个常见的检索流程是：

  1. 使用 keyword search 或 embedding 进行低成本粗召回；
  2. 获得一批候选文档；
  3. 使用更强的 reranker 或 LLM 做精排；
  4. 将最相关的信息加入上下文；
  5. 生成带有 grounding 的回答。

其中，第一阶段偏向优化成本和召回率，第二阶段偏向优化准确率。

4. AI 应用：ToC 繁荣，ToB 仍在早期

4.1 好的应用体验不等于使用最强模型

Perfect app experience is correlated, but independent from models.

模型能力很重要，但完整产品还取决于：

  • 用户真实痛点；
  • 交互设计；
  • latency；
  • 数据与上下文获取；
  • workflow integration；
  • 可靠性；
  • 成本；
  • 对特定用户群体的理解。

讲者以浏览器扩展 Elmo 为例：工程师在很短时间内将网页解析、LLM 总结和问答结合起来，形成了真正可用的产品。

这说明从一个模型 API 到一个好产品之间，仍然存在大量系统工程和产品设计工作。

4.2 Consumer AI Landscape 高度流动

消费级 AI 应用已经形成多个主要方向：

  • Chat；
  • Coding；
  • Search；
  • Image / Video / Audio generation；
  • Writing；
  • Education；
  • Productivity。

但这一领域变化极快。底层 foundation model 一旦增强，许多只做简单 API wrapper 的产品可能迅速失去优势。

因此，应用的 moat 不能只来自“调用了某个强模型”，而应更多来自：

  • 自有模型或独特模型能力；
  • 专有数据；
  • 深入工作流的产品设计；
  • 用户关系与品牌；
  • 对垂直领域的理解；
  • 网络效应和持续反馈数据。

4.3 Prosumers 更愿意付费

消费流量并不一定直接转化为收入。较明确的商业机会来自 prosumer：

以个人身份购买产品，但将其用于专业工作或生产活动的用户。

典型产品包括：

  • Cursor：软件开发；
  • Runway：视频与内容创作；
  • Plaud：录音、转写、会议总结和 action items；
  • ElevenLabs：语音生成与语音生产工具。

用户在娱乐性体验上可能不愿长期付费，但当工具能够：

  • 节省工作时间；
  • 提升产出质量；
  • 增加收入；
  • 替代重复劳动；

每月支付几十美元就变得容易接受。

因此，一个重要的产品判断标准是：

这个应用是否嵌入了用户稳定、重复且高价值的工作流？

4.4 企业市场：慢，但可能更大

企业级应用的推进速度通常慢于消费应用，因为企业需要处理：

  • 数据连接；
  • 权限控制；
  • 安全与合规；
  • 内部系统集成；
  • 审计；
  • 稳定性；
  • 组织采购流程。

Glean 是讲者举出的代表案例。企业搜索并不只是“接一个 LLM”：

  1. 连接 Google Drive、Microsoft 365、GitHub、Jira、Wiki 等数据源；
  2. 同步内容和索引；
  3. 保留每个系统的 access control；
  4. 根据当前用户权限过滤结果；
  5. 进行检索、排序、总结和问答。

这些工作不一定在算法上最炫目，却解决了企业长期存在的真实痛点。

企业应用的优势通常来自：

$$ \text{Foundation Model} + \text{Domain Knowledge} + \text{Enterprise Data} + \text{Workflow Integration} $$

底层模型公司和应用公司不一定是零和竞争关系。一个熟悉用户、场景和业务流程的应用公司，可以借助更强的模型创造出“1 + 1 > 2”的产品效果。

4.5 企业 AI Stack

一个初步的 Generative AI Tech Stack：

  1. End-to-End Applications
    面向最终用户提供完整体验。

  2. Apps / Orchestration Layer
    负责模型调用、RAG、agent workflow、工具连接和业务逻辑。

  3. Closed-source / Open-source Foundation Models
    提供通用模型能力。

  4. Model Hubs
    存储、管理和分发模型。

  5. Cloud Platforms
    提供部署、调度和运行环境。

  6. Compute Hardware
    GPU、加速器、网络与存储。

一个应用可能从调用闭源 API 开始；随着数据、用户和业务积累，再逐渐训练或定制自己的模型。

5. AI Infra：企业 IT 的第三大支柱

5.1 The Bitter Lesson

Richard Sutton 在 The Bitter Lesson 中总结：

长期来看，能够利用大规模计算的通用方法，往往比依赖人工设计知识的专用方法更有效。

这一判断意味着：

  • 模型会继续消耗大量计算资源；
  • 计算基础设施不是模型之外的次要问题；
  • 算力、网络、存储、调度和故障恢复直接决定 AI 能力能否落地。

5.2 从 Scientific Computing 到 AI Cloud

基础设施演化时间线：

时期 主要形态 核心需求
1970s Scientific Computing 物理、天气等大规模数值模拟
1990s Virtual Private Servers 托管物理机器
2000s Web Service Cloud 网页、图片、视频与在线服务
2010s Data Cloud 海量数据处理、SQL、推荐与分析
2020s AI Cloud 高性能、异构、紧耦合的 AI 计算

AI Cloud 与 Web Cloud 并不是简单的“更多机器”，而是 workload 结构发生了变化。

5.3 Data Compute、Web Services 与 AI Compute

特征 Data Compute Web Services AI Compute
主要瓶颈 IO 远大于计算 IO 大于计算 计算远大于 IO
程序形态 SQL、MapReduce、Spark 任意服务代码 任意训练与推理代码
并行方式 大规模分布式数据处理 Embarrassingly Parallel 紧耦合分布式计算
用户抽象 较简单 相对简单 较难
基础设施难度 高 中等 很高

Web Services

Web 服务中的一个实例通常可以被复制：

  • 流量上升时增加副本；
  • 流量下降时减少副本；
  • 某个实例失败时由其他实例接管；
  • 不同实例之间耦合较弱。

Data Compute

Spark / MapReduce 会将一个大任务分成多个 shard：

  • 单个 shard 失败时可以重算；
  • SQL 提供简单的用户抽象；
  • 复杂调度由基础设施隐藏。

AI Compute

大规模训练具有不同特征：

  • 大量 GPU 同时参与一个 job；
  • GPU 之间需要高带宽、低延迟通信；
  • 机器拓扑和物理位置非常重要；
  • 一个节点失败可能使整个任务停止；
  • 训练代码和资源状态不一定容易迁移。

因此，AI workload 更接近传统 HPC / MPI / Slurm，而不是普通微服务。

5.4 传统 Cloud Value Proposition 不再完全成立

传统云的两个重要价值是：

  1. Software Supply Chain
    用户无需自己安装和维护数据库、中间件等复杂软件。

  2. Flexible Compute Supply Chain
    虚拟机可以快速创建、迁移和释放。

但 AI Cloud 中：

软件种类更少

大部分训练集中在：

  • PyTorch；
  • CUDA 与通信库；
  • Megatron、DeepSpeed、Lightning 等训练工具；
  • vLLM、SGLang 等推理系统。

其软件栈虽然复杂，但应用类别不像传统企业软件那样多样。

Workload 更统一

主要是：

  • 矩阵乘法；
  • attention；
  • collective communication；
  • checkpoint；
  • 大模型推理。

供应链灵活性更低

  • 多台 GPU 需要被固定地组织在同一拓扑中；
  • 很难对正在运行的大规模训练进行 live migration；
  • GPU 集群难以像 CPU 虚拟机一样随意拆分和重新组合；
  • GPU 主要用于 AI 数值计算，workload interchangeability 较低。

5.5 Neocloud 的出现

由于传统 hyperscaler 并不总能以最简单、最低成本的方式提供大规模 GPU，市场中出现了更加聚焦 AI workload 的 neocloud，例如课上提到的：

  • CoreWeave；
  • Lambda；
  • Nebius；
  • 其他专门聚合 GPU 的云服务商。

它们重点提供：

  • GPU availability；
  • 高性能网络；
  • AI workload 调度；
  • 更直接的训练与推理体验；
  • 面向 AI 的成本与利用率优化。

5.6 “You shouldn’t run on bare metal. Also, K8s is wrong.”

这句话不是说 bare metal 和 Kubernetes 都不能使用，而是在说明两种极端都不理想。

直接管理裸机的问题

研究人员自己维护机器时，需要处理：

  • GPU 故障；
  • 驱动和 CUDA；
  • 网络异常；
  • 机器分配；
  • 任务恢复；
  • 集群健康检查。

这会消耗宝贵的研发时间。

直接暴露 Kubernetes 的问题

Kubernetes 的抽象主要面向 SRE 和平台工程师。AI 研究人员通常不希望直接理解：

  • Pod；
  • Deployment；
  • Replica；
  • Service；
  • 各类调度和网络对象。

研究人员更希望看到的抽象是：

给我若干完整 GPU 节点，配置好通信环境，运行这个 PyTorch job；失败后自动换节点重试。

因此，真正需要的是：

底层可以使用 Kubernetes，但上层应提供 AI-native abstraction，隐藏无关的运维细节。

这也是 Slurm researcher mindset 与 Kubernetes operator mindset 之间的矛盾。

5.7 Developer Efficiency 与 Infra Efficiency

AI 平台必须同时优化两个目标。

Developer Efficiency

  • 研究人员的时间非常昂贵；
  • 提交实验应尽量简单；
  • 环境、资源和数据准备应自动化；
  • 失败后应快速恢复；
  • 训练、评估和部署流程应统一。

Infra Efficiency

  • GPU 价格高；
  • GPU 会发生硬件、显存、网络和软件故障；
  • 空闲 GPU 会造成巨大浪费；
  • 集群需要利用率管理；
  • 平台需要识别并隔离故障节点。

这两个目标可能冲突：

  • 开发者希望资源像“永远可靠的完整机器”；
  • 基础设施需要在背后进行调度、迁移、恢复和资源复用。

平台的价值，就是在不增加开发者认知负担的情况下实现基础设施效率。

5.8 AI Infra Best Practices

课件给出四项实践：

  1. Multi-cloud supply chain management
    GPU 供应短缺时，应能够在不同云和不同地区寻找资源。

  2. Elasticity and utilization management
    监控 GPU 使用率，减少闲置，合理安排训练和推理任务。

  3. AI-native platform
    用统一平台连接开发、训练和推理，屏蔽 Kubernetes 等底层细节。

  4. 将核心团队集中在模型和应用上
    对通用基础设施尽量复用成熟方案，把主要精力投入真正形成竞争优势的部分。

课程提到的相关生态包括：

  • Ray / Anyscale；
  • SkyPilot；
  • 面向训练和推理的 AI 平台。

5.9 AI Infra 不同，但也相同

模型服务进入生产环境后，传统分布式系统经验仍然适用。

6. Hardware and Software：Back to the Future

6.1 从 Mainframe 到小型服务器

早期 Cray-2 等 mainframe 将：

  • CPU；
  • Memory；
  • Storage；

连接到统一系统中，对程序员呈现为一台大型计算机。

优点是编程模型简单，缺点是：

  • 系统庞大；
  • 扩展不灵活；
  • 故障域大；
  • 成本高。

互联网与云计算时代则偏向 Open Compute Project 式的小型、模块化服务器：

  • 每台服务器相对独立；
  • 可以独立替换；
  • 适合运行大量松耦合微服务；
  • 通过网络和 RPC 进行通信。

6.2 AI 重新走向大型 Scale-up System

随着模型规模增加，AI 服务器不断变大：

  1. 单机中增加 GPU accelerator；
  2. 一台服务器安装 8 张 GPU，例如 DGX；
  3. 多台服务器通过高速网络组成集群；
  4. 进一步形成 GB200 NVL72 等 rack-scale system。

在 rack-scale system 中：

  • 高带宽 NVSwitch 将大量 GPU 连接在一起；
  • 不同 GPU 可以更直接地访问大规模共享内存资源；
  • 整个机架越来越像一台大型计算机；
  • 系统设计重新接近 mainframe 和 shared-memory HPC。

课件配图建议：第 34 页，Cray-2、模块化服务器、DGX 与 rack-scale system 的演化图。

6.3 为什么 Scale-up 很重要

传统 scale-out 集群中，每个节点是独立故障域和内存域。跨节点访问通常需要：

  • RPC；
  • MPI；
  • collective communication；
  • 显式数据切分。

当硬件提供更强的 rack-level interconnect 后，可以减少模型切分带来的系统复杂度。

这可能使以下技术更容易实现：

  • tensor / pipeline parallelism；
  • disaggregated prefill and decoding；
  • prompt caching；
  • KV cache sharing；
  • 大模型部署；
  • 更灵活的内存分配。

但硬件能力的变化需要软件栈同步演进。PyTorch、推理框架和调度系统仍需进一步利用 rack-scale architecture。

7. 从语言模型走向真实世界

7.1 Smart Airport / Digital Twin 案例

以智能机场项目说明真实世界问题的复杂性。

机场希望减少飞机降落后到达登机口之间的等待时间，需要综合处理：

  • 地图与跑道拓扑；
  • 航班计划；
  • 飞机实时位置；
  • 摄像头和传感器；
  • 空管资源；
  • 滑行道拥堵；
  • 安全规则。

系统需要建立 digital twin，对机场运行状态进行建模，识别：

  • 拥堵热点；
  • 低效滑行路径；
  • 被阻塞的区域；
  • 潜在调度优化空间。

这类任务并不是简单的：

输入 prompt，输出一段文本。

它要求模型：

  • 接收非文本数据；
  • 理解动态物理环境；
  • 与现有系统交互；
  • 做出影响现实世界的决策和动作。

因此，AI 的下一步机会还包括：

  • Physical AI；
  • Robotics；
  • World Models；
  • Embodied Agents；
  • 工业调度与控制。

8. 问答与创业经验

8.1 Hyperscaler：从卖资源转向卖结果

传统 hyperscaler 主要销售：

  • 虚拟机；
  • 存储；
  • 网络；
  • 数据库；
  • 运维便利性。

LLM 时代则逐渐转向销售：

  • tokens；
  • model endpoints；
  • managed inference；
  • 完整 workload；
  • 任务结果。

大型云厂商的优势在于：

  • 拥有更大的 GPU pool；
  • 内部也有大量训练和推理需求；
  • 可以在不同 workload 之间提高整体利用率；
  • 具有成熟的集群运维能力。

但 AI workload 仍需要变得更加：

  • interruptible；
  • resumable；
  • idempotent；
  • fault-tolerant。

例如，一个理想训练任务应能从 checkpoint 恢复，而不是某个节点失败后完全重来。

8.2 Token Economics

部分 AI 应用采用固定订阅价格，但重度用户产生的 token 成本可能超过订阅收入。

这类似“保险模型”：

  • 价格建立在大多数用户不会消耗到极限的假设上；
  • 若大量用户高强度使用，单位经济模型可能失衡。

讲者的观点相对乐观：

  • 推理成本会持续下降；
  • 同样价格下模型质量会快速提高；
  • 行业高速发展阶段允许一定程度的 loss-leading；
  • 关键是产品是否创造了真实价值。

风险在于：如果没有真实价值，只依赖资本补贴，商业模式就会成为 house of cards。

8.3 Multi-cloud 为什么仍然困难

“在任意云上启动相同 AI workload”比启动普通微服务难得多。

Compute Locality

大规模训练希望 GPU：

  • 位于同一 rack；
  • 连接同一高速 switch；
  • 具有一致拓扑；
  • 通信延迟足够低。

因此，VM 抽象无法完全隐藏物理位置。

Data Locality

训练可能需要 TB 或 PB 级数据。即使可以在另一家云上快速安装 PyTorch，也不能快速迁移全部数据。

不同环境可能使用：

  • Object Storage；
  • Elastic Block Storage；
  • Elastic File System；
  • NFS；
  • Lustre；
  • Parallel File System。

真正的 multi-cloud 不仅要迁移程序，还要处理数据复制、缓存和一致性。

8.4 创业：产品之外还需要 Attention

技术团队容易采用 inside-out 思维：

先把产品做得非常好，用户自然会发现它。

但创业还需要：

  • 市场认知；
  • Distribution；
  • Customer acquisition；
  • 持续展示产品；
  • 尽快获得反馈；
  • 根据反馈快速迭代。

讲者用一句玩笑概括：

Attention is all you need.

这里并不是鼓励夸大，而是强调优秀产品若没有被用户看见，也难以形成增长飞轮。

8.5 连接 Research 与 Engineering

研究人员通常关注：

  • 模型质量；
  • 算法创新；
  • 实验速度；
  • 是否能完成任务。

基础设施工程师通常关注：

  • 稳定性；
  • 成本；
  • 资源利用率；
  • 故障恢复；
  • 运维复杂度。

双方的目标并不矛盾，但优先级和语言不同。能够理解论文、模型训练，同时理解系统运维的人，可以在两者之间发挥桥梁作用。

这是 AI Engineer 的重要能力：

不只理解模型，也能将模型转化为可靠、可扩展的系统。

8.6 Startup 的差异化机会

若没有极强的研究团队和巨额资本，直接训练“又一个通用 foundation model”通常很难形成优势。

更现实的创业方向是：

  • 选择尚未被解决的 vertical domain；
  • 深入理解该行业的数据和 workflow；
  • 将通用模型能力与领域知识结合；
  • 完成大型模型公司不愿投入的大量集成工作；
  • 形成独特数据和用户反馈闭环。

企业服务尤其适合这种路径，因为通用模型公司不可能深入覆盖每一个行业。

8.7 Centralized Scaling 与 Edge AI

近两年的主要趋势是：

  • 追求更大的集中式训练集群；
  • 使用更多 inference compute；
  • 先提升能力上限，再考虑效率。

但在以下场景中，efficient AI 会越来越重要：

  • Robotics；
  • Mobile devices；
  • Edge computing；
  • Telecom / CDN nodes；
  • 资源受限的本地设备。

这反映了 exploration 与 exploitation 的平衡：

  • Exploration：不惜成本追求能力上限；
  • Exploitation：压缩模型和系统成本，使其进入真实应用。

8.8 Lepton AI 最大的现实挑战

讲者认为，最大的困难不一定是某个纯技术问题，而是 GPU supply chain management：

  • GPU 稀缺；
  • 不同供应商价格差异大；
  • 合同周期不灵活；
  • 需要谈判 committed spending；
  • 资源需求又会快速变化。

创业团队不得不花费大量时间：

  • 找 GPU；
  • 比价；
  • 谈合同；
  • 协调不同云；
  • 管理利用率。

理想的未来是形成更加流动的全球算力市场，使模型和应用团队专注于核心创新，而不是持续进行 InfraOps。
