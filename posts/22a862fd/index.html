LLM Agents：历史、核心范式与未来方向

1. 什么是 Agent

1.1 基本定义

一个宽泛定义是：

Agent 是一个能够与某种环境交互的“智能”系统。

Agent 与环境之间形成持续的交互闭环：

$$ \text{Agent} \xrightarrow{\text{Action}} \text{Environment} \xrightarrow{\text{Observation}} \text{Agent}. $$

环境可以是：

  • Physical Environment：机器人、自动驾驶汽车、现实世界；
  • Digital Environment：游戏、网页、操作系统、软件与 API；
  • Human Environment：对话机器人通过语言与人类交互。

因此，AlphaGo、Atari Agent、自动驾驶汽车、Alexa 和 Chatbot 都可以被视为不同形式的 Agent。

1.2 Agent 的定义会随时代变化

要严格定义 Agent，还需要回答两个问题：

  1. 什么叫作 intelligent？
  2. 什么属于 environment？

这两个概念并非固定不变。早期只依靠少量规则进行对话的系统可能被认为具有智能，而在今天，即使 ChatGPT 展现出复杂能力，人们也会迅速提高对“智能”的要求。

因此，Agent 并不是只由某个固定架构定义，而是由我们对智能、环境和交互能力的理解共同决定。

2. 从 Text Agent 到 Reasoning Agent

2.1 Level 1：Text Agent

Text Agent 使用语言形式的 observation 和 action：

$$ o_t \in \mathcal{L}, \qquad a_t \in \mathcal{L}, $$

其中 $\mathcal{L}$ 表示语言序列空间。

它并不一定使用大语言模型。典型例子包括：

  • ELIZA；
  • LSTM-DQN 文本游戏 Agent。

2.2 Level 2：LLM Agent

LLM Agent 使用 LLM 产生动作。例如：

  • SayCan；
  • Language Planner。

模型可能根据当前 observation 直接输出下一步 action，但不一定显式生成任务分析、计划或状态总结。

2.3 Level 3：Reasoning Agent

Reasoning Agent 使用 LLM reason to act：先通过语言进行推理，再决定如何行动。

典型例子：

  • ReAct；
  • AutoGPT。

三者的关系可以概括为：

$$ \text{Reasoning Agent} \subseteq \text{LLM Agent} \subseteq \text{Text Agent}. $$

本课程和当前 LLM Agent 研究的重点，主要是第三层：能够通过推理指导行动的 Agent。

2.4 两种早期 Text Agent

方法 核心机制 主要局限
ELIZA（1966） 人工规则匹配与模板回复 Domain-specific、依赖人工设计、难以超出简单场景
LSTM-DQN（2015） 在文本游戏中利用 Deep RL 学习 observation-to-action 映射 Domain-specific、需要标量奖励、训练交互量大

LLM 的重要承诺在于：模型通过大规模语料上的 next-token prediction 获得丰富先验，然后可以在推理阶段通过 zero-shot / few-shot prompting 适配大量新任务，而不必为每个领域重新设计规则或训练一个新策略。

3. LLM Agent 的发展主线

ReAct 之后的发展又分为两条主线：

  • 新应用与新任务：Web Browsing、Software Engineering、Scientific Discovery 等；
  • 新方法：Memory、Learning、Planning、Multi-agent 等。

这条历史线索的核心不是某一篇论文，而是两种范式逐渐汇合：

  1. Reasoning：在模型内部增加推理计算；
  2. Acting：与外部环境交互，获取知识、计算结果和反馈。

4. 从任务特化方法到统一抽象

4.1 从 Question Answering 看模型缺少什么

不同 QA 问题会暴露不同能力缺口：

问题类型 所需能力 常见方法
数学文字题、符号题 Reasoning Chain-of-Thought
最新事实、外部知识 Knowledge RAG / Search
大数运算、质因数分解 Computation Program of Thoughts / Calculator / Python
多跳知识问题 Reasoning + Knowledge Self-Ask、IRCoT、交错检索与推理

例如，纯自回归语言模型不适合可靠完成较复杂的精确计算。Program of Thoughts 的思路是：

$$ \text{Question} \xrightarrow{\text{LLM}} \text{Program} \xrightarrow{\text{Executor}} \text{Answer}. $$

RAG 则通过检索器从外部语料库中取得相关内容：

$$ D_q = \operatorname{Retrieve}(q, \mathcal{D}), $$

$$ y \sim P_{\theta}(y\mid q,D_q). $$

Tool Use 进一步允许模型调用搜索引擎、计算器、翻译模型、天气 API 等。不过，早期工具调用方案通常采用特殊 token 或特殊格式，可能需要针对特定工具微调，并且不容易自然地完成多轮调用。

4.2 问题：方法越来越碎片化

同样属于 QA，知识问答、数学推理、多跳问答和常识问答却形成了许多独立 benchmark 与 task-specific method。

能否找到一个简单、统一、跨任务的解决方案？

答案需要建立在更高层的抽象上，而不是继续为每一个 benchmark 添加一个独立模块。

4.3 Reasoning 与 Acting 的抽象

Reasoning

Reasoning 在模型内部更新当前认知、计划和中间状态。CoT 的关键作用是增加 test-time compute：

$$ \text{Input} \rightarrow r_1 \rightarrow r_2 \rightarrow \cdots \rightarrow r_T \rightarrow \text{Answer}. $$

优点：

  • 灵活、通用；
  • 能通过更长的推理过程处理复杂任务；
  • 不需要修改外部环境。

局限：

  • 无法获得实时事实；
  • 无法访问模型参数之外的知识；
  • 不擅长保证精确计算；
  • 无法获得真实环境反馈。

Acting

Acting 让模型向环境发出动作并接收 observation：

$$ a_t \rightarrow \text{Environment} \rightarrow o_{t+1}. $$

环境可以是检索系统、搜索引擎、计算器、Python、代码执行器、网页或机器人。

优点：能够扩展知识、计算、感知和反馈能力。
局限：若缺乏推理，Agent 可能无法选择正确工具、制定调用顺序，也无法根据异常 observation 调整计划。

5. ReAct：Reasoning 与 Acting 的结合

5.1 核心思想

ReAct = Reasoning + Acting。

Agent 交替生成：

形成闭环：

$$ \text{Thought}_t \rightarrow \text{Action}_t \rightarrow \text{Observation}_{t+1} \rightarrow \text{Thought}_{t+1}. $$

它不是单纯的 prompting trick，而是一种使用语言模型组织交互轨迹的通用方式。ReAct 可以通过：

  • One-shot prompting；
  • Few-shot prompting；
  • Fine-tuning；

来实现。

5.2 ReAct 为什么形成协同

Reasoning 与 Acting 的关系不是简单相加，而是相互支持：

  • Acting supports reasoning：搜索、计算和执行结果为推理提供新信息；
  • Reasoning guides acting：推理决定查什么、调用什么工具、何时停止，并能在失败后重新规划。

在课件的市值问题中，Agent 需要：

  1. 搜索 Apple、Nvidia 和 Microsoft 的实时市值；
  2. 对数值求和；
  3. 与 7 trillion dollars 比较；
  4. 输出资金缺口。

如果搜索未直接返回结果，Agent 可以改为分别搜索各家公司；如果 observation 只返回股价，Agent 能识别“股价不等于市值”，并继续查找流通股数：

这说明 Agent 不只是机械调用工具，而是在 observation 不完整或具有误导性时调整方案。

5.3 ReAct 超越 QA

很多任务都可以转换为某种 text game：

  • 将视觉或环境状态转成文本 observation；
  • 由 LLM 输出文本 action；
  • 再由 controller 将文本 action 映射为真实操作。

因此，ReAct 可以从问答扩展到网页、游戏、机器人和软件环境。

5.4 Acting without Reasoning 的失败

在 ALFWorld 的 household task 中，目标是找到 pepper shaker 并放入 drawer。

只生成 action 的模型可能出现：

模型只在模仿局部 observation-to-action pattern，无法：

  • 系统探索不同位置；
  • 维护已搜索位置；
  • 理解失败反馈；
  • 重新规划。

加入 $Think[...]$ 后，Agent 可以先提出搜索策略，依次检查 cabinet 和 countertop，找到目标后再规划打开 drawer 并放置物体。

5.5 实验结果

课件给出的 PaLM-540B 结果为：

方法 HotpotQA FEVER ALFWorld
Reason only 29.4 56.3 N/A
Act only 25.7 58.9 45
ReAct 35.1 64.6 71

ReAct 在 NLP 与交互式 RL 任务中都优于单独 Reasoning 或单独 Acting，支持其跨领域统一性。

6. 为什么 Reasoning 是一种内部动作

6.1 传统 Agent 的动作空间

传统 Agent 的动作空间由环境定义。设环境动作空间为 $\mathcal{A}$，历史 context 为：

$$ c_t=(o_1,a_1,o_2,a_2,\ldots,o_t), $$

策略从中选择动作：

$$ a_t\sim \pi(a\mid c_t), \qquad a_t\in\mathcal{A}. $$

例如：

  • Atari 中的动作可能只有上、下、左、右；
  • 机器人环境中可能是移动、抓取和释放；
  • 网页环境中可能是 click、type 和 scroll。

6.2 ReAct 扩展了动作空间

ReAct 将语言推理也视为一种动作：

$$ \hat{\mathcal{A}} = \mathcal{A}\cup\mathcal{L}, $$

其中 $\mathcal{L}$ 是开放的语言序列空间。

语言内部动作可表示为：

$$ \hat a_t\in\mathcal{L}. $$

一次 reasoning action 不直接改变外部世界，而是更新 Agent 的内部 context：

$$ c_{t+1} = (c_t,\hat a_t,a_t,o_{t+1}). $$

因此：

  • External Action 改变环境并产生外部反馈；
  • Reasoning Action 只改变 Agent 的内部状态、记忆和后续决策。

6.3 语言内部动作的特殊性

传统环境动作通常是有限、固定的；语言空间则近似开放：Agent 可以生成一个词、一句话、一段计划，甚至很长的推理轨迹。

Reasoning 因此提供了：

  • Planning；
  • State Tracking；
  • Error Diagnosis；
  • Replanning；
  • Inference-time Scaling。

这也是 Reasoning Agent 相较于“直接从 observation 映射到 action”的 LLM Agent 更根本的差异。

7. Long-term Memory 与新的学习方式

7.1 Context Window 是 Short-term Memory

ReAct 轨迹中的 instruction、thought、action 和 observation 都保存在模型 context 中，可以视为 short-term memory。

它存在四个限制：

  1. Append-only：通常只能继续追加新 token；
  2. Limited Context：上下文长度始终有限；
  3. Limited Attention：即使窗口很长，过多内容也会产生干扰；
  4. Do not persist over new tasks：新会话或新任务中不会自然保留过去经验。

即使一个 Agent 今天解决了非常困难的问题，如果没有更新参数或写入外部记忆，下一次仍可能从头开始。

7.2 Long-term Memory 的特征

长期记忆应满足：

  • 可读、可写；
  • 存储 experience、knowledge、skills；
  • 能跨任务、跨时间持续存在；
  • 在需要时能够被检索并影响后续行为。

可写为一个概念化流程：

$$ M_{t+1}=\operatorname{Write}(M_t,e_t), $$

$$ m_t=\operatorname{Retrieve}(M_t,q_t), $$

$$ a_t\sim\pi(a\mid c_t,m_t). $$

其中 $M_t$ 是长期记忆库，$e_t$ 是新经验，$m_t$ 是当前任务检索到的相关记忆。

7.3 Reflexion：Verbal Reinforcement Learning

Reflexion 在失败后生成语言反思，并将反思持久化：

$$ r_t=\operatorname{Reflect}(\tau_t,f_t), $$

$$ M_{t+1}=M_t\cup\{r_t\}, $$

其中 $\tau_t$ 是本轮轨迹，$f_t$ 是单元测试、执行错误或其他反馈。

例如，编码任务失败后，Agent 可以记录：

之后再解决同类问题时，Agent 读取这段经验，而不需要修改模型参数。

课件给出的 HumanEval pass@1 示例：

方法 pass@1
GPT-3.5 zero-shot 48.1
GPT-4 zero-shot 67
GPT-4 + Reflexion 91

7.4 与传统 Reinforcement Learning 的区别

传统 RL 通常根据标量奖励更新策略参数：

$$ \theta \leftarrow \theta+\alpha\nabla_\theta J(\theta). $$

其特点是：

  • 反馈常被压缩为 scalar reward；
  • 需要 credit assignment；
  • 学习通过梯度更新模型权重完成。

Reflexion 式学习则：

  • 可以使用文本反馈、编译错误、单元测试、教师意见等丰富信号；
  • 不必进行 gradient descent；
  • 通过更新语言形式的长期记忆影响未来行为。

因此，它是一种 learning by updating language，而不是只通过更新 weights 学习。

7.5 不同类型的长期记忆

Procedural Memory：Voyager


Voyager 在 Minecraft 中将已掌握的能力保存为代码形式的 skill，例如制作工具、建造结构或完成某种操作。未来任务可以检索并复用这些 skill，而不是重新探索。

Episodic Memory：Generative Agents


Episodic Memory 记录具体事件：

  • 某时发生了什么；
  • 与谁进行了交互；
  • 执行了什么动作；
  • 环境如何变化。

它类似高粒度日记，为后续计划提供历史依据。

Semantic Memory：Reflective Knowledge

Agent 可以从 episodic logs 中抽象出更稳定的结论，例如：

  • 某个人具有怎样的性格；
  • 自己偏好什么；
  • 哪类策略更有效。

这种反思所得的抽象知识会进一步影响未来行为。

7.6 CoALA：统一 Agent 架构的抽象

讲者介绍的 Cognitive Architectures for Language Agents（CoALA）认为，可以用三个部分描述 Agent：

  1. Memory：信息保存在哪里；
  2. Action Space：Agent 能做什么；
  3. Decision-making Procedure：如何从候选动作中选择下一步。

可以概念化为：

$$ \text{Agent} = (\text{Memory},\text{Action Space},\text{Decision Making}). $$

参数化语言模型本身也可被看作一种长期记忆；外部文本、代码和向量数据库是另一种长期记忆。由此，fine-tuning 和写入外部记忆可以被理解为两种不同的学习途径。

7.7 两个边界问题

课程留下了两个开放问题：

  1. 外部环境与内部记忆如何区分？
    Agent 将内容写入 Google Docs，是改变环境，还是写入长期记忆？检索互联网归档，是工具调用，还是 memory retrieval？

  2. 长期与短期记忆如何区分？
    如果模型拥有千万 token 的 context，它仍然只是短期记忆吗？

对数字 Agent 而言，这些边界比人类“身体内外”的划分更加模糊。

8. LLM Agent 与传统 Agent 的本质区别

8.1 一个极简的 Agent 历史

这不是完整的历史，而是为了突出不同 Agent 使用何种“内部语言”连接 observation 与 action。

8.2 三种中间表示

所有 Agent 都在解决：

$$ \text{Observation} \rightarrow \text{Internal Representation} \rightarrow \text{Action}. $$

Symbolic AI Agent

将 observation 转换为 symbolic state，再通过人工规则或逻辑推导动作。

  • 优点：结构明确、可解释；
  • 局限：需要大量人工设计，难以跨领域复用。

Deep RL Agent

将 observation 编码为 neural embedding，再通过神经策略网络输出动作。

  • 优点：能从交互数据中学习复杂策略；
  • 局限：通常需要大量训练步骤，策略和表示高度 task-specific。

Reasoning Agent

直接以 open-ended natural language 作为中间表示：

语言带来三个关键变化：

  1. Rich Priors：LLM 已从大规模文本中学到大量知识和模式；
  2. Inference-time Scalable：可以通过生成更长 reasoning 增加推理计算；
  3. General and Generalizable：相同语言表示可用于游戏、代码、网页、生活决策等不同任务。

8.3 为什么“这一次不同”

Symbolic state 和 neural embedding 往往是固定维度、任务定制的表示；自然语言则具有开放长度和广泛语义，可连接人类知识、工具说明、环境反馈与任务计划。

因此，LLM Agent 的关键不只是“模型参数更大”，而是 Agent 开始使用语言作为通用的认知与控制接口。

9. 从 Questions and Games 到 Digital Automation

9.1 传统 Agent Benchmark 的局限

早期 Agent 常在：

  • GridWorld；
  • MiniWoB；
  • TextWorld；
  • BabyAI；
  • 小型模拟游戏；

中进行评估。

这些环境通常具有可扩展的交互数据，但任务规模小、合成性强，与真实工作价值存在距离。即使 Agent 在一个简单游戏中达到 100%，也很难说明其能处理现实任务。

9.2 Digital Automation 的意义

LLM Agent 打开了一类新任务：自动化现实数字世界中的工作，例如：

  • 填写报销和处理行政流程；
  • 编写、调试和测试代码；
  • 运行实验；
  • 搜索、阅读与整理论文；
  • 浏览网页和操作企业软件。

这类任务需要：

  1. 在真实语言与多模态信息上推理；
  2. 在开放动作空间中决策；
  3. 完成长程、多步骤交互；
  4. 根据执行结果不断调整。

传统 sequence-to-sequence 映射通常不足以支持这些要求。

9.3 WebShop

WebShop 将在线购物构造成大规模文本交互环境：

  • 使用约 116 万个 Amazon 商品；
  • Agent 可搜索、点击、返回、选择属性并购买；
  • 根据指令与商品属性匹配自动计算 reward；
  • 同时考察语言理解、视觉理解和长程决策。

与小型 GridWorld 相比，它既更贴近真实应用，也保留了自动化、可重复的评价机制。

9.4 WebArena

WebArena 将网页 Agent 扩展到更广泛的真实网站与任务，要求 Agent 在多个站点和应用中执行复杂操作。它代表了从单一购物环境走向通用 web interaction benchmark 的发展。

9.5 SWE-bench

SWE-bench 的任务定义为：

  • Input：GitHub repository + issue；
  • Output：能够修复问题的 file diff；
  • Evaluation：Pull Request 中的 unit tests。

Agent 需要浏览代码库、定位文件、分析 issue、修改代码、运行测试并根据错误继续迭代。这是典型的 long-horizon software engineering task。

9.6 ChemCrow

ChemCrow 将 ReAct 式 Agent 用于化学研究：

  • 访问化学数据、搜索、Python 与领域工具；
  • 通过推理提出候选分子或实验方案；
  • 将数字空间中的建议连接到实验室合成；
  • 从物理实验反馈中继续改进。

它说明 Language Agent 的 action space 可以从数字工具延伸到物理世界，也可以从重复性事务扩展到 scientific discovery。

9.7 三类环境的权衡

Digital World 同时拥有现实价值、丰富复杂度以及低成本快速交互，是 LLM Agent 发展的关键新空间。

10. 对研究方法的启示

10.1 Simplicity and Generality

CoT 和 ReAct 的核心思想都非常简单，但正因为简单，才容易跨任务迁移。

Simple often means general，但真正提出简单而通用的抽象并不容易。

10.2 研究需要两种能力

Thinking in Abstraction

不能只盯住单个数据点或 benchmark，需要识别不同任务背后的共同结构。例如，从大量 QA 方法中抽象出 Reasoning 与 Acting，而不是继续增加一个 task-specific module。

Familiarity with Tasks

抽象不能脱离具体问题。研究者需要深入理解：

  • 数据长什么样；
  • 环境如何交互；
  • 模型究竟在哪里失败；
  • 评价指标是否真实反映任务需求。

讲者特别区分了“熟悉任务”和“熟悉所有任务特化方法”：后者有时会将研究思路限制在增量改进上。

10.3 学习历史与其他学科

理解 Symbolic AI、RL、Cognitive Architecture、HCI 和心理学中的记忆概念，有助于建立更高层抽象，而不是把每个新 Agent 问题都视为完全独立的新问题。

11. LLM Agent 的未来方向

讲者提出五个关键词：

  • Training；
  • Interface；
  • Robustness；
  • Human；
  • Benchmark。

这些方向彼此关联：需要针对 Agent 训练模型，为 Agent 设计合适接口，并在包含真实用户和真实约束的 benchmark 上评估可靠性。

11.1 Training：为 Agent 训练模型

当前常见流程是：

但通用 LLM 主要训练于互联网文本，并未直接训练以下能力：

  • Planning；
  • Tool Use；
  • Self-evaluation；
  • Calibration；
  • 多轮 action-observation trajectory；
  • 错误恢复。

互联网通常只包含最终文章或最终代码，而不包含完整的思考、行动和修正轨迹。这造成 model training data 与 agent deployment data 的分布差异。

FireAct 的思路是：

  1. 先用 prompted agents 生成 ReAct trajectories；
  2. 使用轨迹 fine-tune 模型；
  3. 建立 model-agent synergy。

Agent trajectory 还可能成为模型训练的“next trillion tokens”：不是继续依赖逐渐耗尽的高质量互联网文本，而是让 Agent 与环境交互产生具有规划、反馈和修正信息的新数据。

11.2 Interface：Agent-Computer Interface

HCI 的基本思想是：即使人不变，更好的工具界面也能提高任务表现。对 Agent 同样如此：

如果暂时无法提高 Agent 本身，可以优化它所面对的环境与接口。

传统命令行中的 $ls$、$cd$、逐项搜索等接口是为人类设计的，不一定适合 LLM。

SWE-agent 提出 Agent-Computer Interface（ACI）：

  • 为 Agent 设计更容易解析和操作的命令；
  • 一次返回适合模型 context 的信息量；
  • 减少无关输出；
  • 让 Agent 更容易浏览、编辑、测试与恢复。

使用相同 LLM 和相同 prompt，仅改变接口就可能显著提升 SWE-bench 表现。

为什么 Agent 与人需要不同接口

人类短期记忆较小，通常需要分页、菜单和一次显示一个搜索结果；模型拥有更长 context，可能更适合一次接收多个结果并统一比较。

因此：

$$ \text{Best Interface for Humans} \neq \text{Best Interface for LLM Agents}. $$

ACI 不仅提升性能，也能帮助研究人类与模型在记忆、注意和决策方面的差异。

11.3 Robustness：从“偶尔成功”到“稳定不失败”

在代码生成等具有自动验证的 autonomous task 中，研究者常关心 $pass@k$：采样 $k$ 次，至少一次成功。

若每次独立成功概率为 $p$，其直观形式为：

$$ \operatorname{pass@}k = 1-(1-p)^k. $$

随着 $k$ 增大，该指标自然上升。对于证明难题或生成一个可通过测试的程序，“只要大量尝试中成功一次”可能有价值。

但在客户服务等 human-in-the-loop 场景中，更重要的是连续 $k$ 次都成功，可用 $pass^k$ 表达：

$$ \operatorname{pass}^{k} = p^k $$

（上式是在独立同分布假设下的直观解释。）

这里的问题从：

我能否在 1000 次尝试中成功一次？

变为：

我会不会在 1000 次服务中失败一次？

现实部署需要稳定完成相对简单的任务，而不仅是在高采样预算下偶然完成困难任务。

11.4 Human：把真实交互纳入 Agent 系统

许多 benchmark 一开始就向 Agent 提供完整、无歧义的任务描述。但现实用户可能只说：

Agent 需要追问：

  • 哪个订单？
  • 改到什么日期？
  • 是否接受价格变化？
  • 用户身份和权限是什么？

此外，Agent 可能无法直接看到用户私有信息或企业数据库，必须分别通过 chat 和 tools 获取。

因此，真实 Agent 需要同时与：

$$ \text{User} \leftrightarrow \text{Agent} \leftrightarrow \text{Tools / Backend} $$

交互。

11.5 Benchmark：τ-bench

τ-bench 面向 Tool-Agent-User Interaction in Real-World Domains，重点包括：

  • 实际业务规则作为 system policy；
  • Agent 通过工具访问后台系统；
  • Agent 与模拟用户多轮对话；
  • 用户信息和数据库信息不直接暴露；
  • 评价任务是否正确且可靠地完成。

如图中，各模型的 $pass@k$ 随采样次数增加而上升，但表示连续全部成功的 $pass^k$ 明显下降。这说明当前模型即使有较高单次能力，也仍缺乏稳定性。

11.6 从“能替代哪些工作”反推研究问题

可以通过分析 Agent 尚不能承担哪些真实工作来发现研究方向：

  1. 简单但要求高度可靠的任务；
  2. 必须与人类持续协作的任务；
  3. 极难、需要自主探索和创造的任务。

关键问题不是泛泛讨论“Agent 会替代谁”，而是定位每类工作缺少的具体能力：鲁棒性、长期记忆、接口、隐私控制、交互理解，还是开放式探索。

一句话总结

LLM Agent 的关键突破，不只是让模型能够调用工具，而是让自然语言成为连接推理、行动、记忆与学习的通用内部表示；ReAct 将 reasoning 与 acting 统一起来，长期记忆使经验能够持续积累，而未来研究需要进一步解决专门训练、Agent-native interface、人类交互和现实环境下的稳定可靠性。
